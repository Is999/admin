# AGENTS.md

本文件是 Codex 参与 `admin` 开发的短入口。开始修改代码、配置、SQL、脚本或文档前，必须先读：

- `docs/site/角色文档/后端开发/AI开发提示词.md`

## 工作硬约束

- 先读现有代码、目录结构、调用链和相关文档，再改代码。
- 需求或问题表述不充分、存在多种解释、目标仓库/入口不清或涉及接口、数据、权限、安全、大表、缓存、迁移时，必须先结合代码和文档说明理解、证据、待确认点和执行边界，未经确认不得直接落代码。
- 优先延续当前项目风格和现有分层；遇到成熟通用能力、高风险基础设施或手写实现容易出错的场景，必须先检查现有实现和成熟依赖/框架。确需新增抽象、目录、依赖或框架时，先说明必要性、收益、替代方案、影响范围和验证方式，并让开发确认后再落地；禁止为简单逻辑过度设计，也禁止因怕新增依赖而手搓不可靠实现。
- 需求必须按真实入口、调用链和交付面逐项闭环；禁止只实现基础能力、样例、脚手架、单一分支或 happy path 后伪装已完成。
- 修改要小步聚焦；不要顺手重构无关代码，不要回滚用户已有改动。
- 代码注释、配置注释和文档说明必须简洁清晰，不能啰嗦，也不能含糊。
- AI 实现的代码必须把简洁、逻辑清晰、易读易维护放在高优先级；合理封装只能用于降低真实复杂度、复用稳定边界或隔离外部依赖，禁止用过度抽象、隐式魔法或复杂堆叠牺牲可读性。
- 目录、变量、方法、常量、结构体、接口和字段命名必须短而准确、贴合业务职责；禁止泛名、错名、过长命名或实现职责变化后沿用旧名。
- 每轮落代码前、中、后都必须持续对照 AI 开发规范，尤其是中文注释要求；新增或重构的变量、常量、方法、结构体、字段、map/数组/切片字段、路由、GORM Model、接口和关键逻辑必须补简洁中文注释，避免多轮实现后变成难以审阅维护的无说明代码堆叠。
- 接口、配置、SQL/Lua 资产、权限码、前端类型和文档必须与代码同步。
- MySQL/SQL 编写必须优先使用 GORM 链式调用；只有 GORM 无法安全表达或会严重影响性能时才允许原生 SQL，并必须说明原因、索引依据、扫描范围、分页/批量上限、超时/并发和 DB 压力，禁止参考遗留风格随意手写 SQL。
- 涉及大表、大结果集或高频查询，先评估索引、执行计划、并发和库压力；禁止在线接口无边界重查询。

## 必用 Skills

- 功能开发、问题修复、重构、生产级复核和交付前检查必须优先使用 `$standard-development-flow`，再按场景调用专项 skill。
- Go 后端实现、修复、重构或生产级复核时使用 `$go-ai-implementation-guard`。
- 需求意图不清、问题需先分析、可能存在理解偏差或需要确认执行边界时使用 `$requirement-intent-gate`。
- 需求实现、收口、闭环、命名审查或交付前完整性复核时使用 `$implementation-closure-review`。
- 全仓库注释、可读性、AI 开发规范或“确保没有类似问题”检查时使用 `$go-comment-style-audit`。
- Redis Key 收口、共享缓存、TTL、失效、SCAN/KEYS 风险或缓存治理时使用 `$redis-key-governance`。
- 新增或修改接口、路由、业务码、权限码、接口文档、前端类型或安全字段时使用 `$api-contract-sync`。
- GORM/SQL、慢查询、索引、读写库、大表分页、批量查询或 DB 压力风险时使用 `$go-db-query-performance-guard`。
- Go 测试、vet、race、编译、CI 或历史噪声排查时使用 `$go-test-failure-debugger`。
- 配置默认值、YAML/env、热加载、启动期能力或重启边界时使用 `$go-config-runtime-guard`。
- worker、Kafka、Asynq、scheduler、锁、重试、超时、幂等、归档或告警时使用 `$go-worker-concurrency-guard`。
- 登录态、JWT/session、MFA、权限、签名、加密、RouteMeta 或安全路由时使用 `$go-security-route-guard`。
- DDL、Model 对齐、迁移、回填、数据修复、缓存重建或补偿任务时使用 `$go-migration-backfill-runbook`。
- 多个兄弟仓库需要对齐、同步修复或比较同类文件时使用 `$sibling-repo-sync-review`。
- 交付前 staged 文件、验证结果、迁移/回填/缓存/重启/回滚说明收口时使用 `$release-handoff-guard`。

## 必跑检查

- 后端改动优先运行 `go test ./...`；范围小或环境受限时至少运行相关包测试，并说明原因。
- 前端改动在 `../admin-vue` 运行 `pnpm -F @vben/web-antd run typecheck`，必要时补 lint/build。
- 完成后运行对应仓库的 `git diff --check`。
- 每次任务结束前必须检查 `git status --short`；本轮新增且属于交付范围的文件必须执行 `git add` 加入本地仓库索引，不能让应提交文件停留在未跟踪状态，也不能添加无关历史未跟踪文件或本地数据目录。

## 交付说明

最终回复必须说明改了什么、怎么验证、是否需要数据迁移或补偿任务。
