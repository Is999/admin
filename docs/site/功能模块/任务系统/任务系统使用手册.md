# 任务系统使用手册

## 文档定位

本文只说明任务系统框架能力：队列、Worker、Scheduler、工作流、任务列表和配置热加载。不描述具体业务模块的按钮、口径或排障流程。

字段级请求体和响应体见 [任务系统接口总览](接口文档/任务系统/任务系统接口.md)；开发接入见 [任务队列与工作流指南](角色文档/后端开发/任务队列与工作流指南.md)。

## 核心能力

- 异步任务投递与消费
- 延迟任务与定时任务
- 自动重试、归档和人工重跑
- DAG 工作流、分片和灰度
- 周期任务调度
- 队列暂停、恢复和状态观察
- 多站点共享 task Redis 时的 `app_id` 隔离

## 运行模式

`run_mode` 使用位掩码组合：

| run_mode | 组成 | 适用场景 | 会做什么 | 不会做什么 |
| --- | --- | --- | --- | --- |
| `1` | API | 只提供管理入口 | 接收手工触发、查询状态 | 不消费任务、不跑周期任务 |
| `2` | Worker | 执行面 | 消费队列、执行工作流节点 | 不提供接口、不注册周期任务 |
| `4` | Scheduler | 调度面 | 注册并投递周期任务 | 不执行节点、不提供接口 |
| `5` | API + Scheduler | 推荐控制面 | 提供接口、维护周期调度 | 不执行工作流节点 |
| `6` | Worker + Scheduler | 较少使用 | 调度并执行任务 | 不提供接口 |
| `7` | API + Worker + Scheduler | 本地联调或单实例环境 | 接口、调度、执行都在同一进程 | 无 |

除 `0` 用于“未显式指定，回退配置或默认值”外，启动模式只允许 `1..7`。

## 推荐部署

生产建议拆成控制面和执行面：

- `cron-control`：`run_mode=5`，负责 API 和周期调度。
- `cron-worker`：`run_mode=2`，负责消费队列和执行任务。

本地开发可以使用 `run_mode=7`。

## 后台菜单和接口关系

| 后台菜单 | 主要接口 | 作用 |
| --- | --- | --- |
| 任务总控 | `POST /api/tasks`、`POST /api/tasks/workflows` | 手工投递任务、手工触发工作流 |
| 任务队列 | `GET /api/tasks/queues`、`POST /api/tasks/queues/pause/:queue`、`POST /api/tasks/queues/resume/:queue` | 查看队列、Worker 与调度器状态，暂停或恢复队列 |
| 任务列表 | `GET /api/tasks/overview`、`GET /api/tasks/:taskId`、`POST /api/tasks/run/:taskId`、`DELETE /api/tasks/:taskId` | 查看任务总览、单任务详情、立即执行、删除任务 |
| 工作流状态 | `GET /api/tasks/workflows/:workflowId` | 按 workflowId 查看工作流实例、节点进度与失败原因 |
| 配置热加载 | `GET /api/tasks/config-reload`、`GET /api/tasks/config-reload/items`、`POST /api/tasks/config-reload` | 查看配置热加载状态、搜索运行态配置项与手工重载 |

## 页面观察顺序

1. 在任务队列页确认队列是否存在、Worker 是否在线、队列是否被暂停。
2. 在任务总控页查看 `data.scheduler`，确认 Scheduler 是否启用、是否已抢到 leader、最近同步和入队是否成功。
3. 在任务列表页按队列、状态、时间范围查任务，进入详情看 payload、headers、result 和失败原因。
4. 在工作流状态页按 `workflowId` 查看 DAG 节点进度。
5. 在配置热加载页确认运行期配置是否加载成功，并通过配置项搜索核对当前进程实际生效值；`restartRequired=true` 时需要重启进程才能完全生效。

## 任务列表查询

- 默认按任务主时间倒序，最新计划、最新失败或最新完成的任务排在前面。
- `scheduled` 使用 `nextProcessAt` 排序和过滤。
- `retry`、`archived` 优先使用 `lastFailedAt`。
- `completed` 使用 `completedAt`。
- `active` 使用 `startedAt`。
- 生产排障建议同时指定队列和状态，减少高积压队列扫描范围。

## 队列职责

默认逻辑队列：

- `critical`：高优先级任务。
- `default`：普通后台任务。
- `maintenance`：长耗时、批量、维护类任务。

建议：

- 高优先级队列只放少量必须尽快执行的任务。
- 长耗时和批量任务优先放到 `maintenance`。
- 同一类周期任务和手工补跑任务尽量使用同一队列，便于排查积压。

## 任务和工作流

- 普通任务：直接投递一个 handler，对应一次性执行动作。
- 工作流：先投递 `workflow:trigger` 入口任务，再由 Worker 创建工作流状态、调度根节点，并按 DAG 推进后续节点。

接口返回成功通常只表示已经成功入队，不表示任务或工作流已经执行完成。

## 周期任务

周期任务生效需要同时满足：

1. `task.enabled=true`
2. 当前实例具备 Scheduler 能力，也就是 `run_mode` 包含 `4`
3. 周期任务配置有效且未禁用
4. Scheduler 能抢到 Redis leader 锁
5. 唯一键没有阻止本次重复投递

周期任务配置建议放在运行期配置文件的顶层 `task_periodic`。

## 多站点隔离

多个站点共用同一套 `task.redis` 时必须配置 `app_id`：

- 同一站点下的 `cron-control` 和 `cron-worker` 使用相同 `app_id`。
- 不同站点使用不同 `app_id`。

系统会按 `app_id` 隔离：

- 实际消费队列名
- 工作流状态 key
- 唯一键 key
- Scheduler leader 锁
- 聚合分组 key

后台页面展示逻辑队列名，不展示底层命名空间前缀。

## 发布验收

上线或变更任务系统配置前至少确认：

1. `run_mode` 与部署职责一致。
2. `cron-control` 和 `cron-worker` 指向同一套 `task.redis`。
3. 同站点实例的 `app_id` 完全一致。
4. Worker 权重包含目标队列。
5. 周期任务实例具备 Scheduler 能力。
6. 任务列表、任务队列、工作流状态和配置热加载页面都能正常查询。

## 常见误区

- “接口成功”不等于“任务执行成功”。
- “队列名一样”不等于“多站点会串任务”，底层已按 `app_id` 隔离。
- “周期任务没跑”不一定是 Worker 问题，优先查 Scheduler、周期配置和唯一键。
- “配置热加载成功”不等于“所有基础设施连接都已重建”，要看 `restartRequired`。
