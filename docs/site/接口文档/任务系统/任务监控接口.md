# 任务监控接口

## 1. 适用菜单

- 后台菜单：`任务监控`

## 2. 查询工作流状态

### 接口

```http
GET /api/tasks/workflows/:workflowId
```

### 请求示例

```http
GET /api/tasks/workflows/wf_xxx
```

### 查询参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `workflowId` | string | 是 | 工作流实例 ID，放在路径参数 `:workflowId` |

### 可查看内容

- 工作流当前状态：`pending/running/success/failed`
- 每个节点的执行状态和错误信息
- 节点的开始时间、完成时间、分片执行情况

## 3. 查询配置热加载状态

### 接口

```http
GET /api/tasks/config-reload
```

### 请求示例

```http
GET /api/tasks/config-reload
```

### 可查看内容

- 是否启用热加载
- 当前配置摘要
- 最后一次重载是否成功
- 最后一次触发来源和失败分类
- 本次热加载是否仍需重启进程才能完全生效
- 重复失败是否正在被限频压制

### 关键字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `enabled` | bool | 是否启用配置热加载 |
| `watching` | bool | 当前是否正在后台监听配置文件 |
| `configFile` | string | 当前监听的配置文件路径 |
| `checkIntervalSeconds` | int | 当前轮询间隔，单位秒 |
| `configVersion` | string | 当前生效配置版本指纹 |
| `configSummary` | string | 当前关键配置摘要 |
| `restartRequired` | bool | 本次热加载后是否仍需重启进程才能完全生效 |
| `restartReason` | string | 需要重启的原因摘要 |
| `lastStatus` | string | 最近一次处理结果：`idle/success/failed` |
| `lastMessage` | string | 最近一次处理结果说明 |
| `lastTriggerSource` | string | 最近一次触发来源：`startup/watcher/manual_api` |
| `lastFailureCategory` | string | 最近一次失败分类：`fingerprint/load/not_bound` |
| `lastCheckedAt` | string | 最近一次检查配置文件时间，RFC3339 |
| `lastReloadAt` | string | 最近一次触发配置重载时间，RFC3339 |
| `lastSuccessAt` | string | 最近一次成功加载时间，RFC3339 |
| `lastFailureAt` | string | 最近一次失败时间，RFC3339 |
| `reloadCount` | int64 | 累计成功加载次数 |
| `suppressedFailureCount` | int64 | 限频压制的重复失败日志次数 |

## 4. 查询运行态配置项

### 接口

```http
GET /api/tasks/config-reload/items
```

### 请求示例

```http
GET /api/tasks/config-reload/items?keyword=kafka&page=1&pageSize=20
```

### 查询参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `keyword` | string | 否 | 配置路径、类型或已脱敏展示值关键字，最长 128 字符 |
| `sensitiveOnly` | bool | 否 | 是否只返回敏感配置项 |
| `page` | int | 否 | 页码，默认 1 |
| `pageSize` | int | 否 | 每页条数，默认 20，最大 100 |

### 返回说明

- 数据来源为 `ServiceContext.CurrentConfig()`，用于核对当前进程热加载后实际生效的运行态配置。
- MySQL、ClickHouse、Kafka、Redis 地址、DSN、URL、秘钥、Token、密码等敏感值只保留首尾少量字符。
- 敏感配置不使用原始值参与关键字匹配，避免通过搜索接口探测秘钥原文。
- `snapshotYaml` 返回完整脱敏 YAML 快照，便于按原配置结构核对当前进程实际生效值。
- `runtimeYaml` 按 `runtime.yaml` 顶层结构返回运行期大配置视图，包含 `archive_jobs`、`task_periodic`、`workflows` 等已并入当前运行态的配置块。

### data 内部字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `keyword` | string | 当前查询关键字 |
| `sensitiveOnly` | bool | 是否只返回敏感配置项 |
| `page` | int | 当前页码 |
| `pageSize` | int | 当前页大小 |
| `total` | int64 | 命中总数 |
| `totalItems` | int | 当前运行态快照内配置项总数 |
| `sensitiveTotal` | int | 当前运行态快照内敏感配置项总数 |
| `sections` | array | 顶层配置分组统计，包含总数和敏感数 |
| `source` | object | 当前快照来源与热加载状态，包含 `restartRequired`、`restartReason` |
| `snapshotYaml` | string | 完整脱敏 YAML 快照 |
| `runtimeYaml` | string | 按 `runtime.yaml` 顶层结构生成的脱敏 YAML 视图 |
| `items` | array | 当前页配置项 |

### source 字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `source` | string | 固定为 `runtime_snapshot` |
| `configFile` | string | 当前监听的配置文件 |
| `runtimeFile` | string | 当前声明的运行期外部配置文件 |
| `configVersion` | string | 当前生效配置版本指纹 |
| `lastStatus` | string | 最近一次热加载状态 |
| `lastTriggerSource` | string | 最近一次触发来源 |
| `lastReloadAt` | string | 最近一次触发重载时间 |
| `lastSuccessAt` | string | 最近一次成功加载时间 |
| `restartRequired` | bool | 是否仍需重启进程才能完全生效 |
| `restartReason` | string | 需要重启才能完全生效的原因摘要 |

### items 字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `path` | string | 扁平化配置路径，如 `mysql.write_data_source` |
| `value` | string | 展示值；敏感项已脱敏 |
| `valueType` | string | 值类型：`string/number/bool/list/object/null` |
| `sensitive` | bool | 是否按敏感配置处理 |

## 5. 手动触发配置热加载

### 接口

```http
POST /api/tasks/config-reload
```

### 请求示例

```http
POST /api/tasks/config-reload
```

### 适用场景

- 配置文件已更新，但不想等 watcher 轮询。
- 想立即验证热加载是否生效。
- 想确认当前返回的“已成功”是否还需要重启进程补齐基础设施连接重建。

### 返回说明

- `POST /api/tasks/config-reload` 返回的 `data` 与 `GET /api/tasks/config-reload` 一致，都是最新的热加载状态快照。
- 当 `restartRequired=true` 时，不要只看“配置热加载成功”，还需要根据 `restartReason` 安排重启。

## 6. 排障建议

- “接口返回触发成功”只代表 `workflow:trigger` 已成功入队，不代表整个工作流已执行完成。
- 真正排障时，应先查 `workflowId`，看工作流卡在哪个节点。
- `lastStatus=success` 不等于“所有组件在线热更新完成”，仍要结合 `restartRequired` 一起判断。
- `workflows.user_tag.enabled` 这类插件启停项属于启动期注册开关；如果实例以 `false` 启动，热加载改成 `true` 后仍会保留运行态旧值，需要按 `restartReason` 重启对应实例。
- `suppressedFailureCount` 持续增长时，优先排查同一类重复失败，不要只反复点手动重载。
- 如果一直没有节点推进，再回头检查 `run_mode`、队列、worker 和 Redis 配置。

## 7. 响应字段说明

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| status | bool | 是 | false | 状态: true 成功, false 失败 |
| code | int | 是 | 1 | 业务状态码 |
| message | string | 否 |  | 消息 |
| data | object | 是 |  | 工作流状态或配置热加载状态 |

### 工作流 data 内部字段

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| workflowId | string | 是 |  | 工作流实例 ID |
| status | string | 是 |  | 工作流状态 |
| nodes | array | 是 | [] | 节点执行状态 |
| errorMessage | string | 否 |  | 失败摘要 |
