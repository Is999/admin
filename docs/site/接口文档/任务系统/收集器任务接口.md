# 收集器任务接口

## 1. 功能说明

通用收集器用于降低大量同类型业务“单条新增/更新 MySQL”带来的写入放大问题。业务方只投递结构化事件，不投递 SQL；Collector 负责可靠收集、批量调度和失败重试。

核心接入方式：

- 调用 `Collector.Enqueue(event)` 投递结构化 `collector.Event`。
- 调用 `Collector.RegisterProcessor(bizType, processor)` 注册批量消费器。
- 如业务只想传函数，可调用 `Collector.RegisterProcessorFunc(bizType, fn)`。
- Processor 内部自行决定如何聚合、批量 upsert、落不同表、幂等去重和返回逐事件成功/失败。

可靠性原则：

- 入队优先级：Kafka 可用先写 Kafka；Kafka 不可用写 Redis Stream；Redis 不可用写 DB outbox。
- DB outbox 是最终兜底，只有 DB outbox 写入也失败时，`Enqueue` 才返回失败。
- 投递语义是至少一次，业务 Processor 必须使用 `EventID` 或业务唯一键做幂等。
- Kafka/Redis worker 会先把事件落到 DB outbox，再提交 offset 或 ACK 消息，避免消息已确认但事件丢失。
- DB worker 使用 `SKIP LOCKED` 领取任务，支持多实例并发消费和租约超时回收。

## 2. 代码入口

| 场景 | 文件 | 方法 | 说明 |
| --- | --- | --- | --- |
| 启动组件 | `internal/bootstrap/component_builtin.go` | `newCollectorComponent()` | 创建 Collector，并在 Worker 模式启动后台消费协程 |
| 启动清单 | `internal/bootstrap/registrations.go` | `DefaultRegistrationManifest()` | 统一记录组件和路由注册项 |
| 事件类型 | `internal/collector/types.go` | `Event` / `Processor` / `ProcessorFunc` | 定义业务投递和批量消费接口 |
| 投递事件 | `internal/collector/manager.go` | `(*Manager).Enqueue(...)` | 写 Kafka/Redis/DB outbox，返回事件 ID |
| 注册处理器 | `internal/collector/manager.go` | `RegisterProcessor` / `RegisterProcessorFunc` | 按 bizType 注册批量消费逻辑 |
| 管理路由 | `internal/handler/routes_collector.go` | `registerCollectorRoutes(...)` | 注册 Collector 运维接口 |

## 3. 管理接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/collector/overview` | 查询 Collector 概览、热点业务排行与 transport 分布 |
| `GET` | `/api/collector/tasks` | 查询 Collector outbox 任务列表 |
| `POST` | `/api/collector/run` | 手动触发一轮 DB outbox 消费 |
| `POST` | `/api/collector/tasks/retry` | 对指定任务人工重试/延迟重试 |
| `GET` | `/metrics` | Prometheus 指标抓取入口，包含 Collector 批次吞吐与耗时指标 |

## 4. 任务状态

| 状态值 | 状态名 | 说明 |
| --- | --- | --- |
| `0` | 待处理 | 已写入 outbox，等待执行 |
| `1` | 处理中 | 已被 worker 领取并投送 Processor |
| `2` | 已完成 | Processor 返回成功 |
| `3` | 待重试 | Processor 返回失败或执行异常，等待 `nextRunAt` 后重试 |
| `4` | 死信 | 达到最大重试次数，等待人工处理 |

## 5. 接入示例

```go
// 注册批量消费函数，业务方在函数内部决定如何聚合、upsert、落表和幂等。
err := svcCtx.Collector.RegisterProcessorFunc("user_stat", func(ctx context.Context, events []collector.Event) ([]collector.ProcessResult, error) {
    results := make([]collector.ProcessResult, 0, len(events))
    // 方法内部逻辑说明：按业务维度聚合 events，再批量写入目标表；这里仅展示结果返回结构。
    for _, event := range events {
        results = append(results, collector.ProcessResult{
            EventID: event.EventID,
            Success: true,
        })
    }
    return results, nil
})
```

```go
// 投递结构化事件，Payload 是业务数据，不允许放 SQL。
eventID, err := svcCtx.Collector.Enqueue(ctx, collector.Event{
    EventID:      "user_stat:1:10001:20260506",
    BizType:      "user_stat",
    PartitionKey: "10001",
    Payload:      json.RawMessage(`{"uid":10001,"delta":1}`),
})
```

## 6. 查询概览

`GET /api/collector/overview`

### 请求示例

```http
GET /api/collector/overview
```

重点字段：

| 字段 | 说明 |
| --- | --- |
| `pendingCount / runningCount / retryCount / deadCount` | 当前全局状态分布 |
| `readyCount` | 已到执行时间、理论上应该尽快被消费的数量 |
| `runningTimeoutCount` | 超过 running lease 的疑似卡住任务数量 |
| `oldestReadyAgeSeconds` | 最老积压任务的等待秒数 |
| `recent1m / recent5m / recent15m` | 最近窗口成功数、失败数、平均耗时、最大耗时 |
| `bizTypeTop15m` | 最近 15 分钟热点业务排行 |
| `transportStats` | 当前 `db/kafka/redis` 来源分布 |

## 7. 查询任务

`GET /api/collector/tasks`

### 请求示例

```http
GET /api/collector/tasks?page=1&pageSize=10&bizType=user_stat&state=3
```

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | int | 否 | 页码，默认 `1` |
| `pageSize` | int | 否 | 每页数量 |
| `bizType` | string | 否 | 按业务类型过滤 |
| `transport` | string | 否 | 按来源过滤：`db/kafka/redis` |
| `state` | int | 否 | 按任务状态过滤 |

## 8. 手动执行

`POST /api/collector/run`

### 请求示例

请求体：

```json
{
  "limit": 500
}
```

说明：

- 该接口只触发 DB outbox worker 立即跑一轮。
- 若本轮有部分事件失败，失败事件会按重试策略回写，接口返回本轮成功数量和失败提示。

## 9. 人工重试

`POST /api/collector/tasks/retry`

### 请求示例

请求体：

```json
{
  "ids": [1, 2, 3],
  "delaySeconds": 0,
  "resetAttempt": true
}
```

说明：

- `ids` 为 outbox 任务 ID。
- `delaySeconds` 大于 `0` 时，会延迟到指定秒数后再允许消费。
- `resetAttempt=true` 会把失败次数重置为 `0`。
