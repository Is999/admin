# 收集器观测与失败事件接口

## 1. 功能说明

通用收集器用于降低大量同类型业务“单条新增/更新 MySQL”带来的写入放大问题。业务方只投递结构化事件，不投递 SQL；Collector 正常链路只走 Kafka，负责异步削峰、应用解耦、批量调度和投送确认。运行期观测使用内存滑动窗口和 Prometheus 指标，不把正常成功事件写入 DB。

核心接入方式：

- 调用 `Collector.Enqueue(event)` 投递结构化 `collectorx.Event`。
- 调用 `Collector.RegisterProcessor(bizType, processor)` 注册批量消费器。
- 如业务只想传函数，可调用 `Collector.RegisterProcessorFunc(bizType, fn)`。
- Processor 内部自行决定如何聚合、批量 upsert、落不同表和返回逐事件成功/失败；任务开启幂等时 Collector 才按 `BizType + EventID` 去重。

可靠性原则：

- `Enqueue` 只投递 Kafka，Kafka 写成功才返回成功。
- Kafka 写失败直接返回错误。
- `Enqueue` 按 `Event.BizType` 读取 `collector.tasks.<bizType>.topic`，未配置时继承 `collector.default_task.topic`。
- 每个 Topic 启动独立 Kafka reader；同 Topic 多任务必须使用同一个 group_id，避免重复消费。
- `EventID` 必填且必须在同 `BizType` 内稳定唯一；是否使用 Collector Redis 去重由 `collector.tasks.<bizType>.idempotency_enabled` 决定。
- 开启去重的任务会在消费侧使用 Redis `SET NX + TTL` 去重，避免业务表自行增加幂等字段。
- 未开启去重的任务不访问 Redis，不记录 duplicate 指标；调用方或业务 Processor 自行承担幂等语义。
- Redis 幂等 key 使用 `BizType + EventID` 摘要并设置 TTL，默认 1 小时回收；处理中的短租约默认 10 分钟，避免 Worker 异常退出后长期阻塞重投。
- Redis 幂等 key 只保存同任务事件状态，不保存正常事件数据；Redis 不参与正常事件投递或消息中转。
- Redis 幂等状态在 TTL 内支持进程重启、实例切换和 Kafka 重投后的重复跳过；已开启去重的任务 Redis 异常时批处理失败关闭，不提交 Kafka offset。
- Redis 操作按 `collector.idempotency.pipeline_batch_size` 分块 Pipeline；不使用跨 slot 多 key 操作，兼容 Redis 单节点和 Cluster。
- Kafka 写入 key 为 `BizType` 或 `BizType:PartitionKey`，用于在对应任务 Topic 内保证同键稳定进入同一 partition。
- Kafka worker 按 partition 建立独立顺序 lane；同 partition 内按 offset 顺序推进，不跨 offset 抢跑。
- 同 partition 内只聚合同 `BizType` 事件，达到该 `BizType` 的 `collector.tasks.<bizType>` 或 `collector.default_task` 聚合策略后调用 `Processor.ProcessBatch`。
- 同 partition 遇到不同 `BizType` 时先处理并确认当前任务批次，再开始下一个任务批次。
- 批次全成功后提交 Kafka offset。
- 已开启去重的任务遇到同 `BizType` 下重复 `EventID` 时不会再次调用 Processor，去重计数进入 Prometheus 和运行态指标。
- 批次部分失败时只把失败事件写入 `collector_failed_event`，失败账本写入成功后再提交 Kafka offset。
- 失败账本按 `BizType + EventID` 唯一，避免不同任务使用相同 EventID 时互相覆盖。
- 失败账本写入失败时不提交 Kafka offset，等待 Kafka 重投。
- 失败账本写入失败时会按当前占用 token 释放 Redis processing 状态，避免重投被错误拦截。
- Kafka offset 提交失败时不推进该 partition 后续批次，直到提交确认成功或 worker 停止。
- 投递语义是至少一次，不承诺 exactly-once。
- Collector 运行异常通过告警钩子接入任务系统 runtime alert，已注册 Lark 时会发送 Lark 预警。

## 2. 代码入口

| 场景 | 文件 | 方法 | 说明 |
| --- | --- | --- | --- |
| 启动组件 | `internal/bootstrap/components/builtin/collector.go` | `newCollector()` | 创建 Collector，并在 Worker 模式启动 Kafka 消费和失败账本重试协程 |
| 启动清单 | `internal/bootstrap/manifest` | `manifest.Default()` | 返回由 manifest 派生的组件和路由注册项 |
| 事件类型 | `internal/infra/collectorx/types.go` | `Event` / `Processor` / `ProcessorFunc` | 定义业务投递和批量消费接口 |
| 投递事件 | `internal/infra/collectorx/manager.go` | `(*Manager).Enqueue(...)` | 同步写 Kafka，返回事件 ID |
| 注册处理器 | `internal/infra/collectorx/manager.go` | `RegisterProcessor` / `RegisterProcessorFunc` | 按 bizType 注册批量消费逻辑 |
| 管理路由 | `internal/handler/collector/routes.go` | `RouteSpecs()` | 注册 Collector 运维接口 |

## 3. 管理接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/collector/overview` | 查询 Collector 运行态指标、失败账本概览与失败热点排行 |
| `GET` | `/api/collector/failures` | 查询 Collector 失败事件列表 |
| `POST` | `/api/collector/failures/run` | 手动触发一轮失败账本重试 |
| `POST` | `/api/collector/failures/retry` | 对指定失败事件人工重试/延迟重试 |
| `GET` | `/metrics` | Prometheus 指标抓取入口，包含 Collector Kafka 投递、消费、处理和失败账本指标 |

`admin_collector_kafka_consume_events_total{result="duplicate"}` 表示被 Collector 单任务 EventID 去重跳过的事件数。

## 4. 失败事件状态

| 状态值 | 状态名 | 说明 |
| --- | --- | --- |
| `0` | 待重试 | 已写入失败账本，等待重试 |
| `1` | 重试中 | 已被失败账本 worker 领取并投送 Processor |
| `2` | 重试成功 | Processor 返回成功 |
| `3` | 延迟重试 | Processor 返回失败或执行异常，等待 `nextRunAt` 后重试 |
| `4` | 死信 | 达到最大重试次数，等待人工处理 |

## 5. 接入示例

```go
// 注册批量消费函数，业务方在函数内部决定如何聚合、upsert 和落表。
err := svcCtx.Collector.RegisterProcessorFunc("user_stat", func(ctx context.Context, events []collectorx.Event) ([]collectorx.ProcessResult, error) {
    results := make([]collectorx.ProcessResult, 0, len(events))
    // 方法内部逻辑说明：按业务维度聚合 events，再批量写入目标表；这里仅展示结果返回结构。
    for _, event := range events {
        results = append(results, collectorx.ProcessResult{
            EventID: event.EventID,
            Success: true,
        })
    }
    return results, nil
})
```

```go
// 投递结构化事件，Payload 是业务数据，不允许放 SQL。
eventID, err := svcCtx.Collector.Enqueue(ctx, collectorx.Event{
    EventID:      "user_stat:1:10001:20260506",
    BizType:      "user_stat",
    PartitionKey: "10001",
    Payload:      json.RawMessage(`{"uid":10001,"delta":1}`),
})
```

## 6. 查询概览

`GET /api/collector/overview`

重点字段：

| 字段 | 说明 |
| --- | --- |
| `pendingCount / runningCount / retryCount / deadCount` | 当前失败账本状态分布 |
| `readyCount` | 已到重试时间、理论上应该尽快被消费的数量 |
| `runningTimeoutCount` | 超过 running lease 的疑似卡住失败事件数量 |
| `oldestReadyAgeSeconds` | 最老待重试失败事件的等待秒数 |
| `kafkaBatchSize` | Kafka producer 写入批次配置；业务 Processor 聚合规则以 `collector.tasks.<bizType>` 为准 |
| `failureRunnerBatchSize / failureRunnerIntervalSecs` | 失败账本重试批次与轮询间隔 |
| `runningLeaseSeconds / maxRetryTimes` | 失败账本运行租约与最大重试次数 |
| `recent1m / recent5m / recent15m` | 最近窗口重试成功数、失败数、平均耗时、最大耗时 |
| `bizTypeTop15m` | 最近 15 分钟失败热点排行 |
| `runtimeMetrics.totals` | 当前进程 Kafka 投递、Kafka 拉取、Processor 处理、死信等累计计数 |
| `runtimeMetrics.totals.duplicate` | 当前进程被 Collector 幂等去重跳过的事件数 |
| `runtimeMetrics.recent1m / recent5m / recent15m` | 当前进程最近窗口处理量、成功量、失败量、去重量、平均批次、平均耗时和最大耗时 |
| `runtimeMetrics.bizTypeTop15m` | 当前进程最近 15 分钟处理量最高的 bizType 排行 |
| `runtimeMetrics.scope` | 指标作用域，当前为 `current_process`；多实例全局趋势以 `/metrics` 的 Prometheus 汇总为准 |

## 7. 查询失败事件

`GET /api/collector/failures`

请求示例：

```http
GET /api/collector/failures?page=1&pageSize=10&bizType=user_stat&state=3
```

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | int | 否 | 页码，默认 `1` |
| `pageSize` | int | 否 | 每页数量 |
| `bizType` | string | 否 | 按业务类型过滤 |
| `state` | int | 否 | 按失败事件状态过滤 |

## 8. 手动执行失败重试

`POST /api/collector/failures/run`

请求体：

```json
{
  "limit": 500
}
```

说明：

- 该接口只触发失败账本 worker 立即跑一轮。
- 若本轮有部分事件失败，失败事件会按重试策略回写，接口返回本轮成功数量和失败提示。

## 9. 人工重试

`POST /api/collector/failures/retry`

请求体：

```json
{
  "ids": [1, 2, 3],
  "delaySeconds": 0,
  "resetAttempt": true
}
```

说明：

- `ids` 为失败事件 ID。
- `delaySeconds` 大于 `0` 时，会延迟到指定秒数后再允许重试。
- `resetAttempt=true` 会把失败次数重置为 `0`。
