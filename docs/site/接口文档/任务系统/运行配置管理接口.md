# 运行配置管理接口

## 范围

本接口只管理运行期会随业务增加的大列表配置：

- `task_periodic`
- `archive_jobs`

`workflows.user_tag` 等单一小配置仍由 YAML 管理。数据库模式下，运行态只读取 active release 快照；YAML 中同名大列表只作为首次导入种子。

## 权限与 MFA

| 权限码 | 说明 |
| --- | --- |
| `200050` | 运行配置页面入口 |
| `200051` | 列表、发布历史和快照查询 |
| `200071` | 查询运行配置概览 |
| `200052` | 保存、删除周期任务和归档任务草稿 |
| `200053` | 预检草稿 |
| `200054` | 发布草稿 |
| `200055` | 回滚发布 |
| `200056` | 导入当前运行配置 |

发布、回滚、导入必须携带 MFA 二次票据，场景号为 `12`。

## 接口列表

### 查询概览

```http
GET /api/runtime-config/overview
```

返回当前来源、运行环境、active 版本、草稿数量、当前运行态快照和全量草稿快照。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `currentSnapshot` | 当前运行态快照，仅包含 `archive_jobs` 与 `task_periodic`。 |
| `draftSnapshot` | 后端从全量草稿表生成的草稿快照，排序和发布快照一致。 |
| `draftChecksum` | 草稿快照 SHA256，与预检和发布口径一致。 |
| `draftChanged` | 草稿快照 checksum 是否不同于当前 active checksum。 |

### 周期任务草稿

```http
GET /api/runtime-config/periodic?page=1&pageSize=20
POST /api/runtime-config/periodic
DELETE /api/runtime-config/periodic/:id
```

查询支持 `keyword`、`workflow`、`enabled`。保存字段对应 `task_periodic` 单项配置，`cron` 与 `everySeconds` 二选一；`enabled` 省略时默认启用，显式 `false` 才关闭。

### 归档任务草稿

```http
GET /api/runtime-config/archive-jobs?page=1&pageSize=20
POST /api/runtime-config/archive-jobs
DELETE /api/runtime-config/archive-jobs/:id
```

查询支持 `keyword`、`database`、`enabled`。保存字段对应 `archive_jobs` 单项配置。

### 预检草稿

```http
POST /api/runtime-config/validate
```

返回草稿快照 checksum 和校验信息。预检不改变 active release，也不刷新运行态缓存。

预检执行完成均返回 HTTP 200。草稿不合法时 `valid=false`，`messages` 返回需要修正的配置项；只有读取草稿或生成快照发生服务端异常时才返回错误业务码。

通过示例：

```json
{
  "valid": true,
  "messages": ["归档任务 1 条", "周期任务 4 条"],
  "checksum": "b1dbf79448b31e09700ba3765613936264a2c23254b1270b83a9c3e6c6c2342f"
}
```

未通过示例：

```json
{
  "valid": false,
  "messages": [
    "归档任务 1 条",
    "周期任务 4 条",
    "归档任务 table_name 不能为空 name=archive-admin-log-hourly"
  ],
  "checksum": ""
}
```

### 发布草稿

```http
POST /api/runtime-config/publish
```

请求体：

```json
{
  "remark": "发布说明",
  "twoStepKey": "mfa-ticket-key",
  "twoStepValue": "mfa-ticket-value"
}
```

发布会读取草稿表，生成不可变快照，更新 `runtime_config_state`，并刷新 table-cache 精确 key。

响应示例：

```json
{
  "releaseId": 13,
  "versionNo": 13,
  "checksum": "b1dbf79448b31e09700ba3765613936264a2c23254b1270b83a9c3e6c6c2342f",
  "applied": true,
  "restartRequired": false,
  "restartReason": ""
}
```

| 字段 | 说明 |
| --- | --- |
| `applied` | 当前实例是否已经完成缓存刷新和运行态应用。数据库版本提交成功、后续应用失败时返回 `false`，避免客户端把已提交版本当成整次失败重复发布。 |
| `restartRequired` | 当前热加载完成后是否仍需重启才能完全生效。 |
| `restartReason` | 需要重启时的原因。 |

`applied=false` 表示版本已经写入数据库，不能再次提交同一发布动作。`runtime_config.source=database` 时 watcher 会按轮询周期继续应用 active release，应关注热加载状态和失败告警；`file` 模式不启动 DB watcher，数据库发布不会自动应用到当前进程。

### 发布历史与快照

```http
GET /api/runtime-config/releases?page=1&pageSize=20
GET /api/runtime-config/releases/:releaseId
```

详情返回发布快照 JSON 和 YAML。快照只包含：

```yaml
archive_jobs:
  - ...
task_periodic:
  - ...
```

### 回滚发布

```http
POST /api/runtime-config/rollback
```

请求体：

```json
{
  "releaseId": 12,
  "remark": "回滚说明",
  "twoStepKey": "mfa-ticket-key",
  "twoStepValue": "mfa-ticket-value"
}
```

回滚不会复写历史 release，而是基于目标快照创建一个新版本。

### 导入当前运行配置

```http
POST /api/runtime-config/import-current
```

只导入当前运行态的 `task_periodic` 与 `archive_jobs`，用于从 YAML 种子迁移到数据库模式。导入会直接发布新版本；DB 模式首次启动且 active release 为空时优先发布迁移种下的草稿表，草稿为空才回退当前运行态导入链路。

## 热加载与缓存

- 实例只按 `runtime_config.poll_interval_seconds` 读取 `runtime_config:state`。
- 版本和 checksum 未变化时不读取 release 快照。
- 版本变化时读取 `runtime_config:release:{releaseID}`，缓存 miss 才回源 `runtime_config_release` 单行。
- 保存草稿不刷新运行态缓存；发布、回滚、导入才刷新。
- table-cache 或 Redis 短暂异常时保留当前运行配置并记录日志。
