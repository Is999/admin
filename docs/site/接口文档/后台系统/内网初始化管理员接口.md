# 内网初始化管理员接口

## 1. 适用场景

该接口用于项目首次上线、自举恢复或运维兜底场景，在系统里已经存在超级管理员账号的前提下，将该账号重置为首次登录前状态，避免因为历史密码未知或 MFA 状态残留导致后台无法登录。

该接口具备以下特点：

- 仅允许内网或回环地址访问。
- 免 JWT，不依赖现有后台登录态。
- 同一超级管理员用户名重复调用时具备幂等语义。
- 仅允许重置既有超级管理员账号。
- 不允许新建账号。
- 不允许修改目标账号角色。

## 2. 接口地址

```http
POST /internal/auth/init-admin-bootstrap
```

### 请求示例

```http
POST /internal/auth/init-admin-bootstrap
Content-Type: application/json

{
  "username": "super999",
  "password": "Temp@1234",
  "realName": "超级管理员",
  "email": "super999@example.com",
  "phone": "13800000000",
  "description": "首次上线初始化账号"
}
```

## 3. 访问限制

- 该接口不是公网接口。
- 请求来源必须命中内网 IP 或回环地址，否则会被内网中间件直接拒绝。
- 不需要携带 `Authorization` 请求头。

### 3.1 生产运维可直接执行的 Curl

完整参数版本：

```bash
curl -sS -X POST 'http://127.0.0.1:8888/internal/auth/init-admin-bootstrap' \
  -H 'Content-Type: application/json' \
  --data-raw '{"username":"super999","password":"Temp@1234","realName":"超级管理员","email":"super999@example.com","phone":"13800000000","description":"首次上线初始化账号"}'
```

最小参数版本（仅必填字段）：

```bash
curl -sS -X POST 'http://127.0.0.1:8888/internal/auth/init-admin-bootstrap' \
  -H 'Content-Type: application/json' \
  --data-raw '{"username":"super999","password":"Temp@1234"}'
```

返回成功后重点关注：

- `code = 1`
- `data.operation = "reset"`
- `data.needResetPassword = 1`
- `data.mfaStatus = 0`

## 4. 请求参数

```json
{
  "username": "super999",
  "password": "Temp@1234",
  "realName": "超级管理员",
  "email": "super999@example.com",
  "phone": "13800000000",
  "avatar": "",
  "description": "首次上线初始化账号"
}
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `username` | 是 | 已存在的超级管理员登录用户名 |
| `password` | 是 | 首次登录临时密码，调用成功后需要由该账号自行修改 |
| `realName` | 否 | 真实姓名；显式传值时覆盖原值 |
| `email` | 否 | 邮箱；既有账号未传值时保留原值 |
| `phone` | 否 | 电话；既有账号未传值时保留原值 |
| `avatar` | 否 | 头像地址；既有账号未传值时保留原值 |
| `description` | 否 | 备注说明；既有账号未传值时保留原值 |

## 5. 处理规则

### 5.1 用户不存在

- 直接拒绝，不允许通过该接口创建管理员账号。

### 5.2 用户已存在但不是超级管理员

- 在当前写事务内通过 `admin_role_rel(user_id, role_id)` 与 `admin_role(id)` 索引点查确认启用的超级管理员角色，不读取角色缓存或副本。
- 校验不通过时直接拒绝，不允许通过该接口提升、变更或覆盖账号角色。

### 5.3 用户已存在且已经是超级管理员

- 重置该账号为首次登录前状态。
- 保留其既有超级管理员角色关系，不做任何角色调整。
- 清空既有 MFA 秘钥、MFA 登录完成标记和 MFA 二次票据缓存。
- 清空最近登录时间、登录 IP 与登录归属地。
- 登录后必须先修改密码，再按系统策略决定是否继续绑定 MFA。

### 5.4 数据库已提交但运行态清理失败

- 账号重置事务先提交，随后清理管理员登录态、权限缓存、MFA 登录标记和二次票据。
- 任一运行态清理失败时仍返回成功业务码和账号重置结果，`data.syncPending=true` 明确表示登录态或权限缓存仍待清理。
- 此时数据库重置不会回滚；运维应先恢复 Redis 并执行缓存刷新，不要重复提交账号重置请求。

## 6. 首次登录状态定义

接口执行成功后，目标账号会被统一收敛为以下状态：

- `needResetPassword = 1`
- `mfaStatus = 0`
- `mfaSecureKey = ""`
- `status = 1`
- `lastLoginTime = 零值`
- `lastLoginIP = ""`
- `lastLoginIpaddr = ""`

这保证该账号首次登录时：

1. 可以正常使用临时密码进入系统。
2. 会被要求先修改登录密码。
3. 修改密码后可继续绑定并启用 MFA。

## 7. 响应示例

```json
{
  "status": true,
  "code": 1,
  "message": "操作成功",
  "data": {
    "id": 1,
    "username": "super999",
    "roleID": 1,
    "needResetPassword": 1,
    "mfaStatus": 0,
    "status": 1,
    "operation": "reset"
  }
}
```

响应字段说明：

| 字段 | 说明 |
| --- | --- |
| `id` | 管理员 ID |
| `username` | 登录用户名 |
| `roleID` | 当前绑定的超级管理员角色 ID |
| `needResetPassword` | 是否必须修改登录密码 |
| `mfaStatus` | 当前 MFA 状态 |
| `status` | 当前账号状态 |
| `operation` | 本次操作类型；当前固定为 `reset` |

## 8. 使用建议

- 仅在首次上线、自举恢复或后台完全无法登录时使用。
- 执行成功后应立即使用返回账号登录，并完成密码修改。
- 若系统开启强制 MFA，建议在首次改密后立即完成 MFA 绑定。
- 不建议把该接口暴露给公网网关或第三方调用链路。

## 9. 首次上线操作清单

推荐按以下顺序执行：

1. 确认服务已启动，且当前调用方位于内网或宿主机本机。
2. 确认目标账号已经存在且本身就是超级管理员。
3. 通过内网地址调用 `POST /internal/auth/init-admin-bootstrap` 重置该超级管理员账号。
4. 确认响应中的 `status=1`、`needResetPassword=1`、`mfaStatus=0`。
5. 使用返回的 `username` 和本次设置的临时密码登录后台。
6. 首次登录后立即修改密码。
7. 如果系统启用了强制 MFA，继续完成 MFA 绑定与启用。
8. 登录成功并确认菜单、权限、个人中心和 MFA 状态正常后，再交付业务或运维继续使用。

## 10. 常见用法

### 10.1 首次上线重置既有超级管理员

适用场景：

- 数据库里已经存在超级管理员账号。
- 需要把该账号恢复为首次登录状态。

### 10.2 已有超级管理员账号但无法正常登录

适用场景：

- 历史密码未知。
- 旧账号已残留不可用的 MFA 状态。
- 历史登录痕迹或 MFA 票据导致登录流程异常。

此时接口会把既有账号恢复为首次登录前状态，返回的 `operation` 固定为 `reset`。

## 11. 失败排查

- 返回未授权：
  先确认请求来源 IP 是否属于内网、回环地址或宿主机本机。
- 返回参数错误：
  重点检查 `username` 是否为空、`password` 是否满足后台密码规则。
- 返回禁止访问：
  重点检查目标账号是否已经存在，且该账号当前是否本身拥有超级管理员角色。
- 返回数据库错误：
  重点检查后台管理库连接，以及角色关系查询链路是否正常。
- 调用成功但 `data.syncPending=true`：
  账号重置已经提交，但 Redis 登录态、权限缓存或 MFA 运行态尚未全部清理；恢复 Redis 后执行缓存刷新，不要重复提交账号重置请求。
- 调用成功但登录后仍异常：
  检查是否命中了浏览器本地缓存或过期 token，必要时清理浏览器本地存储后重新登录。
