# 前台用户管理接口

admin 后台直接使用默认主库管理 `user` 业务用户表，不通过 API 代理 CRUD，也不要求单独配置用户库。只有资料缓存失效、登录态失效和 API 配置热加载这类必须在 API 进程内完成的运行态能力，才调用 API 内网运维接口。

## 配置边界

| 配置项 | 说明 |
| --- | --- |
| 顶层 `mysql` | `user` 表所在默认主库，用户管理与 admin 表共用该连接 |
| `api_service.internal_base_url` | API 内网地址，仅 admin 后端调用 |
| `api_service.ops_token` | 对应 API `ops.config_reload_token`，用于 `X-Ops-Token` |
| `api_service.timeout_seconds` | API 内网调用超时，默认 5 秒，最大 30 秒 |

## 接口总览

| 方法 | 路径 | 权限码 | 说明 |
| --- | --- | --- | --- |
| `GET` | `/api/users` | `user.list` | 查询前台用户列表 |
| `GET` | `/api/users/:id` | `user.info` | 查询前台用户详情 |
| `POST` | `/api/users` | `user.add` | 新增前台用户 |
| `PATCH` | `/api/users/:id` | `user.update` | 编辑前台用户资料 |
| `PATCH` | `/api/users/status/:id` | `user.status.update` | 启用或禁用前台用户 |
| `POST` | `/api/users/password/reset/:id` | `user.password.reset` | 重置前台用户密码 |
| `POST` | `/api/users/runtime-sync/:id` | `user.runtime.sync` | 手动同步用户资料缓存或登录态 |
| `GET` | `/api/api-runtime/config-reload` | `api_runtime.config_reload.status` | 查询 API 配置热加载状态 |
| `GET` | `/api/api-runtime/config-reload/items` | `api_runtime.config_reload.items` | 查询 API 运行态配置项 |
| `POST` | `/api/api-runtime/config-reload` | `api_runtime.config_reload.run` | 手动触发 API 配置热加载 |

## 通用响应结构

后台接口统一返回 `status/code/message/data`。本页各接口只展开 `data` 的业务结构：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| status | bool | 是 | 请求是否成功 |
| code | int | 是 | 业务状态码 |
| message | string | 否 | 响应消息 |
| data | object | 否 | 当前接口业务数据 |

### UserItem 字段

列表、详情和写操作响应中的 `item` 使用同一结构：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 用户雪花 ID，后台新增时由分布式雪花算法生成；响应以字符串返回，避免前端 JS 安全整数精度丢失 |
| shardNo | int | 是 | ID 哈希分片号，来源 `CRC32(雪花 ID 字符串) % 1024`，范围 `0-1023` |
| username | string | 是 | 登录用户名 |
| nickname | string | 否 | 昵称 |
| email | string | 否 | 邮箱 |
| phone | string | 否 | 手机号 |
| avatar | string | 否 | 头像地址 |
| status | int | 是 | 用户状态：`1` 正常，`0` 禁用 |
| lastLoginAt | string | 否 | 最后登录时间 |
| lastLoginIP | string | 否 | 最后登录 IP |
| createdAt | string | 否 | 创建时间 |
| updatedAt | string | 否 | 更新时间 |

`shardNo` 主要用于后续物理分表路由、并发分片查询和分片游标查询，避免不同分片互相干扰。

### RuntimeSync 字段

新增、编辑、状态修改、重置密码和手动同步接口会返回 API 运行态同步回执：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| enabled | bool | 是 | admin 是否已配置 API 内网同步能力 |
| success | bool | 是 | 本次同步是否成功 |
| userId | string | 是 | 用户雪花 ID，响应以字符串返回，避免前端 JS 安全整数精度丢失 |
| profileCacheInvalidated | bool | 是 | 是否已处理用户资料缓存 |
| sessionsInvalidated | bool | 是 | 是否已处理登录态 |
| message | string | 否 | 同步结果说明 |

## 1. 查询前台用户列表

```http
GET /api/users?id=778919762005200896&shardNo=896&username=demo&page=1&pageSize=10&orderBy=id&order=desc
Authorization: Bearer <token>
```

查询只支持 `id`、`shardNo`、`status` 精确筛选，以及 `username`、`email`、`phone` 左前缀筛选，避免昵称模糊查询造成全表扫描。

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 否 | 用户雪花 ID 数字字符串，精确筛选 |
| shardNo | int | 否 | 用户 ID 哈希分片号，范围 `0-1023` |
| username | string | 否 | 用户名前缀 |
| email | string | 否 | 邮箱前缀 |
| phone | string | 否 | 手机号前缀 |
| status | int | 否 | `1` 正常，`0` 禁用 |
| page | int | 否 | 页码，默认 `1` |
| pageSize | int | 否 | 每页数量，默认 `10`，最大 `100` |
| orderBy | string | 否 | 单表阶段支持 `id/username/shardNo/email/phone/status/lastLoginAt/createdAt/updatedAt` |
| order | string | 否 | `asc` 或 `desc`，传入非法值会归一化为 `desc` |

### 响应示例

```json
{
  "status": true,
  "code": 1,
  "message": "查询成功",
  "data": {
    "list": [
      {
        "id": "778919762005200896",
        "shardNo": 896,
        "username": "demo_user",
        "nickname": "Demo",
        "email": "demo@example.com",
        "phone": "",
        "avatar": "",
        "status": 1,
        "lastLoginAt": "",
        "lastLoginIP": "",
        "createdAt": "2026-06-24 10:00:00",
        "updatedAt": "2026-06-24 10:00:00"
      }
    ],
    "total": 1
  }
}
```

### data 字段

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| list | UserItem[] | 是 | 当前页用户列表 |
| total | int64 | 是 | 命中总数 |

## 3. 新增前台用户

```http
POST /api/users
Content-Type: application/json

{
  "username": "demo_user",
  "password": "demoPass123",
  "nickname": "Demo",
  "email": "demo@example.com",
  "phone": "",
  "avatar": "",
  "status": 1,
  "twoStepKey": "ticket-key",
  "twoStepValue": "ticket-value"
}
```

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| username | string | 是 | 登录用户名，长度 `3-32` |
| password | string | 是 | 登录密码，长度 `8-64`，不能包含中文 |
| nickname | string | 否 | 昵称；为空时后端使用 `username` |
| email | string | 否 | 邮箱 |
| phone | string | 否 | 手机号 |
| avatar | string | 否 | 头像地址 |
| status | int | 否 | `1` 正常，`0` 禁用；未传默认正常 |
| twoStepKey | string | 是 | MFA 二次校验票据 Key |
| twoStepValue | string | 是 | MFA 二次校验票据值 |

响应 `data` 为 `UserMutationResp`：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| item | UserItem | 是 | 新增后的用户资料 |
| sync | RuntimeSync | 是 | 新增用户无既有运行态缓存，仍返回统一同步回执 |

## 4. 编辑前台用户资料

```http
PATCH /api/users/778919762005200896

{
  "nickname": "Demo",
  "email": "demo@example.com",
  "phone": "13800000000",
  "avatar": "",
  "twoStepKey": "ticket-key",
  "twoStepValue": "ticket-value"
}
```

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 路径参数，用户雪花 ID 数字字符串 |
| nickname | string | 否 | 昵称；未传表示不修改 |
| email | string | 否 | 邮箱；未传表示不修改 |
| phone | string | 否 | 手机号；未传表示不修改 |
| avatar | string | 否 | 头像地址；未传表示不修改 |
| twoStepKey | string | 是 | MFA 二次校验票据 Key |
| twoStepValue | string | 是 | MFA 二次校验票据值 |

响应 `data` 为 `UserMutationResp`。如果资料字段发生变化，admin 会调用 API 内网接口失效用户资料缓存；如果 API 内网同步未配置，接口会拒绝本次写操作。

## 5. 启用或禁用前台用户

```http
PATCH /api/users/status/778919762005200896
Content-Type: application/json

{
  "status": 0,
  "twoStepKey": "ticket-key",
  "twoStepValue": "ticket-value"
}
```

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 路径参数，用户雪花 ID 数字字符串 |
| status | int | 是 | `1` 正常，`0` 禁用 |
| twoStepKey | string | 是 | MFA 二次校验票据 Key |

## 分表路由边界

- 新增用户会同步写入 `user_account` 全局账号索引，保存 `username -> user_id/shard_no/route_shard_count`。
- `user.route_shard_count` 控制新增用户默认物理表数量，配置为 `1` 时后台列表读取单表 `user`。
- `user.route_shard_count` 提升为 `2/4/8/.../1024` 后，后台列表只支持按 `id/shardNo/username` 走 `user_account` 分页定位当前页用户；`email/phone/status` 查询或 `lastLoginAt/createdAt/updatedAt` 排序需要接入搜索或额外后台索引，接口会明确拒绝而不是扫描所有分表。
- 后续用户量增长时，物理表按 `1/2/4/8/.../1024` 阶梯拆分；表后缀表示逻辑分片范围起点，例如 2 表阶段 `user_0000/user_0512`，4 表阶段 `user_0000/user_0256/user_0512/user_0768`。
- 登录和按 ID 查询依赖 `user_account` 定位具体物理表，不能按 username 扫描所有分表。
- 历史用户迁移必须先分批回填 `user_account`，再按 `shard_no` 范围复制到目标表、校验并更新索引路由，避免生产一次性大事务。
- 从 `1 -> 2 -> 4 -> 8 -> ... -> 1024` 扩容允许不同用户批次临时处于不同 `route_shard_count`；后台列表或批量任务未来改造时必须按账号索引路由分片游标处理，不能直接扫所有物理表。
- 账号索引未全量回填、并发更新 delta 未补偿前，不允许提升 `user.route_shard_count`；该配置属于启动期写入路由，必须统一重启所有会写 `user` 的 admin/api 实例。

## 写操作安全

- 新增、编辑、状态修改、重置密码、手动同步和 API 热加载均要求 MFA 二次票据。
- 前台用户管理使用 `MFAScenarioUserManage = 13`。
- API 热加载使用 `MFAScenarioAPIRuntimeManage = 14`。
- 密码使用 API 现有 bcrypt 规则，不复用 admin 管理员密码加盐规则。
- 编辑资料、禁用用户、重置密码后，admin 会调用 API 内网接口同步资料缓存或失效登录态；如果 API 内网配置缺失，相关写操作会拒绝执行。
