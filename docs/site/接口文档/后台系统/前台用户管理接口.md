# 前台用户管理接口

admin 后台直接使用默认主库管理 `user` 业务用户表，不通过 API 代理 CRUD，也不要求单独配置用户库。只有资料缓存失效、登录态失效和 API 配置热加载这类必须在 API 进程内完成的运行态能力，才调用 API 内网运维接口。

## 配置边界

| 配置项 | 说明 |
| --- | --- |
| 顶层 `mysql` | `user` 表所在默认主库，用户管理与 admin 表共用该连接 |
| `api_service.internal_base_url` | API 内网地址，仅 admin 后端调用 |
| `api_service.ops_token` | 对应 API `ops.config_reload_token`，用于 `X-Ops-Token` 和 API 内网 HMAC 签名 |
| `api_service.timeout_seconds` | API 内网调用超时，默认 5 秒，最大 30 秒 |

| 方法 | 路径 | 权限码 | 说明 |

| `PATCH` | `/api/users/:id` | `user.update` | 编辑前台用户资料 |
| `PATCH` | `/api/users/status/:id` | `user.status.update` | 启用或禁用前台用户 |

## 通用响应结构

后台接口统一返回 `status/code/message/data`。本页各接口只展开 `data` 的业务结构：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| status | bool | 是 | 请求是否成功 |
| code | int | 是 | 业务状态码 |
| message | string | 否 | 响应消息 |
| data | object | 否 | 当前接口业务数据 |

### UserItem 字段

列表、详情和写操作响应中的 `item` 使用同一结构：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 用户雪花 ID，后台新增时由分布式雪花算法生成；响应以字符串返回，避免前端 JS 安全整数精度丢失 |
| shardNo | int | 是 | ID 哈希分片号，来源 `CRC32(雪花 ID 字符串) % 1024`，范围 `0-1023` |
| username | string | 是 | 登录用户名 |
| nickname | string | 否 | 昵称 |
| emailMasked | string | 否 | 邮箱脱敏展示值 |
| phoneMasked | string | 否 | 手机号脱敏展示值 |
| avatar | string | 否 | 头像地址 |
| status | int | 是 | 用户状态：`1` 正常，`0` 禁用 |
| lastLoginAt | string | 否 | 最后登录时间 |
| lastLoginIP | string | 否 | 最后登录 IP |
| createdAt | string | 否 | 创建时间 |
| updatedAt | string | 否 | 更新时间 |

`shardNo` 主要用于后续物理分表路由、并发分片查询和分片游标查询，避免不同分片互相干扰。

### RuntimeSync 字段

新增、编辑、状态修改、重置密码和手动同步接口会返回 API 运行态同步回执：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| enabled | bool | 是 | admin 是否已配置 API 内网同步能力 |
| success | bool | 是 | 本次同步是否成功 |
| userId | string | 是 | 用户雪花 ID，响应以字符串返回，避免前端 JS 安全整数精度丢失 |
| profileCacheInvalidated | bool | 是 | 是否已处理用户资料缓存 |
| sessionsInvalidated | bool | 是 | 是否已处理登录态 |
| authVersion | string | 是 | 本次会话失效使用的已提交认证版本，以十进制字符串返回；未处理会话时为 `"0"` |
| message | string | 否 | 同步结果说明 |

## 1. 查询前台用户列表

```http
GET /api/users?id=778919762005200896&shardNo=896&username=demo&page=1&pageSize=10&orderBy=id&order=desc
Authorization: Bearer <token>
```

查询只支持 `id`、`shardNo`、`status` 精确筛选、`username` 左前缀筛选，以及完整 `email`、`phone` 精确匹配。邮箱和手机号不会明文落库，后台会按完整输入计算 HMAC 后匹配 `email_hash/phone_hash` 或身份表 `identity_hash`。

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 否 | 用户雪花 ID 数字字符串，精确筛选 |
| shardNo | int | 否 | 用户 ID 哈希分片号，范围 `0-1023` |
| username | string | 否 | 用户名前缀 |
| email | string | 否 | 完整邮箱，精确匹配 |
| phone | string | 否 | 完整手机号，精确匹配 |
| status | int | 否 | `1` 正常，`0` 禁用 |
| page | int | 否 | 页码，默认 `1` |
| pageSize | int | 否 | 每页数量，默认 `10`，最大 `100` |
| cursorId | string | 否 | 分表阶段翻页游标；第一页不传，后续页使用上一页 `meta.nextCursorId` |
| orderBy | string | 否 | 单表阶段支持 `id/username/shardNo/email/phone/status/lastLoginAt/createdAt/updatedAt`；分表阶段仅支持 `id` |
| order | string | 否 | `asc` 或 `desc`，传入非法值会归一化为 `desc` |

```json
{
  "status": true,
  "code": 1,
  "message": "查询成功",
  "data": {
    "list": [
      {

        "avatar": "",
        "status": 1,
        "lastLoginAt": "",
        "lastLoginIP": "",
        "createdAt": "2026-06-24 10:00:00",
        "updatedAt": "2026-06-24 10:00:00"
      }
    ],
    "total": 1,
    "meta": {
      "exactTotal": false,
      "hasMore": false,
      "nextCursorId": "0",
      "statusFilterSupported": false

  }
}
```

### data 字段

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| list | UserItem[] | 是 | 当前页用户列表 |
| total | int64 | 是 | 单表阶段为精确总数；分表阶段为已浏览行数加下一页占位，用于分页控件且不执行实时 `COUNT` |
| meta.exactTotal | bool | 否 | 是否精确总数；分表阶段为 `false` |
| meta.hasMore | bool | 否 | 分表阶段是否仍有下一页 |
| meta.nextCursorId | string | 否 | 分表阶段下一页主键游标，末页为 `"0"` |
| meta.statusFilterSupported | bool | 是 | 当前列表路径是否支持状态筛选；物理分表阶段为 `false` |

## 2. 异步导出前台用户列表

```http
POST /api/users/exports
Content-Type: application/json

{
  "username": "demo",
  "email": "",
  "phone": "",
  "status": 1
}
```

导出筛选条件与列表页一致。导出的邮箱、手机号仅使用 `emailMasked`、`phoneMasked` 脱敏展示值，不导出明文、密文或 HMAC。导出提交阶段不执行大表 `COUNT` 或 `OFFSET`；后台任务通过身份索引表按 `user_id` 游标定位物理用户表，运行中 `total=0` 表示总量未知，完成后回写实际处理行数。

单个 Excel 文件的数据行数由 `user.export_split_rows` 控制，默认 `500000`。超过阈值时后台按顺序拆成多个文件，文件名固定带 `part_0001`、`part_0002` 编号；每个文件生成并上传后会立即出现在 `files` 列表中，前端可按 `files[].partNo` 立即下载，后续文件继续在后台生成。

同一归一化筛选条件下，排队中或执行中的任务会继续复用；已完成结果仅在完成后 `20` 分钟内复用，超过该窗口再次提交会创建新任务并重新读取数据。任务状态和导出文件从完成时间起保留 `24` 小时，进度查询、复用授权和下载均不会延长保留时间。

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 否 | 用户雪花 ID 数字字符串，精确筛选 |
| shardNo | int | 否 | 用户 ID 哈希分片号，范围 `0-1023` |
| username | string | 否 | 用户名前缀 |
| email | string | 否 | 完整邮箱，精确匹配；请求字段级加密 |
| phone | string | 否 | 完整手机号，精确匹配；请求字段级加密 |
| status | int | 否 | `1` 正常，`0` 禁用；分表阶段不支持按状态筛选 |

### 提交响应 data

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| jobId | string | 是 | 导出任务 ID |
| taskId | string | 否 | Asynq 任务 ID |
| queue | string | 是 | 实际投递队列 |
| status | string | 是 | 初始状态：`queued/running/succeeded/failed` |

### 查询导出进度

```http
GET /api/users/exports/status/{jobId}
Authorization: Bearer <token>
```

响应 `data` 字段：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| jobId | string | 是 | 导出任务 ID |
| taskId | string | 否 | Asynq 任务 ID |
| queue | string | 是 | 队列名称 |
| status | string | 是 | `queued/running/succeeded/failed` |
| progress | int | 是 | 进度百分比；总量统计和导出阶段为 `0-99`，完成后为 `100` |
| processed | int64 | 是 | 已处理行数 |
| total | int64 | 是 | 总行数；worker 开始时按导出条件统计，完成后校正为实际导出行数 |
| estimatedSeconds | int64 | 是 | 按全局平均处理速度计算的预计剩余秒数；总量统计阶段和完成后为 `0` |
| fileName | string | 否 | 当前可下载文件名；多文件时为首个已就绪文件的兼容字段 |
| downloadReady | bool | 是 | 是否存在至少一个可下载文件 |
| files | UserExportFileItem[] | 是 | 已生成的导出文件列表，按 `partNo` 升序返回 |
| partCount | int | 是 | 已生成文件数量 |
| splitRows | int64 | 是 | 当前任务使用的单文件数据行拆分阈值 |
| errorMessage | string | 否 | 失败原因 |
| downloadUrl | string | 否 | 当前可下载文件地址；多文件时为首个已就绪文件的兼容字段 |
| createdAt/startedAt/finishedAt/updatedAt | string | 否 | 任务时间 |
| lastProcessedAt | string | 否 | 最近进度更新时间 |
| averageRowsPerSec | int64 | 否 | 平均导出速度 |

`UserExportFileItem`：

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| partNo | int | 是 | 文件编号，从 `1` 递增，用于确认下载顺序 |
| fileName | string | 是 | 下载文件名，包含 `part_0001` 形式的固定宽度编号 |
| rowCount | int64 | 是 | 当前文件数据行数，不含表头 |
| processedFrom/processedTo | int64 | 是 | 当前文件覆盖的全局处理行序号 |
| fileSize | int64 | 是 | 文件字节数 |
| downloadReady | bool | 是 | 当前文件是否可下载 |
| downloadUrl | string | 是 | 当前文件下载地址，包含 `partNo` 查询参数 |
| createdAt | string | 是 | 当前文件生成时间 |

### 下载导出文件

```http
GET /api/users/exports/download/{jobId}?partNo=1
Authorization: Bearer <token>
```

已生成对应文件且当前管理员为发起人或同条件复用授权人时返回 `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` 文件流。`partNo` 为可选参数，建议前端始终使用 `files[].partNo` 精确下载；不传时后端会选择首个已就绪文件。任务完成 `24` 小时后状态和文件过期，需重新提交导出。

## 6. 启用或禁用前台用户

```http
PATCH /api/users/status/778919762005200896
Content-Type: application/json

{
  "status": 0,
  "twoStepKey": "ticket-key",
  "twoStepValue": "ticket-value"
}
```

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 路径参数，用户雪花 ID 数字字符串 |
| status | int | 是 | `1` 正常，`0` 禁用 |
| twoStepKey | string | 是 | MFA 二次校验票据 Key |
| twoStepValue | string | 是 | MFA 二次校验票据值 |

响应 `data` 为 `UserMutationResp`。状态变化会先在用户物理表内原子更新 `status` 并递增 `auth_version`，提交后再携带新版本通知 API 清理资料缓存和全部登录态。API 同步失败时数据库状态与认证版本仍是请求期最终失败关闭边界，响应通过 `sync.success=false` 返回可重试告警，不会把已提交写入误报为失败。

## 7. 重置前台用户密码

```http
POST /api/users/password/reset/778919762005200896
Content-Type: application/json

{
  "password": "newPass123",
  "twoStepKey": "ticket-key",
  "twoStepValue": "ticket-value"
}
```

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 路径参数，用户雪花 ID 数字字符串 |
| password | string | 是 | 新密码，长度 `8-64`，不能包含中文 |

## 8. 手动同步用户 API 运行态

```http
POST /api/users/runtime-sync/778919762005200896
Content-Type: application/json

{
  "profile": true,
  "sessions": true,
  "twoStepKey": "ticket-key",
  "twoStepValue": "ticket-value"
}
```

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 路径参数，用户雪花 ID 数字字符串 |
| profile | bool | 否 | 是否失效用户资料缓存 |
| sessions | bool | 否 | 是否失效登录态 |
| twoStepKey | string | 是 | MFA 二次校验票据 Key |
| twoStepValue | string | 是 | MFA 二次校验票据值 |

`profile` 和 `sessions` 都不传或都为 `false` 时，后端默认按 `profile=true` 处理。`sessions=true` 时 admin 会先递增并提交用户表 `auth_version`，再把新版本传给 API 原子清理全部登录态；同步失败会在回执中明确提示重试。响应 `data` 为 `RuntimeSync`。

## 9. 查询 API 配置热加载状态

```http
GET /api/api-runtime/config-reload
Authorization: Bearer <token>
```

该接口由 admin 后端代理 API 内网运维接口，前端不直接访问 API 服务；admin 后端会自动追加 `X-Ops-Token`、`X-Ops-Timestamp`、`X-Ops-Body-SHA256` 和 `X-Ops-Signature`。

### data 字段

| 字段 | 类型 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| connected | bool | 是 | 是否已配置并成功访问 API 内网接口 |
| status | object | 否 | API 返回的热加载状态快照；结构同 API `GET /internal/system/config-reload/status` |
| message | string | 否 | 调用结果说明 |

`status` 常用字段包括：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| enabled | bool | 是否启用配置热加载 |
| watching | bool | 是否正在后台监听配置 |
| configFile | string | 当前监听的配置文件路径 |
| configVersion | string | 当前生效配置版本指纹 |
| restartRequired | bool | 热加载后是否仍需重启进程才能完全生效 |
| restartReason | string | 需要重启的原因摘要 |
| lastStatus | string | 最近一次处理结果：`idle/success/failed` |
| lastMessage | string | 最近一次处理结果说明 |
| lastTriggerSource | string | 最近一次触发来源 |
| lastReloadAt | string | 最近一次触发重载时间 |
| lastSuccessAt | string | 最近一次成功加载时间 |

## 10. 查询 API 运行态配置项

```http
GET /api/api-runtime/config-reload/items?keyword=mysql&sensitiveOnly=false&page=1&pageSize=20
```

## 分表路由边界

- 新增用户会按身份类型同步写入 `user_identity_username/user_identity_email/user_identity_phone/user_identity_oauth` 登录身份索引：用户名保存 `identity_value`，邮箱和手机号只保存 HMAC `identity_hash`，三方登录保存大小写敏感的 `provider/identity_value`；四张表都保存 `user_id/user_shard_no/user_route_shard_count`。
- `user.route_shard_count` 控制新增用户默认物理表数量，配置为 `1` 时后台列表读取单表 `user`。
- `user.route_shard_count` 提升为 `2/4/8/.../1024` 后，后台列表走登录身份索引定位当前批用户：`username` 走 `user_identity_username.identity_value` 前缀，`email/phone` 走对应身份表 `identity_hash` 精确匹配；分表阶段按 `user_id` 游标翻页，不执行实时 `COUNT` 或深分页，`status` 查询及非 ID 排序会明确拒绝而不是扫描所有分表。前端首次成功响应收到 `statusFilterSupported=false` 后会清空并禁用状态筛选。
- 后续用户量增长时，物理表按 `1/2/4/8/.../1024` 阶梯拆分；表后缀表示逻辑分片范围起点，例如 2 表阶段 `user_0000/user_0512`，4 表阶段 `user_0000/user_0256/user_0512/user_0768`。
- 登录先按身份类型选择对应身份表，再按 `identity_value`、`identity_hash` 或 `provider/identity_value` 定位身份索引，最后通过 `user_shard_no/user_route_shard_count` 定位具体物理用户表，不能按 username/email/phone 扫描所有分表。
- 历史用户迁移必须先分批回填四张登录身份索引表，再按 `shard_no` 范围复制到目标表、校验并更新 `user_route_shard_count`，避免生产一次性大事务。
- 从 `1 -> 2 -> 4 -> 8 -> ... -> 1024` 扩容允许不同用户批次临时处于不同 `user_route_shard_count`；后台列表或批量任务未来改造时必须按登录身份索引路由分片游标处理，不能直接扫所有物理表。
- 登录身份索引未全量回填、并发更新 delta 未补偿前，不允许提升 `user.route_shard_count`；该配置属于启动期写入路由，必须统一重启所有会写 `user` 的 admin/api 实例。

## 写操作安全

- 新增、编辑、状态修改、重置密码、手动同步和 API 热加载均要求 MFA 二次票据。
- 前台用户管理使用 `MFAScenarioUserManage = 13`。
- API 热加载使用 `MFAScenarioAPIRuntimeManage = 14`。
- API 用户和后台管理员密码都直接使用标准 bcrypt；随机盐由 bcrypt 哈希内部生成和保存，不再叠加旧 MD5 派生盐。
- 编辑资料依赖 API 资料缓存同步，内网配置缺失时会在落库前拒绝；状态修改、重置密码和手工会话失效则先在主库提交新的 `auth_version`，再调用 API 清理登录态，内网配置缺失或调用失败时通过 `sync.success=false` 返回可重试告警，不阻断紧急禁用或改密。
