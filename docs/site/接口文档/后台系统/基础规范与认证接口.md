# 基础规范与认证接口

本文档覆盖 admin 后台系统管理接口的基础规范与认证部分，接口风格参考 laravel-admin，统一返回：

```json
{
  "status": true,
  "code": 1,
  "message": "操作成功",
  "data": {}
}
```

## 鉴权

除登录接口外，后台系统接口均需要请求头：

| 请求头 | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 是 | `Bearer <token>` |
| `Accept-Language` | 否 | 前端当前语言 |
| `X-Trace-Id` | 否 | 链路追踪 ID |
| `X-Timestamp` | 开启签名后是 | 秒级 Unix 时间戳 |

### 获取验证码

#### 请求示例

```http
GET /api/auth/captcha
```

#### 响应示例

```json
{
  "status": true,
  "code": 1,
  "message": "查询成功",
  "data": {
    "captchaId": "captcha_xxx",
    "captchaImage": "data:image/png;base64,..."
  }
}
```

#### data 内部字段

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| captchaId | string | 是 |  | 验证码 ID |
| captchaImage | string | 是 |  | Base64 图片内容 |

### 登录

`POST /api/auth/login`

请求：

```json
{
  "username": "admin",
  "password": "123456"
}
```

响应 `data`：

```json
{
  "token": "jwt-token",
  "user": {
    "id": 1,
    "username": "admin",
    "realName": "管理员",
    "needResetPassword": 0,
    "mfaStatus": 0,
    "existMFA": false,
    "buildMFAURL": "otpauth://totp/...",
    "forceMFAEnabled": true,
    "mfaBindRequired": true,
    "mfaCheck": 1,
    "frequency": 120
  }
}
```

### 业务码约定（与登录相关）

| 业务码 | 含义 | 说明 |
| --- | --- | --- |
| `6` | 需要校验 MFA 动态码 | 已启用账号的登录态 MFA 校验 |
| `5` | 需要先绑定并启用 MFA | 当前登录需要 MFA，且账号未启用或没有可用 MFA 秘钥时返回，用于区分“绑定”与“仅校验动态码” |
| `7` | 需要先修改登录密码 | `needResetPassword=1` 时优先引导改密 |
| `8` | MFA 校验已过期，需要重新验证 | 敏感操作的二次校验票据失效 |

### 登录态初始化

`GET /api/auth/profile`

响应 `data` 包含当前管理员资料：`userId`、`username`、`realName`、`email`、`phone`、`avatar`、`roles`、`token` 等。

### 内网管理员自举

`POST /internal/auth/init-admin-bootstrap`

- 该接口仅用于首次上线或运维兜底时重置既有超级管理员账号。
- 免 JWT，但要求来源 IP 命中内网白名单。
- 不允许新建账号，也不允许修改目标账号角色。
- 成功后会确保目标账号处于“首次登录状态”，可先改密，再继续绑定 MFA。
- 详细说明见 [内网初始化管理员接口](接口文档/后台系统/内网初始化管理员接口.md)。

### 权限码

`GET /api/auth/codes`

响应 `data` 为当前用户拥有的权限标识数组，例如：

```json
["admin.list", "role.list", "permission.list"]
```

### 刷新与退出

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/auth/refresh` | 刷新访问令牌 |
| `POST` | `/api/auth/logout` | 退出登录 |

### 健康检查

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/live` | 存活检查，不访问外部依赖 |
| `GET` | `/api/ready` | 就绪检查，探测 MySQL / Redis / Kafka / TaskQueue / Collector |
| `GET` | `/api/metrics` | Prometheus 指标抓取入口 |

- 公开探活接口，不要求 JWT。
- 容器 `livenessProbe` 使用 `/api/live`，`readinessProbe` 使用 `/api/ready`。
- Prometheus 使用 `/api/metrics` 抓取指标，不要求 JWT。
- 访问日志默认被白名单屏蔽，探针高频访问时不会刷出 access log。
- 联调时应以响应状态和响应体为准，不要以“日志是否出现”作为可用性判断依据。

## 签名与加密

后端支持 `X-App-Id`、`X-Trace-Id`、`X-Timestamp`、`X-Signature`、`X-Crypto`、`X-Cipher`，并在版本化秘钥模式下额外支持 `X-Key-Version`、`X-Gray-Key`。后台接口是否执行签名验签、加密解密，统一由路由安全策略和秘钥管理中的有效版本决定，不再通过 `sys_config` 全局开关短路。

| 请求头 | 说明 |
| --- | --- |
| `X-App-Id` | `base64(appID)`，后端按 `secret_key.uuid` 查找密钥 |
| `X-Trace-Id` | 请求追踪 ID，参与签名和防重放；开启签名后必填，前后端必须保持同一值 |
| `X-Timestamp` | 秒级 Unix 时间戳，参与签名并校验 5 分钟时间窗口；开启签名后必填 |
| `X-Signature` | 签名方式：`M`=MD5，`A`=AES，`R`=RSA |
| `X-Crypto` | 加密方式：`A`=AES，`R`=RSA |
| `X-Cipher` | 字段级加密列表，值为 base64(JSON 字段数组)，不再支持 `cipher` 整包加密 |
| `X-Key-Version` | 可选，显式指定当前请求使用的秘钥版本；未传时按稳定/灰度规则选路 |
| `X-Gray-Key` | 可选，灰度分桶键；用于 `grayVersion + grayPercent` 的稳定路由 |

### 签名规则

待签名字符串生成规则：

1. 按签名字段 ASCII 排序。
2. 跳过不存在、`null`、空字符串字段。
3. 拼接 `key=value&`。
4. 追加 `key=md5(appID + "-" + traceID + "-" + timestamp)`。
5. 按 `X-Signature` 指定方式生成 `sign` 字段。

签名与加解密的最终实现约束：

1. 加密字段是动态的，由 `X-Cipher` 决定。
2. 签名字段是固定的，由路由安全策略决定，不能通过请求头动态改写。
3. 响应处理顺序固定为“先按 `X-Cipher` 解密，再按固定签名字段验签”。
4. 签名字段的首层顺序必须按 ASCII 排序。
5. 若签名字段值本身是对象或数组，前后端都必须使用递归稳定序列化：
   - 对象的每一层 key 都按字典序输出；
   - 数组保持原顺序；
   - 禁止直接依赖普通 `JSON.stringify` / `json.Marshal` 的默认对象遍历顺序参与签名。
6. 为降低跨端序列化差异风险，登录、刷新令牌等核心鉴权接口的响应签名应优先只覆盖稳定首层标量字段，例如 `token`，避免直接对复杂对象整体回签。

说明：

- 调整某个接口的加密字段，只会影响 `X-Cipher` 和对应解密流程，不应影响签名/验签规则。
- 调整某个接口的签名字段，需要前后端同步更新该路由的固定安全策略，但不通过请求头动态协商。

### 版本化秘钥选路

1. 若显式传入 `X-Key-Version`，后端优先使用该版本。
2. 未显式传版本时，默认命中 `secret_key.stable_version`。
3. 当 `gray_version` 非空且 `gray_percent > 0` 时，后端按 `X-Gray-Key` 做稳定分桶，命中灰度版本后会在响应头回写最终版本到 `X-Key-Version`。
4. 秘钥缓存拆分为 `secret_key_route:{uuid}` 与 `secret_key_aes:{uuid}:{keyVersion}` / `secret_key_rsa:{uuid}:{keyVersion}`，并通过 `secret_key_version:index:{uuid}` 精确索引版本材料缓存，避免轮换时整组缓存互相污染，也避免生产 Redis 使用前缀 `SCAN`。

## 统一响应字段说明

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| status | bool | 是 | false | 状态: true 成功, false 失败 |
| code | int | 是 | 1 | 业务状态码 |
| message | string | 否 |  | 消息 |
| data | object/array/null | 否 |  | 登录信息、权限码、健康检查状态或空对象 |
