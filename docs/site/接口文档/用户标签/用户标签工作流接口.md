# 用户标签工作流接口

## 1. 适用菜单

- 后台菜单：`用户标签`

当前接口只触发用户标签骨架工作流。`tagTypes` 是不透明标签类型编号，默认实现不会按编号计算具体标签。业务计算阶段实现前只允许 `dryRun=true`，服务端会在任何正式结果表操作前拒绝非 dry-run 请求。

## 2. 公网触发入口

### 接口

```http
POST /api/user-tags/workflows
```

### 字段说明

| 字段                 | 类型      | 必填 | 说明                                           |
|--------------------|---------|----|----------------------------------------------|
| `mode`             | string  | 否  | `full/delta/targeted/recalculate`，默认 `full`  |
| `tagTypes`         | int[]   | 否  | 指定标签类型；`targeted/recalculate` 使用，`full` 禁止传入 |
| `uids`             | string[] | 否 | 指定用户 UID 十进制字符串；仅 `targeted` 使用，避免前端大整数精度丢失 |
| `queue`            | string  | 否  | 指定任务队列                                       |
| `shardTotal`       | int     | 否  | 分片数                                          |
| `batchSize`        | int     | 否  | 批次大小                                         |
| `workerCount`      | int     | 否  | 节点内部 worker 数                                |
| `dryRun`           | bool    | 是  | 当前骨架阶段必须为 `true`，只验证链路不落正式结果                    |
| `uniqueKey`        | string  | 否  | 去重键                                          |
| `uniqueTTLSeconds` | int     | 否  | 去重 TTL 秒                                     |
| `retry`            | int     | 否  | 重试次数，范围 `0-25`                           |
| `timeoutSeconds`   | int     | 否  | 超时秒数，范围 `1-86400`                        |

## 3. 硬性约束

- `mode=targeted` 时，必须提供 `uids`，并且必须提供 `tagTypes`。
- `mode=recalculate` 时，必须提供 `tagTypes`。
- `mode=full` 是全站切表重建，不允许提供 `uids/tagTypes`；指定 UID 请使用 `targeted`，指定标签请使用 `recalculate`。
- `retry` 最大 `25`，`timeoutSeconds` 最大 `86400`；前端限制不能替代服务端校验。
## 4. 请求示例

### 4.1 full 骨架触发

```json
{
  "mode": "full",
  "queue": "maintenance",
  "shardTotal": 1,
  "batchSize": 2000,
  "workerCount": 4,
  "dryRun": true,
  "uniqueKey": "user_tag:full:manual:20260523",
  "uniqueTTLSeconds": 3600,
  "timeoutSeconds": 3600
}
```

### 4.2 delta 骨架触发

```json
{
  "mode": "delta",
  "queue": "maintenance",
  "shardTotal": 1,
  "batchSize": 2000,
  "workerCount": 4,
  "dryRun": true,
  "uniqueKey": "user_tag:delta:manual:20260523:01",
  "uniqueTTLSeconds": 1800,
  "timeoutSeconds": 1800
}
```

### 4.3 targeted 骨架触发

```json
{
  "mode": "targeted",
  "tagTypes": [1, 2],
  "uids": ["10001", "10002", "10003"],
  "queue": "maintenance",
  "shardTotal": 1,
  "batchSize": 500,
  "workerCount": 2,
  "dryRun": true
}
```

## 5. 成功响应示例

```json
{
  "status": true,
  "code": 1,
  "message": "触发成功",
  "data": {
    "taskId": "task_xxx",
    "workflowId": "wf_xxx",
    "workflowName": "user_tag.full.rebuild",
    "queue": "maintenance"
  }
}
```

## 6. 响应字段说明

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| status | bool | 是 | false | 状态: true 成功, false 失败 |
| code | int | 是 | 1 | 响应状态码 |
| message | string | 否 |  | 消息 |
| data | object | 是 |  | 工作流触发结果 |

### data 内部字段

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| taskId | string | 是 |  | 任务 ID |
| workflowId | string | 是 |  | 工作流实例 ID |
| workflowName | string | 是 |  | 工作流名称 |
| queue | string | 是 |  | 任务队列 |

## 7. 说明

- `tagTypes` 在 `delta` 中可选；默认骨架不会解释具体编号。
- 默认 `uniqueKey` 会按规范化计算参数生成指纹，完全相同的请求在去重 TTL 内会被去重；不同 UID、标签、分片、批次或执行参数不会互相阻塞。
- `targeted/recalculate/delta` 不持有全局写锁，允许多请求并发；`full` 持有全局租约，保护结果表切换。

## 8. 工作流互斥锁释放入口

### 接口

```http
POST /api/user-tags/workflow-lease/release
```

### RouteMeta / 权限 / 审计

| 项      | 值                                                                            |
|--------|------------------------------------------------------------------------------|
| 路由别名   | `user_tag.workflow_lease.release`                                            |
| 权限码    | `200048`                                                                     |
| 审计动作   | `释放用户标签工作流互斥锁`                                                               |
| MFA 场景 | `MFAScenarioUserTagLeaseRelease = 10`，系统强制 MFA 时需要 `twoStepKey/twoStepValue` |
| 请求签名字段 | `workflowId, mode, reason, twoStepKey, twoStepValue`                         |

### 字段说明

| 字段             | 类型     | 必填 | 说明                                                            |
|----------------|--------|----|---------------------------------------------------------------|
| `workflowId`   | string | 是  | 需要释放锁的工作流实例 ID                                                |
| `mode`         | string | 否  | `full/delta/targeted/recalculate`，默认 `full`；释放时会参与 owner 精确匹配 |
| `reason`       | string | 是  | 人工释放原因，最多 500 个字符，会进入审计日志                                     |
| `twoStepKey`   | string | 否  | MFA 二次校验票据 key                                                |
| `twoStepValue` | string | 否  | MFA 二次校验票据 value                                              |

### 请求示例

```json
{
  "workflowId": "ca9c6ad3-9dc7-40ad-9ca2-29e9fce8290b",
  "mode": "full",
  "reason": "prepare 已失败，确认无运行中 worker、未进入 finalize/dispatch_hooks，释放后重新触发 full",
  "twoStepKey": "mfa-ticket-key",
  "twoStepValue": "mfa-ticket-value"
}
```

### 成功响应示例

```json
{
  "status": true,
  "code": 1,
  "message": "用户标签工作流互斥锁已释放",
  "data": {
    "workflowId": "ca9c6ad3-9dc7-40ad-9ca2-29e9fce8290b",
    "mode": "full",
    "owner": "ca9c6ad3-9dc7-40ad-9ca2-29e9fce8290b|full",
    "leaseKey": "user_tag:workflow:write_lock:101",
    "ttlSeconds": 3520,
    "released": true,
    "releasedAt": "2026-05-14T10:00:00+08:00",
    "reason": "prepare 已失败，确认无运行中 worker、未进入 finalize/dispatch_hooks，释放后重新触发 full"
  }
}
```

### 状态码

| code    | 说明                                       |
|---------|------------------------------------------|
| `43001` | 租约不存在或已释放                                |
| `43002` | 当前 Redis owner 与请求 `workflowId/mode` 不匹配 |
| `43003` | 释放过程发生未知失败                               |

### 操作约束

- 该接口只访问 `user_tag:workflow:write_lock:<app_id>` 和当前 workflow 的
  `user_tag:workflow:final_done:<app_id>:<workflowId>`，不执行 Redis `SCAN` 或通配删除。
- 释放前必须确认原 workflow 已失败或终止、没有运行中 worker，且没有处于 `finalize` 切表或 `dispatch_hooks` 事件派发窗口。
- 如果只是归档失败任务需要补跑，优先在任务列表对失败任务点击“立即执行”；新版本会先修复工作流节点失败计数，再让 DAG 继续推进。
