# 前端权限码与 admin_permission 映射说明

## 适用范围

- `admin-cron`
- `admin-cron-vue`
- 前端、测试、联调同学

## 核心结论

- 前端真正用于鉴权的是后端 `/api/auth/codes` 返回的权限码字符串数组。
- 这些权限码字符串来自 `admin_permission.uuid`，不是 `admin_permission.module`。
- 前端路由 `meta.authority`、按钮 `v-access`、表格操作列 `auth`，最终都拿这个 `uuid` 字符串做匹配。

## `roles` / `roleIds` / `accessCodes` 的职责边界

### `roles`

- 表示当前账号启用角色的名称列表
- 主要用于前端布局展示，以及兼容 Vben 路由过滤时传入的 `roles`
- 不要用 `roles` 直接推断按钮、表格操作列或接口权限

### `roleIds`

- 表示当前账号启用角色的 ID 列表
- 主要用于识别特殊角色身份，例如当前项目约定角色 ID `1` 为超级管理员
- 不直接作为页面按钮显隐匹配值使用

### `accessCodes`

- 表示后端 `/api/auth/codes` 返回的权限码字符串数组
- 实际内容来源于 `admin_permission.uuid`
- 前端页面、按钮、表格和运行时权限判断都应优先使用它

### 当前项目推荐理解

- `roles`：给路由过滤兼容和展示层使用
- `roleIds`：给身份判定使用，例如超级管理员识别
- `accessCodes`：给真正的前端权限显隐与运行时判断使用

## 后端返回链路

### 角色权限如何变成 `/auth/codes`

1. 角色与权限的绑定关系，管理后台仍然保存的是权限 `id`
2. 后端根据权限 `id` 查询启用状态下的 `admin_permission.uuid`
3. `/api/auth/codes` 把这些 `uuid` 作为 `string[]` 返回给前端

### 代码位置

- 后端从权限表读取 `uuid`：
    - `internal/logic/admin_permission.go`
- 后端把 `uuid[]` 返回给前端：
    - `internal/logic/admin.go`
- 后端把 `roles`、`roleIds`、`isSuperAdmin` 返回给前端：
    - `internal/logic/admin.go`
- 前端请求权限码：
    - `apps/web-antd/src/api/core/auth.ts`
- 前端把权限码写入 `accessStore.accessCodes`：
    - `apps/web-antd/src/store/auth.ts`

## `uuid` 与 `module` 的职责

### `uuid`

- 面向前端权限显示与隐藏
- 必须和路由 `authority`、按钮 `v-access`、操作列 `auth` 保持一致
- 是 `/api/auth/codes` 的实际返回值

### `module`

- 面向后端接口权限模型
- 用于后端根据路由别名或接口模型控制是否允许访问
- 当前前端不直接用 `module` 做菜单、页面、按钮显隐判断

## 前端生效位置

### 路由权限

- 路由通过 `meta.authority` 配置页面权限码
- 路由守卫会把 `roles + accessCodes` 合并后交给 Vben 权限过滤
- 这里保留 `roles` 是为了兼容 Vben 的过滤入参，不表示页面只按角色做鉴权
- 页面是否显示、按钮是否可见，仍然优先看 `accessCodes`

示例：

```ts
meta: {
  authority: [SYSTEM_ROUTE_PERMISSION_CODES.ADMIN_LIST],
}
```

### 按钮权限

- 按钮使用 `v-access`

示例：

```vue
<VbenButton v-access="[SYSTEM_ACTION_PERMISSION_CODES.ADMIN_ADD]">
  新增用户
</VbenButton>
```

### 表格操作列权限

- 操作列统一通过 `auth` 字段控制

示例：

```ts
options: [
  { code: 'edit', text: '编辑', auth: SYSTEM_ACTION_PERMISSION_CODES.ADMIN_UPDATE },
]
```

## 当前前端推荐写法

### 全局权限中心

- 路径：`apps/web-antd/src/constants/permission-codes.ts`
- 说明：
    - 统一维护系统模块和 `cron` 模块的页面权限、按钮权限
    - 对齐老项目集中枚举的使用方式

### 已提供的导出

- `SYSTEM_ROUTE_PERMISSION_CODES`
- `SYSTEM_ACTION_PERMISSION_CODES`
- `CRON_ROUTE_PERMISSION_CODES`
- `CRON_ACTION_PERMISSION_CODES`
- `ALL_ROUTE_PERMISSION_CODES`
- `ALL_ACTION_PERMISSION_CODES`

## 新增权限时的推荐步骤

1. 后端新增或确认 `admin_permission.uuid`
2. 后端确认 `/api/auth/codes` 能返回该 `uuid`
3. 在前端全局权限中心补充对应常量
4. 页面路由使用 `meta.authority`
5. 按钮使用 `v-access`
6. 表格操作列使用 `auth`
7. 联调时先检查 `/api/auth/codes` 是否包含该权限码

## 与老项目 vue-vben-admin 的对应关系

### 老项目

- 集中权限枚举：`PermissionsEnum`
- 路由权限字段：`meta.uniqId`
- 按钮权限：`v-auth` / `hasPermission`

### 当前项目

- 集中权限常量：`permission-codes.ts`
- 路由权限字段：`meta.authority`
- 按钮权限：`v-access`

## 联调排查建议

- 页面看不到：
    - 先检查路由 `authority` 是否和后端 `uuid` 一致
    - 再检查 `/api/auth/codes` 是否包含该值
- 超级管理员页面看不到：
    - 先检查 `/api/login/after/info` 是否返回 `isSuperAdmin=true` 或 `roleIds` 包含 `1`
    - 再检查前端是否已将超级管理员兜底后的权限码写入 `accessStore.accessCodes`
- 按钮看不到：
    - 检查 `v-access` 或 `auth` 是否引用了错误常量
- 后端接口能调但前端不显示：
    - 大概率是前端没有使用正确的 `uuid`
- 前端权限管理页面录入时：
    - `uuid` 负责前端权限显示
    - `module` 负责后端接口权限模型
- 排查职责混用时：
    - `roles` 只看名称展示和路由兼容
    - `roleIds` 只看身份判定
    - `accessCodes` 才是最终权限显隐依据
