## 联调建议

- 登录后先检查 `/api/auth/codes` 是否返回权限码
- 页面或按钮不显示时，优先核对前端权限常量和后端 `admin_permission.uuid` 是否一致
- 列表页异常时，优先核对筛选参数和分页参数
- 提交类接口联调时，保留请求体和响应体快照便于回归
- 用户标签页面联调优先按“操作手册 -> 接口文档 -> 验证指南”的顺序执行

## 本轮联调发现与修复

- 任务系统管理接口已统一收敛到 `/api/tasks/*`，旧文档里残留的 `/api/task/*` 路径已清理，避免前端按旧地址请求出现 `404` 或菜单联调歧义。
- 任务控制台里的“调度器状态总览 / 参数快照 / 时间轴”不走新接口，而是直接复用 `GET /api/tasks/queues` 返回的 `data.scheduler`；联调时不要误以为还缺一个单独的 scheduler API。
- “配置热加载”已拆成独立菜单；页面除了基础的 `enabled / watching / lastStatus` 外，还依赖 `restartRequired`、
  `restartReason`、`lastTriggerSource`、`lastFailureCategory`、`suppressedFailureCount` 这些字段；前端提示“需重启”时属于后端预期行为，不是渲染错误。
- 后台日志页已按统一日志字典对齐 `route / action / describe / traceId / spanId / httpStatus / bizCode / latencyMs / success / errorMessage`，联调时不要再把页面文案写死在前端常量里。
- `GET /api/live`、`GET /api/ready`、`GET /api/health` 是公开探活接口，且访问日志默认被白名单屏蔽；探针访问后看不到 access log 属于预期，不应作为缺陷提单。
- 缓存管理页的 `GET /api/caches/keys` 现已在后端日志中输出 `request_source / search_path / provider_name / template_key / candidate_count / existing_count / latency_ms`；联调排查缓存搜索慢时，请优先按这些字段在 Kibana 检索，而不要只看接口 RT。
- 缓存搜索来源约定如下：页面手工检索=`manual_search`，业务页跳转自动定位=`route_jump`，模板键自动定位=`template_locate`，模板实例抽屉=`template_drawer`。

## 回归验证结果

- 调度器状态已由 `internal/taskqueue/manager_test.go` 中的 `TestListQueuesReturnsSchedulerStatus` 覆盖，确保 `GET /api/tasks/queues` 会返回 `data.scheduler`，并校验同步间隔、心跳间隔等关键字段。
- 热加载失败抑制与重启提示已由 `internal/bootstrap/app_test.go` 中的 `TestMarkHotReloadFailureSuppressesRepeatedLogs`、`TestDetectHotReloadRestartImpact` 覆盖，确保重复失败会累计到 `suppressedFailureCount`，基础设施配置变更会标记 `restartRequired=true`。
- 统一日志字典已由 `internal/infra/loggerx/logger_test.go` 中的 `TestFieldsFromMetaUsesUnifiedDictionary`、`TestTraceAttributesFromMetaUsesUnifiedChainFields` 覆盖，确保日志字段和 trace attributes 使用同一套语义。
- 探针访问日志屏蔽已由 `internal/middleware/access_log_test.go` 中的 `TestShouldIgnoreAccessLog` 覆盖，确认 `/api/live`、`/api/ready`、`/api/health`、`/api/metrics` 都会跳过 access log。
- 建议联调回归时按“任务总控台 -> 工作流状态 -> 配置热加载 -> 日志页 -> 健康检查”顺序做一轮页面核对，重点确认字段命名、状态提示和排障文案与接口返回一致。
- 排查缓存搜索慢的 Kibana 建议检索：
  - `message:"缓存搜索 完成" AND request_source:"manual_search"`
  - `message:"缓存搜索 完成" AND search_path:"template_candidates" AND template_key:"config_uuid:{uuid}"`
  - `message:"缓存搜索 失败" AND search_path:"unsupported_wildcard"`
  - `message:"缓存搜索 失败" AND route:"cache.search"`

### 骨架重算入口

- 接口：`POST /admin/user_tag/recalculate`
- 请求体示例：

```json
{
  "tag_types": [1, 2]
}
```

- 预期结果：
  - admin 立即返回“已启动 N 个标签重算任务”
  - admin 走内网调用，cron 收到 `/internal/user-tags/recalculations`
  - cron 触发 `user_tag.tag.recalculate` 骨架工作流

### targeted 骨架入口

- 接口：`POST /api/user-tags/workflows`
- 请求体示例：

```json
{
  "mode": "targeted",
  "uids": [10001, 10002],
  "tagTypes": [1, 2]
}
```

- 预期结果：
  - cron 返回 `taskId`、`workflowId`、`workflowName` 和 `queue`
  - 任务列表可看到 `prepare -> business_hook -> finalize -> sync_kafka`
  - `business_hook` 默认跳过属于预期

## 鉴权与网络约束

- `GET /api/live`、`GET /api/ready`、`GET /api/health` 为公开探活接口，不要求 JWT。
- 探针接口默认不打印 access log，联调时请以响应内容和状态码为准，不要用“日志是否出现”判断接口是否可用。
- `cron` 公网接口统一走 `/api/...`，要求 JWT。
- `admin -> cron` 服务间调用统一走 `/internal/...`，不带 JWT，但只允许内网或回环地址访问。
