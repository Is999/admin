# AI开发提示词

下面提示词可直接复制给 AI，用于功能开发、问题修复、重构和文档补全。

<button type="button" class="ai-prompt-copy-button" data-default-label="复制 AI 提示词">复制 AI 提示词</button>

```text
你现在参与 admin / admin-vue 项目的生产级开发与重构。请严格按以下规范执行，不能只给方案，除非我明确要求先不改代码。

一、工作方式
1. 全流程必须按标准开发执行流程推进：意图确认、源码证据、边界说明、实现、同步面检查、验证、交付复核。
2. 修改前必须先阅读现有目录结构、代码风格、命名习惯和调用链。
3. 需求或问题表述不充分、存在多种解释、目标仓库/入口不清，或涉及接口、数据、权限、安全、大表、缓存、迁移时，必须先结合代码和文档说明理解、证据、待确认点和执行边界；未经确认不得直接改代码。用户明确“先分析”“先不改代码”“只给方案”时只分析不落代码。
4. 优先保持现有分层和风格；遇到成熟通用能力、高风险基础设施或手写实现容易出错的场景，必须先检查现有实现和成熟依赖/框架。确需新增框架、目录、依赖或抽象时，先说明必要性、收益、替代方案、影响范围和验证方式，并让开发确认后再落地。
5. 改动必须小步、聚焦、可回滚，不顺手重构无关代码；代码优化、重构和性能复核使用 `$simple-code-guard`，先删无调用方、重复和纯转发代码，再做最小改动。
6. 重构必须保证功能不遗漏，并在交付说明中写清影响范围、迁移入口、验证方式，以及是否需要数据迁移、重刷或补偿任务。
7. 需求任务和拆解后的小任务必须按真实入口、调用链和交付面逐项闭环，禁止只搭骨架、只写样例、只实现基础能力、单一分支或 happy path 后宣称完成；确实无法完成时必须说明未完成项、阻塞原因、影响范围和下一步。
8. 注释、配置和文档必须简洁清晰，只说明用途、取值边界、默认行为或风险点，不写冗长背景。
9. 配置热加载必须区分运行参数和启动期注册开关；handler、workflow、插件启停等启动期能力不能在线补注册，需通过 restartRequired/restartReason 提示重启。

二、代码规则
1. 命名短而准确，贴合业务语境和当前职责；目录、变量、方法、常量、结构体、接口和字段禁止随意命名、泛名、错名、过度缩写、超长描述名、职责变化后沿用旧名和含糊的 `DefaultValue`。代码必须简洁、逻辑清晰、易读易维护；合理封装只能用于降低真实复杂度、复用稳定边界或隔离外部依赖，不能过度封装破坏清晰性。在业务正确和生产闭环前提下，简洁、逻辑清晰、易读、易维护和高性能是最高优先级；每个 helper、interface、wrapper、状态和分支必须解决当前已证明的问题。已对应真实故障窗口的租约、幂等和原子状态迁移不能当作理论防御删除。
2. 错误必须使用 `github.com/Is999/go-utils/errors` 包装。
3. 同一个函数里，同一个 `error` 值只包装一次；跨函数边界可以继续包装，因为那是新的调用语义和上下文，避免裸错误漏出去。
4. 业务中间层只返回错误，不打印 error 级日志；顶层 handler、worker、scheduler 或不可返回错误的兜底协程统一打印 error，同一错误链路只允许最外层打印一次。
5. 错误上下文和任务日志必须带可追踪字段，例如 `workflow_id`、`mode`、`node`、`shard`、`uid`、`task_id`。
6. 常量按领域前缀集中定义，表达稳定业务含义或枚举语义；禁止魔法字符串。
7. 结构体使用名词或领域对象命名；请求以 `Req` 结尾，响应以 `Resp` 结尾，列表项以 `Item` 结尾。
8. 接口按能力命名，优先使用 `Reader`、`Writer`、`Repository`、`Provider` 等自然后缀；只有一个实现且无替换点时不要强行抽接口。新增抽象前必须确认真实替换点、稳定复用边界和维护成本。
9. Redis Key 必须集中定义，禁止在业务代码临时拼接高基数通配 Key。
10. 目录布局必须符合当前项目分层，保持 `handler` / `logic` / `model` / `svc` / `types` 等边界清晰。

三、中文注释
1. 新增或重构的变量、常量、方法、结构体、字段、map 字段、路由、GORM Model、接口和关键逻辑必须补中文注释。
2. 代码注释、配置注释和文档说明必须简洁清晰，不能啰嗦，也不能含糊。一句话能说明白的不要写成长段，注释只说明字段用途、取值边界、默认行为或风险点关键信息，不重复字段名，不写背景故事。
3. 顶层稳定契约中的 map、数组、切片等字段必须说明元素 key/value 或元素对象含义、来源、边界条件、排序/去重和空值语义。

四、数据库与缓存
1. MySQL/SQL 编写必须优先使用 GORM 链式调用；事务内必须沿用同一个 `tx`，禁止参考遗留风格随意手写原生 SQL。
2. 所有数据库表必须有对应 Model；新增表、改表、迁移 DDL 或修复 DDL 时，必须同步创建或更新 Model，并校对表名、字段、类型、默认值、NULL、主键、索引和时间/软删除语义。
3. 只有 GORM 无法安全表达或会严重影响性能时才允许原生 SQL；使用前必须说明原因、索引依据、预估扫描范围、分页/批量上限、超时/并发控制和 DB 压力。
4. 原生 SQL / Lua 必须作为 `*.sql.tmpl` 或 `.lua` 代码资产，通过 `go:embed` 加载。
5. SQL / Lua 调用前必须通过 `embedasset.StripLeadingLineComments` 或当前包 helper 剥离文件头注释。
6. 禁止在 Go 文件中内联多行 SQL / Lua。
7. 文档权限只维护 `internal/database/assets/admin_permission.sql` 和 `internal/routealias/docs.go`；未上线阶段直接同步权限表基线，禁止为单篇文档新增独立 SQL 或迁移。
8. SQL seed 显式保留自增主键 `id` 时，新增行必须按 `id` 递增放在对应位置，禁止按业务相似性插入到中间。
9. 禁止在线接口无界扫描高基数 Redis Key；必须使用精确 Key、索引集合、白名单模板或异步任务。

五、错误处理
1. 错误必须使用 github.com/Is999/go-utils/errors 包装。
2. 同一个函数里，同一个 error 值只包装一次；跨函数边界可以继续包装，因为那是新的调用语义和上下文，避免裸错误漏出去。
3. 业务中间层只返回错误，不打印 error 级日志。
4. 顶层 handler、worker、scheduler 或不可返回错误的兜底协程统一打印 error 级日志，同一错误链路只允许最外层打印一次。
5. 错误上下文必须带 workflow_id、mode、node、shard、uid、task_id 等可追踪字段。

六、日志规范
1. 日志统一使用项目现有 loggerx/logx 风格。
2. 正常流程只打印关键节点，不刷屏。
3. 任务系统、收集器和批处理日志必须带 workflow_id、mode、node、shard 等链路字段。
4. 日志字段类型必须稳定；时间窗口和间隔字段要带单位后缀，例如 window_start_unix、interval_seconds。

七、接口与业务码
1. API 响应必须统一为 status/code/message/data。
2. 不同业务错误必须使用不同业务码；message 不能暴露 SQL、密钥、内部表名或真实下游错误。
3. 新增业务码必须补中文和英文文案。
4. 新增接口必须补接口文档，格式遵循 docs/site/接口文档/接口文档统一规范.md。
5. 新增路由必须补 RouteMeta、审计动作、中文说明和权限码；白名单路由必须写清白名单原因和 token 校验边界。

八、大数据查询与批处理
1. 涉及大表、大结果集或高频查询时，必须先评估数据量、索引、执行计划、并发量和目标库压力。
2. 禁止在线接口做无上限全表查询、深分页 OFFSET、SELECT *、无索引排序/聚合/JOIN、一次性加载超大 ID 集合。
3. 查询必须按时间窗口、主键游标、分片或状态分页推进，并控制 batch_size、最大批次数、超时、并发和 sleep/backoff。
4. 大聚合、历史回补、跨库对账、批量修复和导出应走后台任务、维护队列、汇总表、ClickHouse 或数仓，不放在 HTTP 请求内长时间执行。
5. 批量写入和删除必须小批次按主键或索引推进，避免长事务、大锁、超大 IN/NOT IN；任务要支持幂等、断点续跑、进度水位和可中止。
6. 新增大数据查询必须补索引依据、保护阈值、关键日志/指标和最小测试；无法验证执行计划时，要说明风险和上线观察方式。

九、接口与前端
1. admin-vue 延续 vbenjs/vue-vben-admin 推荐布局和现有项目风格。
2. 前端文案必须支持中文和英文国际化，不能硬编码业务文案。
3. 敏感操作必须接入 MFA、签名、加密、秘钥版本等既有安全机制。
4. 新增业务码必须补中文和英文文案；新增接口必须补接口文档。

十、验证与交付
1. 后端改动必须执行 gofmt；大范围改动执行 go test ./...，小范围至少跑相关包测试。
2. 前端改动至少执行 pnpm -F @vben/web-antd run typecheck；大范围改动优先执行 pnpm run check。
3. 存在 Makefile、CI、lefthook 或 pre-commit 时，优先执行项目已有检查入口。
4. 检查失败必须继续修复，不能绕过或只解释。
5. 交付说明必须逐项说明任务完成情况；存在未完成、降级、跳过验证或只完成部分能力时必须明确标出，不能用笼统“已完成”掩盖半成品。
6. 最终说明必须包含：修改文件、改动原因、执行检查、未通过项、是否需要数据迁移或补偿任务。
```
