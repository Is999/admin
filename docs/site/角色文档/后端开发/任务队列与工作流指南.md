# 任务队列与工作流指南

> **AI 开发必读**：AI 新增或重构任务、工作流、周期任务前，必须先阅读 [AI开发规范](角色文档/后端开发/AI开发规范.md)。任务内层只返回包装错误，`error` 日志由 Worker 或 Scheduler 顶层统一打印。

## 目标

当前项目已接入基于 Redis 的异步任务系统，支持：

- 至少执行一次（At-least-once）
- 调度与延迟执行（Scheduling）
- 自动重试（Retries）
- Worker 崩溃恢复（Crash Recovery）
- 加权优先级队列（Weighted Priority Queues）
- 严格优先级队列（Strict Priority Queues）
- 去重（Unique）
- 任务级超时/截止时间（Timeout / Deadline）
- 聚合批处理（Grouping / Aggregation）
- 周期性任务（Periodic Tasks）
- 手动触发工作流
- DAG / 分片 / 灰度
- 队列暂停 / 恢复
- CLI 管理

## 内置工作流

### `cache.refresh`

用于刷新公共下拉缓存，支持：

- `targets`：指定要刷新的缓存键列表
- `shardTotal`：把批量刷新拆成多个分片并行执行
- `grayPercent`：只执行部分分片，实现灰度发布/灰度校验

当前项目已移除示例缓存刷新目标；后续如需新增业务缓存刷新能力，请通过业务插件显式注册。

### 插件化接入

当前任务系统已支持插件化扩展，推荐新增模块按下面方式接入：

- `internal/bootstrap/components/builtin/task.go`：默认任务插件装配入口
- `internal/jobs/plugins.go`：业务和维护任务插件清单
- `internal/task/runtime/runtime.go`：插件注册中心与通用构造器
- `internal/task/runtime/core_plugin.go`：核心 handler / workflow
- `internal/jobs/<业务>/task`：业务模块自己的 handler、workflow 和周期任务定义

常用注册方式：

- `NewPluginFunc(...)`：快速声明一个通用插件
- `NewCacheRefreshPlugin(...)`：注册单个缓存刷新目标
- `NewPeriodicWorkflowPlugin(...)`：注册单个周期工作流

接入原则：

- 普通任务 handler：放在核心插件或业务插件里，通过 `Runtime.RegisterHandler(...)` 注册
- 工作流定义：通过 `Runtime.RegisterWorkflow(...)` 注册
- 周期任务：通过 `Runtime.RegisterPeriodicTask(...)` 注册
- 缓存刷新目标：通过 `Runtime.RegisterCacheRefreshTarget(...)` 注册

## 管理接口

常用入口如下，字段详情请直接查看接口文档：

- `POST /api/tasks/workflows`
- `GET /api/tasks/workflows/:workflowId`
- `GET /api/tasks/queues`
- `POST /api/tasks/queues/pause/:queue`
- `POST /api/tasks/queues/resume/:queue`

补充说明：

- `GET /api/tasks/queues` 除了返回队列和在线 Worker，还会额外返回 `data.scheduler`，供任务控制台直接展示调度器状态。
- `GET /api/tasks/config-reload` 和 `POST /api/tasks/config-reload` 返回同一份热加载状态快照，前端依赖其中的 `restartRequired`、`restartReason`、`lastTriggerSource`、`lastFailureCategory` 等字段做提示。
- 任务插件、handler、workflow 定义是否注册由启动期配置决定；例如 `workflows.user_tag.enabled` 从 `false` 热加载为 `true` 时不会在线补注册，只会通过 `restartRequired/restartReason` 提示重启。
- `GET /api/tasks/config-reload/items` 只读取运行态配置快照并返回已脱敏配置项；新增配置字段时要同步检查脱敏关键字，避免地址、DSN、秘钥或 Token 原文出现在响应里。

## CLI

任务系统底层使用 [Asynq CLI](https://github.com/hibiken/asynq/tree/master/tools/asynq)。

安装：

```bash
go install github.com/hibiken/asynq/tools/asynq@latest
```

常用命令：

```bash
asynq dash
asynq stats
asynq queues pause maintenance
asynq queues unpause maintenance
asynq tasks list maintenance scheduled
```

## Redis 说明

当前任务系统支持两种 Redis 接入方式：

- 默认复用项目全局 `redis` 配置
- 如配置了 `task.redis`，任务系统将改用独立 Redis

开发时建议优先记住两点：

- 业务缓存可以继续使用现有 Redis 或 Redis Cluster
- 任务系统优先使用单机 Redis 或哨兵主从 Redis

原因是 Asynq 依赖 Lua 脚本和多类 Redis 原子能力，某些 Redis Cluster、代理层或托管服务会限制这些能力。

示例：

```yaml
task:
  enabled: true
  redis:
    addrs:
      - "127.0.0.1:6380"
    password: ""
    db: 0
    pool_size: 20
```

说明：

- `task.redis` 未配置时，任务系统继续复用全局 `redis`
- `task.redis` 配置后，仅 Worker / Scheduler / Asynq Inspector 使用这套 Redis
- 业务逻辑中的缓存读写仍然使用全局 `redis`

如果线上是多实例部署，务必确保同一站点实例共享同一套 `task.redis`，并且使用相同的 `app_id`。详细规则见 [任务系统使用手册](功能模块/任务系统/任务系统使用手册.md)。

## Asynq 定时任务持久化说明

当前依赖版本为 `github.com/hibiken/asynq v0.26.0`。

- Asynq 的任务、延迟任务、重试任务和完成保留记录存储在 Redis。
- Asynq 自带 `Scheduler` 会把已注册的周期 entry 和入队历史写入 Redis，并可通过 `Inspector.SchedulerEntries`、`Inspector.ListSchedulerEnqueueEvents` 查询。
- Asynq 不提供关系型数据库落库能力；如果需要 SQL 审计表，需要项目侧在投递或调度钩子中自行写入。
- 当前项目没有直接使用 Asynq `Scheduler` / `PeriodicTaskManager`，而是使用项目内 `periodicTaskScheduler` 读取 `task_periodic` 配置，通过 Redis leader 锁保证多实例只由一个 Scheduler 投递。
- 因此线上周期配置的权威来源取决于 `runtime_config.source`：数据库模式看运行配置 active release，文件模式才看 `etc/config.d/runtime.yaml` 等运行期配置；实际投递记录和任务状态仍以 task Redis / 任务列表为准。
