# 库表路由规范

当前项目默认只使用一个 MySQL 主库，统一通过 `svc.DatabaseMain` 访问。`site_mysql` 仅作为后续拆库扩展入口，默认不配置。

## 默认模型

- `mysql`：必填主库配置。
- `site_mysql`：可选命名扩展库配置，按 `site_mysql.<name>` 注册。
- `ReadDB(database)`：返回只读连接，空值或 `main` 指向主库。
- `WriteDB(database)`：返回写连接，空值或 `main` 指向主库。

## 扩展示例

确需拆库时，新增一个有明确用途的命名库：

```yaml
site_mysql:
  archive:
    write_data_source: "user:password@tcp(mysql:3306)/admin_archive?charset=utf8mb4&parseTime=true&loc=Local"
    read_data_sources: []
    max_open_conns: 20
    max_idle_conns: 10
    conn_max_lifetime: 300
    debug: false
```

业务配置需要使用扩展库时，显式填写对应名称，例如 `archive_jobs[].database: "archive"`。未配置的名称不会自动回退，启动或执行阶段应尽早暴露错误。

## 调用规则

- 默认使用 `svc.DatabaseMain`。
- 不在业务逻辑中手写 DSN 或临时打开 MySQL 连接。
- 新增扩展库前先确认数据边界、连接池规模、读写分离和健康检查诉求。
- 扩展库名称只做前后空白整理；命名应短而明确，便于运维识别。

## 分片字段规则

业务 UID 表必须把逻辑分片字段前置设计好，避免数据量上来后只能靠全表扫描补救。

### 业务用户表

- `user.id` 是雪花 ID，`user.shard_no = idgen.ShardNo(user.id)`。
- `idgen.ShardNo` 固定为 `CRC32(id字符串) % 1024`，范围 `0-1023`。
- 不能使用裸 `id % 1024`，雪花 ID 低位是同毫秒序列，低并发时会严重偏斜。
- `user_account` 是全局账号索引，保存 `username -> user_id/shard_no/route_shard_count`。
- `route_shard_count` 只允许 `1/2/4/8/16/32/64/128/256/512/1024`。
- `route_shard_count=1` 时读写单表 `user`。
- `route_shard_count>1` 时使用 `range_size = 1024 / route_shard_count`，`range_start = shard_no / range_size * range_size`，物理表名为 `user_%04d`。

示例：

| 物理表数量 | 表名 |
| --- | --- |
| `1` | `user` |
| `2` | `user_0000`, `user_0512` |
| `4` | `user_0000`, `user_0256`, `user_0512`, `user_0768` |
| `8` | `user_0000`, `user_0128`, `user_0256`, `user_0384`, `user_0512`, `user_0640`, `user_0768`, `user_0896` |
| `1024` | `user_0000` 到 `user_1023` |

### 用户标签表

- 用户标签使用普通业务 UID，`shard_no = uid % 1024`。
- `user_tag_0` 是结果表基表，默认只创建单张物理表。
- `result_shard_total` 控制结果表物理数量，表名是连续编号 `user_tag_0`、`user_tag_1`、`user_tag_2`。
- `runtime_shard_total` 控制运行期 UID 索引逻辑分片，默认 `1024`，不等同于结果表物理数量。
- `user_tag_runtime_uid` 和 `user_tag_event_outbox` 是单表索引型运行期表，必须保留 `shard_no` 与对应复合索引。

### 新增 UID 表硬规则

凡 DDL 中有精确业务列 ``uid`` 的表，必须同时满足：

1. 增加 ``shard_no`` 字段：`int NOT NULL DEFAULT 0 COMMENT 'uid取模1024分片'`。
2. 写入时同步计算：`shard_no = uid % 1024`。
3. 至少提供一个以 `shard_no` 靠前的查询索引，例如 `KEY idx_shard_uid (shard_no, uid)`。
4. 如果表还按 `workflow_id/state/tag_type` 等业务条件过滤，索引顺序要贴近真实查询，例如 `(workflow_id, state, shard_no, id)` 或 `(tag_type, shard_no, uid)`。
5. GORM Model 必须补 `ShardNo` 字段和中文注释，DDL、Model、测试和文档保持同一口径。

例外：

- ``user.id`` 使用 `CRC32(id字符串)%1024`，不是 `uid % 1024`。
- ``user_account.user_id`` 代表业务用户雪花 ID，必须带 `shard_no`，但值来源仍是 `idgen.ShardNo(user_id)`。
- `admin_log.user_id`、`admin_role_rel.user_id` 等后台管理员 ID 不属于前台业务 UID，不按本规则追加业务分片字段。
- `cursor_uid` 只是游标水位；如果表已经按节点分片保存 `shard_no`，不需要再新增第二个 UID 分片字段。

## 用户表拆分操作指南

以下步骤适用于 `user` 从 `1 -> 2 -> 4 -> ... -> 1024` 逐级拆分。每次只提升到下一个 2 的幂，不跨档跳跃。

### 拆分公式

给定目标物理表数量 `target_count`：

- `range_size = 1024 / target_count`
- 第 `i` 个目标范围起点：`range_start = i * range_size`
- 第 `i` 个目标范围终点：`range_end = range_start + range_size - 1`
- `target_count=1` 时物理表是 `user`
- `target_count>1` 时物理表是 `user_%04d`，参数为 `range_start`

示例：

| 目标表 | shard_no 范围 | 说明 |
| --- | --- | --- |
| `user_0000` | `0-511` | 2 表阶段左半区 |
| `user_0512` | `512-1023` | 2 表阶段右半区 |
| `user_0000` | `0-255` | 4 表阶段第 1 段，复用 2 表阶段左半区表 |
| `user_0256` | `256-511` | 4 表阶段第 2 段，新建表 |
| `user_0512` | `512-767` | 4 表阶段第 3 段，复用 2 表阶段右半区表 |
| `user_0768` | `768-1023` | 4 表阶段第 4 段，新建表 |

### 拆分总流程

除 `1 -> 2` 初次从 `user` 单表切到带后缀物理表外，后续扩容只做“右半区搬迁”，不要把所有数据重新打散。

1. 确认旧档位 `old_count` 和目标档位 `target_count`，且 `target_count = old_count * 2`。
2. 计算 `old_range_size = 1024 / old_count`、`new_range_size = 1024 / target_count`。
3. `1 -> 2` 时新建 `user_0000`、`user_0512` 并从 `user` 分别复制左右两段。
4. `2 -> 4` 及以后按旧范围逐段生成计划：旧范围左半区继续留在原表，右半区复制到新表。
5. `2 -> 4` 及以后只创建右半区的新表；左半区原表不重建、不搬迁。
6. 对每个待搬迁范围按主键小批复制，复制过程记录批次、最后主键和异常。
7. 每个目标范围独立校验行数、缺失样本和账号索引定位。
8. 最终切换前选择维护窗口或在线 delta 补偿，不能在无写入保护时直接切路由。
9. 分批更新 `user_account.route_shard_count` 到目标档位。
10. 修改 `user.route_shard_count` 配置并统一重启所有读写实例。
11. 冒烟和观察通过后，再按小批删除旧表里的已搬迁副本。

### `1 -> 2` 示例

当前只有单表 `user`，目标是 `user_0000` 和 `user_0512`。

| 源表 | 目标表 | shard_no 范围 | 动作 |
| --- | --- | --- | --- |
| `user` | `user_0000` | `0-511` | 新建目标表并复制 |
| `user` | `user_0512` | `512-1023` | 新建目标表并复制 |

执行顺序：

1. `CREATE TABLE user_0000 LIKE user`。
2. `CREATE TABLE user_0512 LIKE user`。
3. 从 `user` 按 `shard_no BETWEEN 0 AND 511`、`id` 游标小批复制到 `user_0000`。
4. 从 `user` 按 `shard_no BETWEEN 512 AND 1023`、`id` 游标小批复制到 `user_0512`。
5. 校验两个目标表行数和抽样数据。
6. 在最终切换窗口内暂停所有会写 `user` 的入口，或执行已实现的 delta 补偿任务。
7. 再跑一轮增量同步，确认源表在切换窗口内无遗漏更新。
8. 分批更新 `user_account.route_shard_count = 2`。
9. 修改 `user.route_shard_count=2` 并统一重启所有 `api/admin` 写入实例。
10. 冒烟通过后，保留旧 `user` 作为回滚表；观察窗口结束后再归档或清理。

### `2 -> 4` 示例

当前 `route_shard_count=2`，已有 `user_0000` 和 `user_0512`。目标是四张表。

| 源表 | 目标表 | shard_no 范围 | 动作 |
| --- | --- | --- | --- |
| `user_0000` | `user_0000` | `0-255` | 复用原表，不复制 |
| `user_0000` | `user_0256` | `256-511` | 新建表并复制右半区 |
| `user_0512` | `user_0512` | `512-767` | 复用原表，不复制 |
| `user_0512` | `user_0768` | `768-1023` | 新建表并复制右半区 |

执行顺序：

1. `CREATE TABLE user_0256 LIKE user_0000`。
2. `CREATE TABLE user_0768 LIKE user_0512`。
3. 从 `user_0000` 按 `shard_no BETWEEN 256 AND 511`、`id` 游标复制到 `user_0256`。
4. 从 `user_0512` 按 `shard_no BETWEEN 768 AND 1023`、`id` 游标复制到 `user_0768`。
5. 校验 `user_0256`、`user_0768` 完整性；`0-255` 和 `512-767` 继续留在原表。
6. 在最终切换窗口内暂停写入或执行 delta 补偿。
7. 分批更新 `user_account.route_shard_count = 4`。
8. 修改 `user.route_shard_count=4` 并统一重启所有写入实例。
9. 观察通过后，小批清理 `user_0000` 中 `256-511` 的旧副本，以及 `user_0512` 中 `768-1023` 的旧副本。

后续 `4 -> 8`、`8 -> 16` 依此类推：每个旧范围拆成左右两半，左半继续复用旧表，右半新建一个以新范围起点命名的表。

### 单个范围的执行模板

以从 `user_0000` 拆出 `user_0256` 为例。

1. 确认源范围和目标范围：

   ```text
   source_table = user_0000
   target_table = user_0256
   source_old_range = 0-511
   target_new_range = 256-511
   ```

2. 创建目标表：

   ```sql
   CREATE TABLE IF NOT EXISTS user_0256 LIKE user_0000;
   ```

3. 先取本批要搬迁的主键，记录本批最大 `id`：

   ```sql
   SELECT id
   FROM user_0000
   WHERE shard_no BETWEEN 256 AND 511
     AND id > ?
   ORDER BY id ASC
   LIMIT ?;
   ```

4. 按本批主键复制数据，字段必须显式列出，禁止 `SELECT *`：

   ```sql
   INSERT INTO user_0256 (
     id, shard_no, username, nickname, password_hash, email, phone, avatar,
     status, last_login_at, last_login_ip, created_at, updated_at
   )
   SELECT
     id, shard_no, username, nickname, password_hash, email, phone, avatar,
     status, last_login_at, last_login_ip, created_at, updated_at
   FROM user_0000
   WHERE shard_no BETWEEN 256 AND 511
     AND id IN (...)
   ORDER BY id ASC;
   ```

5. 用第 3 步记录的最大 `id` 作为下一批游标；批次建议从 `500-5000` 起步，按主库压力调小或调大。
6. 复制任务必须可重复执行。若使用 `ON DUPLICATE KEY UPDATE` 做补偿，必须确保只用较新的 `updated_at` 覆盖目标表，避免旧源表覆盖切换后的新写入。
7. 校验行数：

   ```sql
   SELECT COUNT(*) FROM user_0000 WHERE shard_no BETWEEN 256 AND 511;
   SELECT COUNT(*) FROM user_0256 WHERE shard_no BETWEEN 256 AND 511;
   ```

8. 校验缺失样本：

   ```sql
   SELECT s.id
   FROM user_0000 AS s
   LEFT JOIN user_0256 AS t ON t.id = s.id
   WHERE s.shard_no BETWEEN 256 AND 511
     AND t.id IS NULL
   ORDER BY s.id ASC
   LIMIT 10;
   ```

9. 校验账号索引可定位：

   ```sql
   SELECT ua.user_id
   FROM user_account AS ua
   LEFT JOIN user_0256 AS u ON u.id = ua.user_id
   WHERE ua.shard_no BETWEEN 256 AND 511
     AND u.id IS NULL
   ORDER BY ua.user_id ASC
   LIMIT 10;
   ```

10. 最终切换前必须选择一种写入保护：
   - 维护窗口：暂停注册、后台新增用户、修改用户状态、改密等写入入口。
   - 在线补偿：使用独立迁移任务持续按 `updated_at/id` 追平，并带新旧行冲突保护。

11. 切路由先取本批账号，再按本批账号更新，避免大事务和不可恢复游标：

    ```sql
    SELECT user_id
    FROM user_account
    WHERE shard_no BETWEEN 256 AND 511
      AND route_shard_count = 2
      AND user_id > ?
    ORDER BY user_id ASC
    LIMIT ?;
    ```

    ```sql
    UPDATE user_account
    SET route_shard_count = 4
    WHERE shard_no BETWEEN 256 AND 511
      AND route_shard_count = 2
      AND user_id IN (...);
    ```

12. 确认该范围已无旧路由账号：

    ```sql
    SELECT COUNT(*)
    FROM user_account
    WHERE shard_no BETWEEN 256 AND 511
      AND route_shard_count <> 4;
    ```

13. 切换后按账号索引抽样读取目标表，覆盖登录、按 ID 查询和后台用户详情。
14. 观察窗口结束后再清理旧副本。清理也要先取主键再按主键删除，避免单条 SQL 扫描和长事务：

    ```sql
    SELECT id
    FROM user_0000
    WHERE shard_no BETWEEN 256 AND 511
    ORDER BY id ASC
    LIMIT ?;
    ```

    ```sql
    DELETE FROM user_0000
    WHERE shard_no BETWEEN 256 AND 511
      AND id IN (...);
    ```

15. 清理后再次确认该源表只保留新范围内的数据。

### 切换前检查清单

- `user_account` 无缺失账号索引。
- 所有目标表已创建，结构、索引、字符集与源表一致。
- 每个目标范围行数校验一致，缺失样本为空。
- 最终切换策略已确认：维护窗口或在线 delta 补偿。
- `api/admin` 的 `user.route_shard_count` 配置已准备，但未提前只重启部分实例。
- 回滚方案已准备：保留旧表数据、旧配置、旧二进制和可逆的路由更新批次记录。

### 回滚规则

- 只创建目标表、未更新 `user_account.route_shard_count`：直接停止迁移并保留目标表待复查。
- 已更新部分账号索引：按批把对应 `route_shard_count` 改回旧档位，实例配置也回到旧档位并统一重启。
- 已清理旧副本：只能从备份或归档表恢复，禁止无备份清理旧数据。
- 回滚后必须重新验证登录、按 ID 查询、按用户名查询和后台用户详情。

## 用户标签拆分操作指南

用户标签结果表和用户主表不是同一套路由。

1. 保持 `runtime_shard_total=1024`，运行期 UID 表继续用 `uid % 1024` 作为 `shard_no`。
2. 需要扩结果表时，提升 `result_shard_total` 到 `2/4/8/.../1024`。
3. 新结果表使用 `CREATE TABLE LIKE user_tag_0` 创建，例如 `user_tag_1`、`user_tag_2`，临时表对应 `user_tag_1_tmp`。
4. 结果表物理编号是连续编号，不使用 `user_0512` 这类范围起点后缀。
5. full 重建时先写临时表，再通过 rename 原子切换；不要直接覆盖线上结果表。
6. `user_tag_runtime_uid`、`user_tag_runtime_checkpoint`、`user_tag_event_outbox` 默认不按物理表拆分；它们依赖 `shard_no` 索引做分片扫描。
7. 扩容后跑一次 full 工作流，确认各分片行数、事件 outbox、hook 派发和运行期清理任务正常。

## 禁止项

- 禁止新增业务 UID 表但不带 `shard_no`。
- 禁止把后台管理员 ID 当作前台业务 UID 分片。
- 禁止在在线接口里扫描所有用户物理表。
- 禁止通过表名后缀猜当前用户位置，用户表只能信 `user_account.route_shard_count`。
- 禁止在迁移中一次性无界 `INSERT ... SELECT`、无游标全表更新或长事务搬迁。
- 禁止对雪花 ID 使用裸 `id % 1024`。
