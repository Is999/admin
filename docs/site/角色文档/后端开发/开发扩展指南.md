# 开发扩展指南

> **AI 开发必读**：AI 辅助开发、重构或补文档前，必须先阅读 [AI开发规范](角色文档/后端开发/AI开发规范.md)。不要在业务内层随意打印 `error` 日志；新增 `*.sql.tmpl` / `.lua` 必须按格式写文件头，并在调用处剥离注释后执行。

本文面向接手 `admin` 的后端开发同学，重点回答 4 个问题：

1. 服务如何部署和发布
2. 新增一个业务功能时要改哪些文件
3. 新增一个后台任务 / 工作流时要改哪些文件
4. 新增一个定时任务时要改哪些文件

AI 辅助开发时，必须先阅读 [AI开发规范](角色文档/后端开发/AI开发规范.md)，再进行代码生成、重构或文档补充。

## 1. 部署与发布

### 1.1 运行形态

服务由 3 个运行单元组成，统一通过 `-mode` 控制：

- `1`：只启动 API
- `2`：只启动 Worker
- `4`：只启动 Scheduler
- `7`：同时启动 API、Worker、Scheduler

对应入口代码：

- `cmd/admin/main.go`：解析 `-f` 和 `-mode`
- `internal/bootstrap/wiring.go`：统一读取配置并装配 `App`
- `internal/bootstrap/runmode`：解析 `run_mode` 位掩码和运行单元组合
- `internal/bootstrap/configload`：读取、合并、校验配置，生成配置包指纹和热加载生效策略
- `internal/bootstrap/configload/runtimefile`：合并外置运行期配置文件
- `internal/bootstrap/hotreload`：管理 config.yaml 热加载运行态、状态记录和失败限频
- `internal/bootstrap/runtimewatch`：管理 DB 运行配置 active release 启动期应用、轮询和重载水位
- `internal/bootstrap/appalert`：组装 App 启动、配置热加载和 DB 运行配置热加载告警
- `internal/bootstrap/resources`：初始化数据库、Redis、Kafka、tracing 等基础设施资源
- `internal/bootstrap/app.go`：根据 `mode` 启动 API、Worker、Scheduler

推荐部署方式：

- 开发 / 测试环境：直接使用 `-mode 7`
- 生产常规环境：使用 `cron-control(-mode 5) + cron-worker(-mode 2)`
- 大规模生产环境：拆为 `cron-api(-mode 1) + cron-worker(-mode 2) + cron-scheduler(-mode 4)`

说明：

- Worker 可以天然横向扩容
- Scheduler 通过 Redis leader 锁抢占，只会有 1 个实例生效
- API、Worker、Scheduler 共享一套 `ServiceContext`

### 1.2 发布前准备

发布前至少确认以下资源：

- MySQL 已创建业务库并导入初始化 SQL
- Redis 已可访问，且所有实例共享同一套 Redis；Collector 使用 Redis 保存 EventID 幂等状态
- 如业务 Redis 为 Cluster，已评估任务系统是否需要改用 `task.redis`
- 配置文件已准备完成
- `jwt_secret` 已替换为正式环境密钥
- 如启用 trace，`observability.otlp_endpoint` 已正确配置

初始化 SQL：

- `internal/database/assets/admin.sql`
- `internal/database/assets/admin_message.sql`
- `internal/database/assets/collector_failed_event.sql`
- `internal/database/assets/20260325_admin_log_observability.sql`

### 1.3 构建命令

本地或 CI 可直接构建：

```bash
go mod tidy
go test ./...
go build -o bin/admin ./cmd/admin
```

如果只做基础校验，建议至少执行：

```bash
go test ./...
go vet ./...
```

### 1.4 启动命令

开发环境：

```bash
go run ./cmd/admin -f ./etc/config.yaml -mode 7
```

生产环境单实例：

```bash
./bin/admin -f ./etc/config.yaml -mode 7
```

如果按推荐方式拆分部署：

```bash
# control 节点：API + Scheduler
./bin/admin -f ./etc/config.yaml -mode 5

# Worker 节点
./bin/admin -f ./etc/config.yaml -mode 2
```

### 1.5 发布检查清单

建议每次发布按以下顺序执行：

1. 拉取最新代码并执行 `go test ./...`
2. 检查配置文件是否与当前代码结构一致
3. 检查 SQL 变更是否已执行
4. 部署二进制或镜像
5. 使用目标模式启动，常规生产优先 `-mode 5 + -mode 2`
6. 验证 `/api/docs`、登录接口、核心查询接口是否正常
7. 验证 Redis / Worker / Scheduler 日志是否正常
8. 如开启周期任务，确认运行配置 active release 中的 `task_periodic` 已生效

### 1.6 发布时常改文件

- `etc/config.sample.yaml`：示例配置
- `etc/config.dnmp.sample.yaml`：DNMP 示例配置
- `internal/config/config.go`：配置结构体定义
- `internal/bootstrap/wiring.go`：应用装配入口
- `internal/bootstrap/runmode`：运行模式位掩码规则
- `internal/bootstrap/configload/runtimefile`：外置运行期配置合并规则
- `internal/bootstrap/hotreload`：config.yaml 热加载状态记录与限频
- `internal/bootstrap/runtimewatch`：DB 运行配置 active release watcher
- `internal/bootstrap/appalert`：App 层运行异常告警模板
- `internal/bootstrap/resources`：基础设施资源初始化与释放
- `internal/bootstrap/app.go`：启动编排
- `cmd/admin/main.go`：启动参数与优雅退出

补充说明：

- `redis`：业务缓存与通用 Redis 能力
- `task.redis`：任务系统专用 Redis；未配置时回退到全局 `redis`
- 如果线上 Redis Cluster 对 Lua 脚本或多 key 操作有限制，建议把 `task.redis` 指向单机 Redis 或哨兵主从 Redis

### 1.7 注册清单总览

当前项目已经把“启动组件 / 路由模块 / 任务插件 / 通用收集器 bizType”四类注册点收口到固定入口。排查启动问题或新增业务能力时，优先从这些文件看：

| 注册类型 | 统一清单文件 | 实际注册方法 | 说明 |
| --- | --- | --- | --- |
| 启动组件 | `internal/bootstrap/components/builtin/specs.go` | `builtin.DefaultSpecs()` / `builtin.DefaultComponents()` / `manifest.Default()` | 统一列出应用启动时要装配的内置组件，并从规格派生真实组件和清单 |
| 启动组件执行器 | `internal/bootstrap/components` | `components.NewRegistry(...).Register(...)` | 通用执行器校验组件名唯一，启动组件类型和注册中心归属到 `components` |
| 启动组件实现 | `internal/bootstrap/components/builtin` | `newCollector()` / `newCDCConsumer()` / `newTaskRuntime()` / `newHTTPServer()` | 具体内置组件下沉到 `builtin`，每个组件只负责装配依赖，真正启动由 `App.Start()` 统一编排 |
| 路由模块 | `internal/handler/routes.go` | `builtinRouteModuleSpecs` / `DefaultRouteSpecs()` / `DefaultRouteContracts()` / `RegisterHandlers(...)` | 内置路由模块集中声明顺序和路由规格来源，真实注册、路由契约和注册清单从规格派生 |
| 任务插件 | `internal/bootstrap/components/builtin/task.go` + `internal/jobs/plugins.go` | `builtin.DefaultTaskPluginSpecs()` / `jobs.PluginSpecs()` | 核心、业务和维护任务插件集中在规格中声明，启动链和清单从规格派生 |
| 运行时扩展 | 各能力归属包 `RuntimeRegistrySpecs()` | `storage.RuntimeRegistrySpecs()` / `file.RuntimeRegistrySpecs()` / `collectorx.RuntimeRegistrySpecs()` | 各包声明自身注册入口，`bootstrap` 直接汇总并派生注册清单 |
| 通用收集器 Processor | `internal/infra/collectorx/types.go` / `internal/infra/collectorx/manager.go` | `RegisterProcessor(...)` / `RegisterProcessorFunc(...)` / `Enqueue(...)` | 业务投递结构化事件，并按 bizType 注册批量消费逻辑 |

补充说明：

- `manifest.Default()` 由 `internal/bootstrap/manifest` 汇总组件、路由、任务插件和运行时扩展规格派生；启动阶段通过 `internal/bootstrap/register` 校验真实注册集合的名称冲突。
- 如果新增的是默认内置能力，应更新对应规格；注册清单会从规格自动派生。
- 如果新增的是外部注入能力，仍可通过 `bootstrap.WithComponents(...)`、`bootstrap.WithTaskPlugins(...)`、`bootstrap.WithRouteModules(...)` 扩展，但默认内置项不要散落到调用方。

#### 1.7.5 通用收集器 bizType 注册点

当前通用收集器的业务接入入口已经统一收口在 `internal/infra/collectorx`：

| 场景 | 文件 | 方法 | 说明 |
| --- | --- | --- | --- |
| 事件定义 | `internal/infra/collectorx/types.go` | `Event` | 业务投递的结构化事件，不允许放 SQL |
| Processor 接口 | `internal/infra/collectorx/types.go` | `Processor` / `ProcessorFunc` | 业务批量消费接口，支持直接传函数 |
| 业务主动注册 | `internal/infra/collectorx/manager.go` | `RegisterProcessor(...)` / `RegisterProcessorFunc(...)` | 按 bizType 注册批量消费逻辑 |
| 业务投递事件 | `internal/infra/collectorx/manager.go` | `Enqueue(...)` | 同步写 Kafka，写失败直接返回错误 |

规则约束：

- 业务只投递结构化事件，不投递 SQL 语句。
- `EventID` 必须使用稳定业务键；任务开启 Collector 幂等时会按 EventID 去重。
- Processor 内部只负责聚合、批量 upsert、落表和逐事件结果返回，不在业务表额外增加去重字段。

## 2. 新增业务功能

这里的“新增功能”指新增一个业务接口，通常包含数据结构、业务逻辑、路由注册、文档和审计。

### 2.1 标准改动路径

建议按以下顺序新增：

1. 定义请求/响应结构
2. 实现业务逻辑
3. 挂接 Handler
4. 注册路由
5. 补齐审计与路由元信息
6. 补齐 i18n / codes / API 文档
7. 补测试

### 2.2 需要改哪些文件

#### A. 请求与响应结构

位置：

- `internal/types`

常见改动：

- 新增 `Req` / `Resp` 结构体
- 为结构体和字段补中文注释
- 为请求结构添加 `Validate()` 方法

参考文件：

- `internal/types/task.go`
- `internal/types/admin.go`

#### B. 业务逻辑

位置：

- `internal/logic/<领域>`，新增业务逻辑不要继续在根目录平铺。RBAC 统一放在 `internal/logic/rbac`，缓存管理和登录态缓存统一放在 `internal/logic/cache`。

常见改动：

- 新增 `XxxLogic`
- 复用 `BaseLogic`
- 明确使用 `ReadDB()` 或 `WriteDB()`
- 事务统一优先使用 `Transaction(func(tx *gorm.DB) error)`

参考文件：

- `internal/logic/task/task.go`
- `internal/logic/admin/admin.go`

#### C. 数据访问

位置：

- `internal/model`

常见改动：

- 新增实体模型
- 复用统一查询 / Repo 风格
- 不要堆重复 CRUD，优先复用现有查询和 Repo 风格

说明：

- 如果只是组合现有模型查询，可能无需新增 model 文件
- 如果要落新表，除了 model 代码，还要补 SQL

#### D. Handler

位置：

- `internal/handler`

常见改动：

- 新增 `XxxHandler`
- 调用 `ActionHandler[...]` 或 `actionLogHandler`
- 把请求参数转换为 `logic` 调用

参考文件：

- `internal/handler/task.go`

#### E. 路由注册

位置：

- `internal/handler/routes.go`
- `internal/handler/<module>/routes.go`

常见改动：

- 在对应领域的 `RouteSpecs()` 中新增路由规格
- 如果是新模块，新增模块 `routes.go` 并在 `BuiltinRouteModuleSpecs` 接入 `Routes` 来源
- `RouteSpecs()` 是领域路由唯一入口，不再额外维护领域注册函数
- 健康探针或高频轮询接口如需跳过普通访问日志，在路由规格设置 `SkipAccessLog`

参考文件：

- `internal/handler/routes.go`
- `internal/handler/<module>/routes.go`

#### F. 审计与路由元数据

位置：

- `internal/handler/common.go`
- `internal/handler/route_meta.go`

常见改动：

- 给新接口补 `RouteMeta`
- 被 `RouteMeta`、安全策略或白名单复用的路由别名，先在 `internal/routealias` 增加常量
- 在 `actionLogRegistry` 中注册 action
- 补齐中文注释，保证审计日志可读

#### G. 国际化与状态码

位置：

- `common/codes`
- `common/i18n`

常见改动：

- 新增业务码先维护 `common/codes/resp_code.go` 常量，再在 `common/codes/contract.go` 补 HTTP 状态、成功标识和默认 i18n key
- 新增默认 i18n key 后同步 `common/i18n/locales/active.zh-CN.json` 和 `common/i18n/locales/active.en-US.json`
- 逻辑层统一返回 `BizResult`

#### H. API 文档

位置：

- `docs/site/接口文档`

常见改动：

- 增加接口说明或示例文档
- 如路由新增较多，更新 `文档首页` 或专题文档

### 2.3 新增功能 checklist

新增一个接口时，建议逐项确认：

- `types` 已补齐请求/响应结构和校验
- `logic` 已完成主流程与错误处理
- `handler` 已挂接
- `routes` 已注册
- `route_meta` 和 `actionLogRegistry` 已补充
- 高频低价值接口已评估访问日志策略，必要时在模块 `RouteSpecs()` 设置 `SkipAccessLog`
- `codes` / `i18n` 已补齐
- `docs/site/接口文档` 已更新
- 单测或路由测试已补充

## 3. 新增后台任务

这里的“后台任务”指异步任务，不一定是周期任务，也可以是接口触发或内部事件触发。

### 3.1 当前任务系统组成

- `internal/svc/taskqueue.go`：任务系统对业务层暴露的接口
- `internal/task/queue`：任务引擎、Worker、调度器、工作流状态
- `internal/task/runtime`：任务插件注册中心和核心插件
- `internal/jobs/<业务>/task`：业务任务插件、handler 和 workflow 定义
- `internal/logic/task/task.go`：任务管理接口逻辑
- `internal/handler/task.go`：任务管理接口入口

### 3.2 新增一个普通异步任务

如果只是新增一个任务类型，不需要 DAG 工作流，可按下面步骤做。

#### 第一步：定义任务类型和负载

位置：

- `internal/task/queue/manager.go`

常见改动：

- 新增任务类型常量
- 定义 payload 结构体
- 如需统一 header 透传，复用 `newTask()`

#### 第二步：注册任务处理器

位置：

- `internal/task/runtime/core_plugin.go` 或 `internal/jobs/<业务>/task/plugin.go`

常见改动：

- 在业务插件中调用 `Runtime.RegisterHandler(...)`
- 新增处理函数并完成 payload 反序列化
- 在处理函数中调用对应 `logic`

#### 第三步：从业务层触发任务

触发方式有两种：

- 通过 `svc.Task.EnqueueTask(...)` 直接投递
- 通过更高层封装后的专用方法触发

常见调用位置：

- `internal/logic/<领域>`
- `internal/handler`
- 其他任务处理器内部

### 3.3 新增任务时要改哪些点

- `internal/task/queue/manager.go`：任务类型 / payload / 入队能力
- `internal/jobs/<业务>/task/plugin.go`：处理器注册
- `internal/logic/<领域>/*.go`：业务触发入口
- `internal/svc/taskqueue.go`：如果需要暴露新接口，则更新抽象
- `common/i18n` / `common/codes`：如果要对外返回任务结果

## 4. 新增工作流

这里的“工作流”指 DAG 编排任务，由多个节点串联而成。

### 4.1 工作流注册入口

工作流统一在启动阶段注册：

- `internal/bootstrap/wiring.go`
- `internal/bootstrap/app.go`
- `internal/bootstrap/components`
- `internal/bootstrap/manifest`
- `internal/jobs/plugins.go`

实际注册调用：

- `taskruntime.Register(svcCtx, manager, taskruntime.ComposePlugins(... )...)`

### 4.1.1 启动阶段注入外部任务插件

标准部署入口可直接使用：

```go
app, err := bootstrap.WireDefault(context.Background(), *configFile, bootstrap.ModeAll)
```

如果某个环境或业务版本需要在默认插件之外追加任务能力，可以直接在启动链注入：

```go
app, err := bootstrap.Wire(
	context.Background(),
	*configFile,
	bootstrap.ModeAll,
	bootstrap.WithTaskPlugins(customPlugins...),
)
```

也可以同时覆盖启动展示名：

```go
app, err := bootstrap.Wire(
	context.Background(),
	*configFile,
	bootstrap.ModeAll,
	bootstrap.WithDisplayName("admin-enterprise"),
	bootstrap.WithTaskPlugins(customPlugins...),
)
```
- `manager.RegisterWorkflow(...)`

### 4.2 新增一个工作流的步骤

#### 第一步：定义工作流

位置：

- `internal/task/queue/workflow.go`
- `internal/jobs/<业务>/task` 下定义业务插件 `Setup(...)`
- `internal/jobs/plugins.go` 使用 `NewPluginFunc(...)` 装配默认业务插件

需要定义的信息：

- 工作流名称
- 默认队列
- 节点列表
- 节点依赖关系
- 每个节点的 `TaskType`
- 每个节点的 `BuildPayload`
- 是否支持分片
- 是否支持灰度
- 重试 / 超时 / 去重策略

参考实现：

- `cache.refresh` 工作流

#### 第二步：注册工作流

位置：

- `internal/task/runtime/core_plugin.go` 或 `internal/jobs/<业务>/task/plugin.go`

常见改动：

- 调用 `manager.RegisterWorkflow(...)`

#### 第三步：提供触发入口

如果要支持人工触发，需要关注：

- `internal/types/task.go`
- `internal/logic/task/task.go`
- `internal/handler/task.go`
- `internal/handler/routes_task.go`

如果只需要内部触发，可以直接在逻辑层或其他任务中调用：

```go
svcCtx.Task.EnqueueWorkflowTrigger(ctx, req)
```

### 4.3 新增工作流时常改文件

- `internal/task/queue/workflow.go`
- `internal/task/queue/manager.go`
- `internal/jobs/<业务>/task/plugin.go`
- `internal/types/task.go`
- `internal/logic/task/task.go`
- `internal/handler/task.go`
- `internal/handler/routes_task.go`
- `docs/site/角色文档/后端开发/任务队列与工作流指南.md`

## 5. 新增定时任务

当前“定时任务”的本质是“周期性触发某个工作流”，配置驱动入口在：

- `internal/config/config.go`
- `etc/config.sample.yaml`
- `internal/task/queue/manager.go`

### 5.1 只新增一个调度项，不新增工作流

如果你只是想定时执行已有工作流，例如 `cache.refresh`，只需要在“任务运维 / 运行配置”新增周期任务草稿并发布：

#### 对应配置项

```yaml
task_periodic:
  - name: "refresh-cache-every-10s"
    # enabled 省略时默认启用；需要停用时显式写 enabled: false。
    # 6 段 cron 的第一段是秒；这里表示每 10 秒触发一次。固定秒级间隔也可以写 every_seconds: 10。
    cron: "*/10 * * * * *"
    workflow: "cache.refresh"
    queue: "maintenance"
    shard_total: 2
    gray_percent: 100
    retry: 3
    timeout_seconds: 300
    unique_key: "periodic:cache.refresh"
    unique_ttl_seconds: 60
```

生效路径：

1. `internal/config/config.go` 解析到 `TaskPeriodicConfig`
2. `internal/bootstrap/configload` 校验 `task.redis` 半配置
3. `internal/bootstrap/components/builtin/task.go` 根据 `task.redis` 选择任务系统使用的 Redis
4. `internal/task/queue/manager.go` 的 `periodicConfigs()` 转换成 Asynq `PeriodicTaskConfig`
5. `StartScheduler()` 启动 leader 后开始同步周期任务

### 5.2 新增一个全新的定时任务

如果定时任务背后是新工作流，建议按下面顺序处理：

1. 先完成“新增工作流”
2. 再通过运行配置页面新增 `task_periodic` 草稿并发布
3. 更新 `docs/site/角色文档/后端开发/任务队列与工作流指南.md`
4. 更新 `文档首页.md`

### 5.3 定时任务改动清单

- `internal/config/config.go`：如需新增配置字段
- `etc/config.sample.yaml`：如需调整运行配置来源或默认说明
- 运行配置页面：新增或修改周期任务草稿并发布
- `internal/task/queue/manager.go`：周期任务转换逻辑
- `internal/jobs/<业务>/task/plugin.go`：新工作流注册
- `docs/site/角色文档/后端开发/任务队列与工作流指南.md`：补文档

## 6. 推荐的开发顺序

### 6.1 新增普通业务接口

1. 先补 `types`
2. 再写 `logic`
3. 再挂 `handler` 和 `routes`
4. 再补 `route meta`、`i18n`、`codes`
5. 最后补文档和测试

### 6.2 新增任务 / 工作流

1. 先定义 payload 和任务类型
2. 再实现处理器
3. 再注册工作流或处理器
4. 再补触发入口
5. 最后补配置、文档、验证

## 7. 提交前检查

建议提交前至少执行：

```bash
go test ./...
go vet ./...
```

如果改动了任务系统，再额外确认：

- `GET /api/tasks/queues` 可用
- 手动触发工作流接口可用
- Redis 中能看到任务入队
- Worker 日志能看到任务执行
- Scheduler 模式下周期任务会按预期注册
