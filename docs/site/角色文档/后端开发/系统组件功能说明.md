# 系统组件功能说明

本文说明 `admin` 与 `api` 当前共用或协作的基础组件，重点回答每个组件负责什么、运行在哪里、如何串联、依赖什么，以及当前可用边界。实现细节以源码和专项文档为准；本文不复制完整配置表和接口字段。

## 阅读约定

组件状态按以下口径描述：

| 状态 | 含义 |
| --- | --- |
| 可生产使用 | 代码、运行入口、失败处理和观测链路已经具备；上线仍需正确配置外部依赖 |
| 条件可用 | 核心能力完整，但是否启用取决于存储、外部软件、运行模式或业务选择 |
| 扩展基础 | 通用框架存在，当前没有完整生产业务处理器，不能按已上线业务能力理解 |

主要运行角色：

| 角色 | 主要职责 |
| --- | --- |
| `api` | 前台 HTTP、用户认证、用户资料读写、Collector 事件生产 |
| `admin` API 模式 | 后台 HTTP、任务投递与查询、文件和配置管理 |
| `admin` Worker 模式 | Collector、Asynq、CDC、归档和导出等后台消费与执行 |
| `admin` Scheduler 模式 | 周期任务选主与调度 |
| migrate 命令 | 执行各仓库自己的数据库迁移 |

## Collector 批量处理

### 服务分工

Collector 不是 API 远程调用 Admin 的 Go 接口。两个服务通过 Kafka 事件契约协作：

1. API 或 Admin 调用本进程的 `Collector.Enqueue`，写入统一结构的事件。
2. Kafka 保存并按 `partitionKey` 分区。
3. Admin Worker 消费事件，按 `bizType` 找到本地注册的处理器。
4. Admin 处理器实现 `ProcessBatch(context.Context, []Event)`，批量落库或聚合指标。

事件契约包含 `eventId`、`bizType`、`partitionKey` 和 `payload`。API 与 Admin 必须保持 `bizType -> topic`、字段大小限制和序列化格式一致。`ProcessBatch` 只属于 Admin 消费端；API 只需要生产事件，不需要也不能跨服务实现这个接口。

当前生产链路包括：

- `admin_log.audit`：Admin 产生审计事件，Admin Worker 批量写入 `admin_log`。
- `auth.security`：API 产生认证安全事件，Admin Worker 批量聚合 Prometheus 指标。

核心源码：

- Admin：`internal/infra/collectorx`、`internal/bootstrap/components/builtin/collector.go`
- API：`internal/infra/collectorx`、`internal/logic/auth/auth_event.go`

### 批处理与可靠性

- Kafka producer 的网络批量发送与业务 `ProcessBatch` 是两个独立层级，不共用批量参数。
- Admin 按 Kafka partition 建立有序消费通道，同一分区内保持顺序；不同分区可并行。
- 单批只交给一个 `bizType` 的处理器，处理器可以返回逐事件结果，表达部分成功和部分失败。
- Redis 使用 `(bizType,eventId)` 作为幂等边界，并通过处理租约、心跳、重试和失败账本恢复异常任务。
- 只有事件成功或失败结果已可靠记录后才推进消费进度；Kafka 语义仍是 at-least-once，业务处理器必须幂等。
- 未注册 `bizType`、处理器 panic、超时或漏返回结果都会进入失败处理，不会静默当作成功。

项目已删除重复的 `pkg/batchprocessor`，当前只保留 Collector 这一套生产批处理框架。

### 扩展规则

新增 Collector 业务时必须同时完成：

1. 在生产端和消费端声明同一个 `bizType` 与 topic 映射。
2. 定义稳定、可版本化且不含敏感明文的 payload。
3. 在 Admin 启动期注册 `Processor` 或直接注册 `ProcessorFunc`。
4. 证明事件重放和部分批次重试不会产生重复副作用。
5. 补生产、消费、幂等、失败恢复和配置漂移测试。

## DAG 工作流

DAG 建立在 Asynq 与 Redis 状态之上，负责任务依赖、节点分片、灰度、业务重试、结果聚合和人工重跑。

主要流程：

1. 启动工作流并写入元数据、节点定义和初始状态。
2. 调度所有依赖已满足的节点。
3. 按节点的分片数生成 Asynq 任务。
4. Worker 执行单个分片并原子记录结果。
5. 节点全部分片收口后推进后继节点，最终收口整个工作流。

正确性边界：

- 节点状态、分片结果和工作流终态通过 Redis Lua 原子迁移，避免并发完成时重复推进。
- 每个分片使用由 `workflowID + nodeName + shardIndex` 生成的确定性 Asynq Task ID。
- 调度可以安全重试：已投递分片由 Task ID 去重，未成功投递的分片会被补齐，不依赖单个全局 `scheduled` 标记。
- 失败分片、归档任务重跑和人工重跑都有明确状态校验，成功终态不会被错误复活。
- 工作流元数据和任务运行快照按配置窗口清理；Asynq 成功终态固定保留 48 小时，失败归档终态按配置清理。

核心源码位于 `internal/task/queue/workflow.go` 和 `internal/task/queue/assets`。接入方法见[任务队列与工作流指南](角色文档/后端开发/任务队列与工作流指南.md)。

## 周期任务与 Leader

周期任务由 Scheduler 模式负责，多个 Scheduler 实例通过 Redis 租约选出唯一 Leader。

- Leader 锁带实例所有者标识和 TTL，续租、释放均校验所有者。
- 续租失败或租约丢失后，实例停止以 Leader 身份投递任务。
- 周期任务首次同步失败会阻止调度器启动，避免空配置或半配置运行。
- 后续同步失败保留上一份已生效配置；成功同步才替换调度项。
- 真正入队前再次校验 Leader 租约和队列背压，缩小切主与积压窗口。
- Leader 只决定谁负责调度，不限制多个 Worker 并行消费 Asynq 任务。

核心源码位于 `internal/task/queue/leader.go` 和 `internal/task/queue/periodic_scheduler.go`。

## Asynq 队列

Asynq 负责普通异步任务、延迟任务、周期任务和工作流分片的承载。

主要能力：

- 多队列权重或严格优先级、并发数、超时、截止时间和最大重试次数。
- API 模式可投递和查询；Worker 模式执行任务；Scheduler 模式运行周期调度。
- 任务 handler 通过插件清单在启动期注册，未知任务类型不会被静默处理。
- Worker、Scheduler、任务 Redis 和运行心跳进入 readiness；当前运行模式要求的后台循环未启动时不会假就绪。
- 停机使用统一 deadline，等待 Worker、Scheduler、Leader 和后台清理协程收口。
- 成功任务在 completed 固定保留 48 小时；失败归档任务按批次和单轮预算清理，避免长 Lua 阻塞 Redis。

任务队列可以使用独立 `task.redis`，也可以复用全局 Redis。队列名、Redis key、调度锁和幂等键都带 `app_id`/部署作用域，跨进程配置必须一致。

## 分布式锁与租约

项目使用分布式锁解决多个实例同时修改同一份共享状态的问题。锁只保护真实临界区，不能替代数据库事务、幂等键或任务状态机。

### Redis 互斥锁

`internal/infra/redsync` 基于 `go-redsync` 提供统一 Redis 互斥锁：

- 锁 key 带 `app_id` 业务命名空间，同一业务资源的所有实例竞争同一个 key。
- 加锁使用随机所有者值和 TTL，不会生成永久锁。
- 持锁期间按 TTL 的一半自动续期，续期失败会报告 `ErrLockLost` 并取消传给业务函数的 context。
- 释放与续期串行执行，解锁只删除当前持有者的锁。
- 锁竞争返回 `ErrLockTaken`，调用方按业务语义返回“服务繁忙”、复用已有任务或跳过本轮周期任务，不在无锁状态下继续执行。

当前主要使用场景：

- 同一上传会话的分片合并和完成收口。
- 相同条件的管理员/用户导出任务创建与结果复用。
- 系统配置 Excel 导入、导出。
- 角色权限写入，避免并发编辑导致可分配范围校验与落库交叉。
- 归档规划、删除和历史清理等不能并发执行的控制段。
- 用户标签异常 outbox 扫描和运行期辅助表清理。

业务函数必须使用锁提供的 context 执行可取消操作。锁丢失后不能继续提交受保护状态；无法及时响应 context 的长操作不应直接放进通用互斥锁临界区。

### 长期租约与所有权

运行时间可能超过固定 TTL、并且需要故障接管的后台任务使用显式租约，而不是一直持有一个普通互斥锁：

- Scheduler Leader：Redis 保存实例 owner，定时续租，投递前再次确认仍是 Leader，释放时校验 owner。
- Snowflake worker：按 namespace 领取节点租约，续租失败后停止使用旧 worker ID 发号。
- 归档区间：数据库状态保存 worker、租约时间和 fencing token，旧 worker 即使恢复也不能覆盖新 owner 的 checkpoint。
- 用户标签工作流：Redis owner 租约配合 Lua 续租、释放和分片完成迁移。
- Collector：处理租约和失败重试租约用于恢复崩溃中的事件，不把超时记录永久视为运行中。

需要长期接管的状态必须同时具备 TTL、owner、续租、所有者校验释放和过期恢复；涉及数据库写入时再增加 fencing 或条件更新，防止旧持有者晚到写入。

### 其它互斥与去重机制

- DB 迁移使用 MySQL 命名锁 `app:schema-migration`，保证 Admin/API 不会并发修改同一数据库结构；它不是 Redis 业务锁。
- DAG 分片确定性 Task ID、Asynq Unique、Collector `(bizType,eventId)` 和签名防重放 key 属于幂等或去重，不等同于持锁执行临界区。
- Redis `SETNX` 初始化 Segment 高水位是一次性原子初始化，不是需要续租和解锁的分布式锁。
- 进程内 `sync.Mutex` 只保护当前进程内存，不能解决多实例竞争。

新增锁前先确认共享资源、竞争入口和锁粒度；锁 key 必须稳定且低基数，TTL 必须覆盖真实临界区并支持续租或失败退出。禁止用大范围全局锁包住无关业务，也禁止因为锁存在就省略数据库唯一约束、事务和幂等校验。

## Redis 分布式限流

`internal/infra/redislimit` 提供两种语义明确、状态隔离的 Redis 分布式限流能力，支持项目现有 Redis 单机和 Cluster：

- `Allow` 基于 `github.com/go-redis/redis_rate/v10` 的单 Key GCRA，适合通用配额、补充速率和允许突发的场景。调用方传入 `Limit{Rate, Burst, Period}`，并显式处理 `Allowed=false` 与 `RetryAfter`。
- `WaitFixedInterval` 使用 `SET NX PX`、活跃令牌指纹和同 Key 进程内有界等待队列，适合数据库批量写入等要求“无突发、固定起点间隔”的场景。暂时无许可时会在当前调用内等待并重试，直至获批、deadline 到期或发生错误；组件本身不持有任务，调用方必须传播失败并由任务层决定是否重试。

两种方法都显式接收稳定的低基数 `scope` 与规范化、非敏感的 `resource`。GCRA 的物理 Key 为 `rate:app:<app_id>:rate_limit:<scope>:<resource>`；固定间隔 Key 为 `app:<app_id>:rate_limit:fixed_interval:<scope>:<resource>`，值格式为 `fixed-v1:<interval_ms>`，TTL 等于 interval。固定间隔状态不复用 GCRA 状态；已有令牌未过期时，混用不同 interval 会失败关闭且不泄露实际 Key 或 Value。

活跃令牌指纹不是持久规则注册中心。所有并行运行的调用方必须为同一 Key 固定使用同一 interval；调整 interval 时，必须先停止并排空旧规则调用方、等待旧令牌过期，再只启动新规则实例，禁止新旧规则混合滚动。组件允许排空后的协调变更，但无法阻止已经违反该调用契约的旧实例在后续令牌过期时重新竞争。

`NewServiceContext` 在 Redis 可用时自动创建进程级 `RedisLimiter`，`ScopedWithContext` 复用同一指针，避免请求作用域产生多套本地等待队列。调用示例：

```go
waitCtx, cancel := context.WithTimeout(ctx, time.Minute)
defer cancel()
if err := svcCtx.RedisLimiter.WaitFixedInterval(waitCtx, "user_tag", "rolling_uid:write", 200*time.Millisecond); err != nil {
	return errors.Tag(err)
}
// 仅在获得许可后执行一次受保护的数据库写入尝试。
```

`WaitFixedInterval` 强制调用方传入带 deadline 的 context；无 deadline 会在访问 Redis 前失败。调用方必须在 Redis 错误、超时、活跃令牌规则冲突或本地队列过载时停止受保护操作，禁止失败后直接绕过。MySQL 瞬态重试也要在每次 SQL 尝试前重新获取许可。`WaitFixedInterval` 只限制相邻成功许可的发放时间，许可返回后仍可能受网络、调度或 GC 延迟影响；它不等待 SQL 完成，也不是分布式锁。若还要求严格 SQL 启动间隔或禁止执行重叠，应使用中央执行队列或另行设计具备 owner、TTL 和续租的互斥租约。

当前组件已经注册到 `ServiceContext`，但 Admin 尚未接入具体 HTTP、Worker 或 Scheduler 业务入口。业务接入时还需同步补充调用结果、等待时长和失败次数指标。`ark-admin-cron` 的 `clean_pay` 已统一使用 `scope=user_tag`、`resource=rolling_uid:write`、`interval=200ms` 的固定间隔协议；两个服务只有在 `app_id`、Redis 实例/DB 和 interval 全部一致时才共享同一额度，否则保持隔离。跨服务修改 interval 时必须协调停止旧调用方并排空令牌，禁止不同规则混合滚动。

核心实现位于 `internal/infra/redislimit`，Key 规则位于 `common/rediskeys/rate_limit.go`。默认测试覆盖并发额度、固定间隔等待、活跃规则冲突、队列上限、取消竞态、应用/Key 隔离、TTL、异常返回、超时和失败关闭；`make integration-test` 在真实 Redis 单机和三主节点 Cluster 验证两种算法的跨客户端共享及节点路由。

生产 Redis ACL 中，GCRA 需要 `EVALSHA`、回退 `EVAL` 及脚本内 `TIME`、`GET`、`SET` 命令权限，并允许访问 `~rate:app:<app_id>:rate_limit:*`；固定间隔需要 `SET`、`MULTI`、`EXEC`、`GETRANGE`、`PTTL` 及 `~app:<app_id>:rate_limit:fixed_interval:*` 权限。仓库真实 Redis 集成环境当前未启用受限 ACL，因此发布前仍需在目标 ACL 用户上单独预检权限，不能把集成测试通过等同于 ACL 已验证。

## table-cache 表缓存

Admin 和 API 都使用 `github.com/Is999/table-cache` 统一管理从 MySQL 派生的 Redis 缓存。该组件在各自进程内运行，API 不会跨服务调用 Admin 回源。它不是通用 Redis 封装，只托管已注册且具备明确回源规则的表数据缓存。

当前 Admin 已有完整生产调用链。API 已实现 Manager、Key 命名空间、`config_uuid` 目标和 `SysConfigLogic`，但生产业务代码尚未调用 `GetCachedValue`，因此目前只是可直接接入的基础能力。后续 API 出现真实表缓存场景时，应继续注册本地目标并调用同一 Manager，不再新建第二套缓存框架。

### 目标注册与 Key 边界

Admin 和 API 分别在本仓库的 `tableCacheTargets` 维护唯一目标清单。每个目标直接声明索引、Key 或 Key 模板、Redis 类型、TTL、空值策略和 Loader。Admin 当前覆盖：

- 角色树、角色状态、角色权限、管理员角色与聚合权限。
- 权限树、权限模块和权限码。
- 管理员公开资料、系统配置、运行配置状态与发布快照。
- 秘钥路由和版本化 AES/RSA 秘钥引用。

API 当前只注册 `config_uuid` 系统配置目标，回源主库并支持空值占位；在没有真实业务调用方前不提前增加其他目标。

物理 Key 统一为 `app:{app_id}:table:{logical_key}`。逻辑 Key、物理 Key、搜索、删除和回源共用同一组转换函数，完整 Key 属于其他 `app_id` 时直接拒绝，避免多站点共用 Redis 时串读或误删。

### 读穿、回源与防击穿

1. 业务按目标读取 Redis，命中时直接返回。
2. 未命中时由目标 Loader 使用 GORM 回源 MySQL，再按 String、Hash 或 Set 写回 Redis。
3. 系统配置、运行配置和秘钥等要求强一致的目标显式读主库；其他目标按已定义的读库边界回源。
4. 选定的模板目标支持空值占位和独立空值 TTL，防止不存在的记录持续打到数据库。
5. 并发 miss 使用 Redis 回源锁，默认锁 TTL 为 10 秒；其他请求每 80 毫秒重试读取，最多等待 3 次，避免同一 Key 被多实例同时回源。

Loader 错误会返回业务链路，不把查询失败写成有效缓存。回源锁只合并同一 Key 的并发查询，不代替写业务的事务和缓存失效。

### 写后失效、管理与指标

- Admin 业务写入成功后按受影响的角色、管理员、权限、配置或秘钥 Key 精确失效；API 的 `SysConfigLogic` 支持按 UUID 精确刷新。两个服务都不应在请求链路使用 `KEYS` 或无边界 `SCAN`。
- 批量失效每批最多 200 个 Key，Redis Cluster 中用 pipeline 发送多条单 Key `DEL`，避免多 Key 跨 slot 失败。
- 缓存管理页只允许查询和刷新内置目标，支持单 Key 回源和显式全量刷新；跨 `app_id` Key 与非内置 Key 不能借该接口刷新。
- 管理页对 token、秘钥和联系方式等敏感值脱敏，缓存搜索候选由数据库有界枚举，不允许任意通配扫描 Redis。
- Admin 和 API 的 Prometheus 指标统一使用组件级 `tcache_*` 前缀，记录命中、miss、刷新、Loader 错误、锁失败、等待超时和批量刷新结果。服务和实例由 Prometheus 抓取目标标签区分；缓存管理页仅展示当前 Admin 进程快照，API 暂无管理页。

核心实现位于 Admin 的 `internal/logic/cache`、`internal/logic/cachemanage`，以及 API 的 `internal/logic/cache`、`internal/logic/config/sys_config.go`。

## 归档任务

归档任务负责把满足时间和主键边界的数据从热表复制到历史表，并在校验后分批删除热表数据。

处理阶段：

1. 根据配置规划连续归档区间。
2. Worker 领取区间租约，按主键游标批量复制。
3. 校验目标表结构、行数和区间完整性。
4. 仅对连续完成的区间推进归档水位。
5. 删除任务领取独立租约，分批删除已经确认复制的数据。
6. 按保留策略清理过期历史数据。

可靠性边界：

- 区间状态迁移带 worker 所有者和 fencing token，过期 worker 不能覆盖新 worker 的进度。
- 复制和删除都在长循环中持续续租；每批删除前会再次确认租约。
- 删除前确认历史表已经包含源数据，删除后精确校验 `RowsAffected`。
- checkpoint 只在批次成功后推进，任务重试从已确认位置继续。
- 批量大小、并发目标数和批间延迟可限制主库压力。

核心源码位于 `internal/jobs/archive`。归档不是跨库事务，正确性依赖幂等复制、区间校验、租约和水位推进共同保证。

## 上传与下载

文件传输支持本地存储和 S3 兼容对象存储，业务策略决定允许的后缀、MIME、大小和归属。

上传流程：

- 创建上传会话并校验业务类型、文件名、总大小和分片数量。
- 分片按会话和序号写入；同一分片串行，不同分片可并行。
- 完成时校验分片完整性、实际大小、MIME 和 SHA-256，再生成最终文件。
- S3 模式可以使用服务端中转或预签名直传，完成时仍由服务端确认对象属性和归属。
- 过期且没有有效会话的临时分片按会话 TTL 小批量清理。
- 持久化对象会预投递精确 key 清理任务；有效会话按 TTL 顺延，会话失效后同时回收服务端中转的 `completed` 文件，已引用头像保留，配置导入源文件成功消费后立即删除。

下载流程支持所有权检查、安全文件名、Range 和缓存条件请求；响应写入器保留流式接口，不会因为访问日志、签名或响应封装把大文件整体缓存到内存。

病毒扫描不是生产强制项：

- `virus_scanner.name=noop`：不依赖外部扫描软件，也不执行恶意文件扫描。
- `virus_scanner.name=clamav`：启动时真实检查 `clamdscan/clamd`，上传扫描异常或命中病毒时失败关闭。

是否启用 ClamAV 由部署环境和业务风险决定，不能因为服务器未安装扫描软件而阻止选择 `noop` 的部署启动。

核心源码位于 `internal/logic/file`、`internal/jobs/file_cleanup.go`、`pkg/transfer` 和 `pkg/storage`。

## 导入与导出

### 导入

`pkg/excel.StreamImport` 按行读取工作簿，支持表头校验、最大行数和上下文取消，避免一次把整个文件加载到内存。当前系统配置 Excel 导入是实际生产调用方：先通过文件传输校验上传归属，再在互斥锁内逐行校验并写入。

运行配置的导入是 YAML 配置版本流程，不走 Excel；导入后仍经过草稿、发布和运行边界校验。

### 导出

管理员和前台用户大数据导出通过 Asynq 异步执行：

- 查询使用稳定排序和主键游标分页，不使用大 offset。
- 格式化使用有上限的并发，结果保持原查询顺序。
- 单文件达到 Excel 行数或业务上限后自动拆分。
- Redis 保存任务状态、进度、结果文件和错误，页面切换后仍可继续查询。
- 支持取消、失败恢复、完成结果短期复用和过期文件清理。
- 下载继续复用文件传输的归属、Range 和对象存储能力。

核心源码位于 `pkg/excel`、`internal/jobs/export`、`internal/logic/admin/admin_export.go` 和 `internal/logic/user/user_export.go`。

## 配置热加载

Admin 与 API 都维护当前配置快照、最近检查时间、成功/失败状态和是否需要重启。

- 文件热加载重新读取启动时绑定的配置文件。
- Admin 还支持从数据库 active release 读取运行配置；启动时先应用 active release，再注册组件。
- watcher 启动时记录当前 active release 水位，不会把启动时已应用的版本再次当作新版本执行。
- 同一进程的热加载串行执行，失败时保留上一份有效配置。
- 日志级别、部分业务参数等明确支持的字段可以在线替换。
- HTTP 地址、数据库、Redis、Kafka、组件启停、任务/工作流注册等启动拓扑变化只标记 `restartRequired`，不会在线重建半套组件。

热加载的目标是安全更新已有运行对象，不是动态依赖注入系统。是否需要重启以状态接口返回值为准。

## DB 迁移

Admin 和 API 各自维护迁移 manifest、SQL 资产和迁移记录，禁止依赖运行时 `AutoMigrate` 修改生产结构。

迁移规则：

- 已执行迁移的 checksum 必须与仓库资产一致，禁止静默改写历史 SQL。
- bootstrap 数据和 destructive 迁移有独立门禁，默认不会隐式执行高风险操作。
- Admin 与 API 迁移命令共用 MySQL 命名锁 `app:schema-migration`，最多等待 60 秒，避免同一数据库并发改结构。
- 锁使用独占连接并在成功、失败和取消路径释放。
- 发布前分别检查并执行两个仓库的 migration status/up，不能只运行其中一个。

操作命令和发布顺序见[数据库迁移治理](角色文档/运维/数据库迁移治理.md)。

## Lark 预警

Lark 是可选的外部通知通道，用于任务终态失败、Collector/CDC 异常、热加载失败和应用生命周期异常等运行告警。

- 统一运行告警组件独立于 `task.enabled` 初始化；关闭 Asynq 不会同时关闭非任务类 Lark 告警。
- 任务系统启用时再向 Manager 注册任务专用 hook。
- 告警按稳定指纹限频，动态错误文本不直接进入指纹，避免同类异常刷屏。
- 通知内容包含类别、操作、组件、时间和精简错误，不输出密钥、Token 或大载荷。
- Lark 发送失败会记录运行异常，但不反向改变数据库事务或任务状态。

因此 Lark 不能替代任务失败账本、Collector 失败账本、Prometheus 或日志。

## CDC 数据清洗

`internal/infra/cdcx` 提供 Debezium/Kafka CDC 消费基础能力：

- 按 `database.table` 注册单条事件处理器。
- 校验启用 topic 是否都有处理器。
- 支持处理超时、有限重试、DLQ、状态指标和停止期限。
- readiness 检查 Kafka topic、DLQ 和当前运行模式要求的消费循环。

当前内置业务只有 `admin_log` 开发验证场景，用于验证 CDC -> Collector -> 可选 Lark/观察文件链路。正式配置默认关闭，它不是完整的数据清洗业务。

CDC offset 提交失败时消息可能重复，新增正式处理器必须自行证明幂等；直接发送外部通知等非幂等副作用不能只依赖 Kafka offset。完成真实处理器、幂等键、补偿方式、DLQ 运维和故障注入测试后，才能把对应清洗任务标记为生产可用。

## 运行指标与健康检查

Admin 与 API 通过 Prometheus 暴露 HTTP、依赖、Collector、任务、缓存和 ID 生成等指标。

主要规则：

- `/live` 只表示进程存活；`/ready` 验证当前运行模式必须具备的依赖与后台循环。
- Admin readiness 包含主库、Redis、Task Redis、Worker/Scheduler 心跳、Collector 和启用后的 CDC 等检查。
- Collector 提供累计值和 1/5/15 分钟运行窗口，便于区分历史故障与当前故障。
- Snowflake/Segment 记录租约事件、号段申请、申请耗时和本地剩余量。
- Admin 和 API 的 table-cache 统一注册 `tcache_*` 指标，Admin 管理页按同一前缀读取当前进程快照。
- 指标标签只使用 route、result、bizType、namespace 等受控低基数字段，不把 user ID、trace ID 或原始错误作为标签。

健康检查用于流量摘除，Prometheus 用于趋势和告警，两者不能互相替代。

## 中间件、请求、响应与路由

Admin 与 API 的全局中间件顺序固定为：

```text
outer recover -> trace -> access log -> inner recover -> route security chain -> handler
```

- outer recover 保护入口中间件本身。
- trace 创建 trace/span 和统一请求上下文。
- access log 在请求结束时记录状态、业务码、耗时和错误链。
- inner recover 把业务 panic 转成标准响应，再交给上层日志收口。

路由由 `RouteSpec` 声明 method、path、稳定 alias、访问类型、安全链和文档位置。Admin 区分 public/auth/internal/docs/health；API 区分 none/public/auth/internal。新增路由必须进入规格清单，不能绕过统一注册直接挂载。

统一响应保留 HTTP 状态与业务码的分工。流式下载使用支持 `Unwrap`/flush 等接口的 writer，不会被普通 JSON body recorder 破坏。

核心源码位于两个仓库的 `internal/handler/routes.go`、`internal/handler/shared` 和 `internal/middleware`。

## 本地 IP 归属地解析

`internal/infra/ipregion` 使用 ip2region XDB 在 Admin 进程内解析 IP，不请求外部 IP 服务。当 `ip_region.enabled=true` 时，启动期至少加载一份 IPv4 或 IPv6 XDB，并校验普通文件、IP 版本、结构和索引完整性。文件不存在、损坏、版本错配或启用文件合计超过 128 MiB 时拒绝启动，避免以为已启用但实际全部返回空值。

请求数据链路如下：

1. 客户端 IP 由统一中间件按 `trusted_proxies` 边界解析并标准化，不盲信外部请求伪造的代理请求头。
2. 管理员登录成功时查询一次，同步更新 `admin.last_login_ip` 和 `admin.last_login_ipaddr`。
3. 同一请求的解析结果缓存在请求上下文；审计 Recorder 优先复用，否则再查询本地库，最终随 `admin_log.audit` Collector 事件写入 `admin_log.ipaddr`。
4. 功能未启用、当前 IP 版本未配库、地址无效、公网库未命中或单次查询异常时返回空归属地，不阻断登录和审计投递主链路。

回环地址显示“本机”，私网、链路本地和运营商级 NAT 地址显示“内网”；公网结果只保留国家、省、市，不保存 ISP 和 ISO 编码。XDB 在启动期完整读入只读内存，查询期间不访问磁盘且不保留文件句柄。路径配置或数据文件变更后必须滚动重启；热加载保留旧查询器并标记需重启，不会在运行中替换内存库。

配置含义见[配置字段说明](角色文档/后端开发/配置字段说明.md#ip_region)；XDB 版本、完整下载命令、SHA-256 校验、只读挂载、发布验证和回滚步骤见[部署发布指南](角色文档/运维/部署发布指南.md#可选本地-ip-归属地数据)。

## 加密、签名、MFA 与 API 鉴权

### 签名与加密

- 请求验签、请求解密、响应回签和响应加密由 `RouteSecurityPolicy` 分别声明，可以独立启用。
- 当前前台协议支持 RSA-SHA256 签名和 AES 加密 SHA-256 摘要的签名方式；不再接受旧 MD5 签名。
- 加解密支持 AES-CBC 和 RSA，具体字段由路由策略白名单控制，不默认加密整个 JSON。
- Admin 密钥管理按 app ID 和 key version 路由密钥，敏感字段不写入日志或接口明文。
- API 内网运维接口同时校验来源、Ops Token、时间戳和 HMAC-SHA256 请求签名。

### Admin MFA 验证

Admin MFA 使用 TOTP，用于登录后验证和后台敏感操作二次确认：

- 动态码为 6 位、30 秒周期，校验时允许前后各一个时间窗口，用于容纳必要的终端时钟偏差。
- 绑定二维码使用本次流程的新秘钥，生成二维码时不提前写库；动态码校验通过后才能完成启用或换密钥。
- MFA 秘钥归一化为 Base32，使用 AES-GCM 加密后存入数据库；秘钥、动态码、绑定地址和二次票据都进入日志与审计脱敏清单。
- 强制 MFA 开启时，未绑定或未完成当次登录验证的管理员不能进入普通业务路由。登录通过标记与最近登录时间关联，新登录不会继承旧会话的 MFA 状态。
- 动态码校验通过后，Redis 签发绑定管理员、场景和 TTL 的随机二次票据。MFA 状态、MFA 秘钥和秘钥管理必须使用同场景票据；其他敏感操作可在配置的频率窗口内复用，减少重复弹框。
- Redis 不可用、票据过期、场景不匹配或票据值不正确时失败关闭。账号重置会精确清理该管理员的登录标记和二次票据。

当前没有 MFA 恢复码流程。终端丢失时，由具备权限的管理员通过账号重置或 MFA 管理流程恢复；操作者本身仍需通过 RBAC、MFA 二次验证和审计记录。核心实现位于 `internal/logic/security/security_mfa.go`、`internal/middleware/auth.go` 和 `internal/logic/profile`。

### API 鉴权

API 登录态以 JWT 中的稳定 session ID、唯一 JTI 和 `auth_version` 标识会话：

1. 先校验签名、Redis session、过期时间和撤销状态。
2. 每个受保护请求再读取主库用户状态和当前 `auth_version`。
3. 用户禁用、删除或版本递增后，旧 token 即使暂时仍在 Redis 中也会失败关闭。

Redis 负责快速拒绝和会话生命周期，主库是即时撤销的最终边界。该设计不能为了减少一次查询直接删除；高 QPS 上线前应通过压测确定主库、索引和连接池容量。

## 后台权限与 RBAC

后台权限由 Admin 后端负责最终授权，admin-vue 只负责按同一权限码过滤路由、菜单和操作入口。前端隐藏按钮不能替代后端权限校验。

### 数据模型与字段职责

- 管理员与角色、角色与权限分别通过关系表维护，角色和权限都必须处于启用状态才参与授权。
- `admin_permission.module` 保存后端稳定路由别名，例如 `admin.list`、`task.queue.pause`，用于接口授权。
- `admin_permission.uuid` 是前端权限码，由 `/api/auth/codes` 返回，用于页面和操作显隐。
- `roles` 是启用角色名称，供布局展示和 Vben 路由过滤入参使用。
- `roleIds` 用于角色身份判断，例如识别约定 ID 为 `1` 的超级管理员角色。
- `accessCodes` 才是前端路由、按钮和表格操作列的实际权限集合。

`module` 与 `uuid` 解决不同问题，不能混用，也不能只维护其中一侧。

### 后端授权链

受保护路由通过 `RouteSpec` 绑定稳定 alias，`AuthMiddleware` 按以下顺序校验：

1. JWT、Redis token 和管理员身份是否合法。
2. 管理员是否启用，当前 IP 是否符合白名单和登录 IP 策略。
3. 是否完成强制改密、MFA 绑定和当前登录 MFA 校验。
4. 根据 route alias 查询启用的 `admin_permission.module` 候选权限。
5. 判断当前管理员的启用角色是否拥有任一候选权限。

安全边界：

- 未配置到权限表的业务路由默认拒绝，避免新增接口漏配权限后直接放行。
- `middleware.Ignore` 只跳过业务权限表，JWT、账号状态、IP 和 MFA 仍然校验。
- 个人中心、会话校验等自助闭环接口使用显式最小白名单，不扩大到管理类敏感接口。
- 超级管理员角色 ID `1` 在后端跳过普通业务权限匹配；角色本身的禁用、删除和越权调整有额外限制。
- 文档资源按 `docs.file.<站内相对路径>` 校验单篇文档权限，父目录权限不会自动兜底放行子文件。

### 前端权限链

admin-vue 登录后读取 `/api/auth/profile` 和 `/api/auth/codes`，构造当前会话的有效权限状态：

- 路由与菜单使用 `meta.authority`。
- 普通按钮使用 `v-access`。
- 表格操作列使用 `auth`。
- 权限常量集中维护在 `apps/web-antd/src/constants/permission-codes.ts`。
- 超级管理员根据后端 `isSuperAdmin` 或 `roleIds` 补齐前端已注册权限码，保持与后端放行语义一致。
- 权限状态按当前登录会话隔离；定时同步、接口返回 403 或保存角色权限后会重新获取资料和权限码，必要时重建动态路由与菜单。
- 权限刷新后当前页面已经失权时，前端离开不可访问路由，不能继续依赖旧菜单缓存。

### 修改权限的闭环要求

角色关系、角色权限或权限定义变更后，后端精确失效受影响管理员、角色和展示缓存；权限定义整体变化时再执行有边界的全量权限缓存失效。Redis 缓存只服务权限码、菜单和管理展示，受保护路由始终以主库启用角色与权限关系作为最终授权依据。新增权限必须同时完成：

1. 创建或迁移 `admin_permission` 数据，明确唯一 `uuid` 和准确 `module`。
2. 将权限 ID 绑定到正确角色，并验证非超级管理员的可分配范围。
3. 确认后端 `RouteSpec` alias 与 `module` 一致，接口无权限时返回 403。
4. 在前端权限常量中加入 `uuid`，同步路由 `authority`、按钮 `v-access` 或操作列 `auth`。
5. 验证 `/api/auth/codes`、权限变更后的缓存失效、前端实时刷新和直接调用接口四条链路。

完整映射和排查方法见[前端权限码与管理员权限映射说明](角色文档/前端与测试/前端权限码与管理员权限映射说明.md)。

## 日志、i18n 与审计日志

### 日志

- 使用结构化字段记录 trace、route、状态、耗时和受控错误链。
- 错误在请求、Worker、Scheduler 等顶层统一输出，内层主要返回带业务上下文的包装错误。
- access token、密码、密钥、签名原文和敏感联系方式经过清理或脱敏。
- caller 和 error chain 保留真正失败位置，避免重复包装后只看到公共 handler。

### i18n

- Admin 和 API 都维护中文、英文语言包和稳定业务码。
- 响应按请求 locale 解析文案，日志保留稳定 code/key，不能用中文文本作为程序判断条件。
- 新增业务码必须同步常量、语言包、接口文档和前端提示。

### 审计日志

Admin handler 通过统一 Recorder 合并操作者、路由、trace、结果和脱敏请求数据，再投递 `admin_log.audit` Collector 事件；Admin Worker 批量写入 `admin_log`，`event_id` 提供数据库幂等。

当前业务响应不会等待审计最终落库，Collector 未就绪或投递失败时记录错误但不回滚已成功业务，因此语义是 best-effort 异步审计。普通后台操作可以采用该边界；要求审计必达的强合规业务必须另行设计事务 outbox 或同步失败关闭，不能把现状描述为必达。

## Snowflake 与 Segment

`idgen.NextID(namespace)` 按业务 namespace 选择 ID 策略：

- 默认 Snowflake：从 Redis 节点池领取 worker ID 租约，本地生成 64 位 ID。
- 高吞吐 namespace 可启用 Segment：Redis `INCRBY` 分配全局号段，本地双缓冲取号并在低水位异步预取下一段。

可靠性规则：

- Admin 与 API 共享同一业务 namespace 时，Redis scope 和 key 规则必须一致。
- Snowflake 租约带续租和所有者校验；租约释放或丢失后停止继续使用旧 worker 发号。
- Segment 的每个 namespace 独立配置 step 和预取阈值；Redis 短暂不可用时可以消费已领取的本地号段，耗尽后返回错误而不是生成重复 ID。
- 启用 Segment 的 namespace 不再保证 Snowflake 位格式，业务不能解析 ID 位段推断时间或节点。
- `idgen.ShardNo` 只负责稳定分片计算，与选择 Snowflake 或 Segment 生成 ID 是两个概念。

两个仓库的实现分别位于 `common/idgen` 和 `internal/bootstrap/configload/id_segment.go`。

## user 分片、UID 表与未来扩表

### user 逻辑分片

- `user.id` 是业务 ID，`user.shard_no = CRC32(id字符串) % 1024`，固定范围为 `0-1023`。
- 不能使用 `id % 1024`，因为 Snowflake 低位在低并发时容易产生分片偏斜。
- `user.route_shard_count` 只允许 `1/2/4/.../1024`。
- 单表阶段访问 `user`；多表阶段按逻辑分片范围路由到 `user_0000`、`user_0512` 等范围起点表。
- Admin 与 API 保持相同的 Model、分片公式和物理表名函数，配置变化必须统一发布和重启。

### 登录身份定位

登录不扫描所有 user 物理表，而是先查询独立身份索引：

- `user_identity_username`：归一化账号。
- `user_identity_email`、`user_identity_phone`：只保存 HMAC 查询哈希。
- `user_identity_oauth`：保存 provider 和大小写敏感身份值。

身份记录保存 `user_id`、`user_shard_no` 和 `user_route_shard_count`，可以直接计算目标 user 表。用户资料与身份索引的新增、修改在同一数据库事务中保持一致。

### 包含 UID 字段的业务表

不是所有名为 `user_id` 的字段都属于前台业务 UID。对 DDL 中精确业务列 `uid` 的表，当前规则是：

- 增加 `shard_no = uid % 1024`。
- 写入时同时写 UID 和分片号。
- 至少建立以 `shard_no` 靠前、符合真实查询条件的复合索引。
- GORM Model、DDL、写入逻辑和测试保持同一公式。

例外包括：`user.id` 使用 CRC32；身份索引的 `user_id` 使用 `user_shard_no = idgen.ShardNo(user_id)`；后台管理员 `admin_log.user_id` 不属于前台 UID。用户标签表遵循 UID 取模规则，但这不表示用户标签业务骨架可以直接开启。

### 随量扩表

当前实现提供确定性路由和运维拆分流程，不自动创建物理表或在线回填。扩容按 `1 -> 2 -> 4 -> ... -> 1024` 逐级进行：

1. 计算目标分片范围并创建目标表。
2. 按主键游标小批复制目标范围，记录 checkpoint。
3. 校验行数、缺失样本和身份索引定位。
4. 在维护窗口暂停写入，或运行已经验证的 delta 补偿。
5. 分批更新四类身份索引的 `user_route_shard_count`。
6. 修改 Admin/API 配置并统一重启全部读写实例。
7. 冒烟和观察通过后，再小批清理旧表副本。

不要仅修改 `route_shard_count` 就切流。完整公式、SQL 模板、回滚和校验步骤见[库表路由规范](角色文档/后端开发/库表路由规范.md)。

## 生产接入检查

新增或修改上述组件时至少确认：

1. 真实入口、生产者、消费者和运行模式已经明确。
2. Admin/API 的事件、Redis key、namespace、分片公式和配置字段没有漂移。
3. 超时、取消、重试、幂等、租约和原子迁移只用于真实故障窗口，并有测试覆盖。
4. 大表查询使用索引、游标和批量上限，没有无边界扫描或大 offset。
5. readiness 能反映当前运行模式必需的底层依赖和后台循环。
6. 指标、日志、告警和失败账本各自职责明确，不互相冒充可靠状态。
7. 接口、配置、SQL/Lua、前端类型、i18n 和文档同步更新。
8. 发布说明明确迁移、回填、缓存、重启、补偿和回滚要求。
