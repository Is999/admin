# 配置字段说明

## 文档目标

本文档用于说明 `admin` 的主要配置项用途，方便开发、测试、运维在本地部署、联调和生产发布时快速理解配置字段含义。

推荐配合以下文件一起阅读：

- `etc/config.yaml`
- `etc/config.sample.yaml`

## 顶层配置

### `Name`

- 服务名称。
- 用于日志、链路追踪、监控平台和服务实例标识。

### `Host`

- HTTP 服务监听地址。
- 本地开发通常使用 `0.0.0.0` 或 `127.0.0.1`。

### `Port`

- HTTP 服务监听端口。
- 用于 API、文档站和健康检查访问。

### `app_id`

- 站点或应用 ID，例如 `203`。
- 必填，用于区分不同站点配置、Redis 命名空间、业务数据和下游消息主题。

### `snowflake.worker_id`

- 当前实例生成雪花 ID 使用的 worker ID，范围 `0-1023`。
- 当前实现基于 `github.com/bwmarrin/snowflake`，成熟库保证单节点并发序列安全，但不会自动分配节点号。
- 所有会写 `user` 业务用户表的 `admin`、`api` 实例必须全局唯一；建议按服务预留号段，例如 `api=0-511`、`admin=512-767`。
- 未配置时可通过环境变量 `SNOWFLAKE_WORKER_ID` 注入；配置缺失或越界会启动失败。
- 属于启动期配置，热加载变更只会标记为需重启，不会在线切换当前进程的 ID 生成器。
- 雪花 ID 生成器无持久化状态，重启不会丢失业务数据；同 `worker_id` 极速重启已有 1ms 启动围栏，生产仍必须保证系统时钟稳定，并保留 `user.id` 主键唯一索引兜底。

### `user.route_shard_count`

- 新增业务用户默认写入的物理表数量，仅支持 `1/10/100/1000`。
- 当前样例配置保持 `1`，即写入单表 `user`；提升前必须完成 `user_account` 回填、目标物理表创建和并发更新补偿方案。
- 提升到 `10/100/1000` 后，后台用户列表走账号索引分页定位，只支持 `id/shardNo/username` 这类账号索引字段；其它组合查询需接入搜索或额外后台索引。
- 属于启动期写入路由配置，热加载变更只会标记为需重启；必须统一重启所有会写 `user` 的 `admin/api` 实例后才允许切换。

### `Mode`

- 运行环境标识。
- 支持 `dev`、`test`、`rt`、`pre`、`pro`。

### `run_mode`

- 进程启动模式位掩码。
- 支持：
  - `1`：仅 API
  - `2`：仅 Worker
  - `4`：仅 Scheduler
  - `3/5/6/7`：组合模式
- 当启动命令显式传入 `-mode` 时，以命令行为准。
- 当启动命令未传 `-mode` 时，进程会回退到配置文件中的 `run_mode`。
- 如果 `-mode` 和 `run_mode` 都未设置，最终默认回退为 `7`。
- 除 `0` 表示走配置或默认值外，非法模式会启动失败，不会被按位裁剪成其它模式。

### `jwt_secret`

- JWT 签名密钥。
- 用于接口鉴权相关逻辑。
- 生产环境必须替换为强随机字符串。

## 后台接口安全配置 `security.secret_key`

`security.secret_key` 只服务当前 `app_id`。请求 AppID 不匹配时，秘钥配置从主库和缓存回源。

- `sign_status`：签名验签开关，`1` 启用，`0` 停用。
- `crypto_status`：加密解密开关，`1` 启用，`0` 停用。
- `key_version`：默认秘钥版本，对应 `X-Key-Version`。
- `aes_key_ref` / `aes_iv_ref`：AES KEY 与 IV 文件路径。
- `rsa_public_key_user_ref`：用户 RSA 公钥文件路径。
- `rsa_private_key_server_ref`：服务端 RSA 私钥文件路径。
- `rsa_public_key_server(_ref)`：服务端 RSA 公钥文本或文件路径；为空时由服务端私钥派生。

## 日志配置 `log`

### `log.level`

- 日志级别。
- 常用值：`debug`、`info`、`warn`、`error`。

### `log.mode`

- 日志输出方式。
- `console` 表示输出到控制台，`file` 表示输出到文件。

### `log.path`

- 日志文件目录。
- 当 `mode=file` 时生效。

### `log.encoding`

- 日志编码格式。
- 常用值：`plain`、`json`。

### `log.rotation`

- 日志切割策略。
- 常用值：`size`、`time`。

### `log.max_size`

- 单个日志文件最大大小，单位 MB。

### `log.max_backups`

- 日志归档保留数量。

### `log.keep_days`

- 日志保留天数。

### `log.compress`

- 是否压缩归档日志。

## MySQL 配置 `mysql`

`mysql` 是当前项目唯一必填的 MySQL 启动入口。默认所有业务表都使用该主库连接。

字段结构：

- `write_data_source`：主库 DSN
- `read_data_sources`：从库 DSN 列表
- `max_open_conns`：最大打开连接数
- `max_idle_conns`：最大空闲连接数
- `conn_max_lifetime`：连接最大存活时间，单位秒
- `debug`：是否输出 SQL 调试日志

启动时必须提供 `mysql.write_data_source`。如果需要读写分离，可填写 `mysql.read_data_sources`；为空时所有读写都走主库连接池。

## 命名扩展库配置 `site_mysql`

`site_mysql` 是可选扩展配置，不参与默认部署。只有确实需要把某类表拆到独立 MySQL 时，才补命名节点：

```yaml
site_mysql:
  archive:
    write_data_source: "user:password@tcp(mysql:3306)/admin_archive?charset=utf8mb4&parseTime=true&loc=Local"
    read_data_sources: []
    max_open_conns: 20
    max_idle_conns: 10
    conn_max_lifetime: 300
    debug: false
```

注意：

- 不配置 `site_mysql` 时只使用 `mysql`。
- 扩展库名称只做前后空白整理；命名应短而明确，便于运维识别。

## Redis 配置 `redis`

这组配置用于主 Redis 连接。

### `redis.type`

- Redis 连接模式。
- `single` 表示按单节点 Redis 客户端建连。
- `cluster` 表示按 Redis Cluster 客户端建连。
- 建议显式配置，避免只靠地址数量推断连接模式。

### `redis.addrs`

- Redis 节点地址列表。
- 单机 Redis 填 1 个地址。
- Redis Cluster 建议填写多个可访问节点，客户端会按集群拓扑路由。

### `redis.addr_map`

- 集群节点地址改写表。
- 主要用于宿主机访问容器内 Redis Cluster 的场景。
- 如果集群返回的是容器内部主机名，而宿主机无法解析，就需要通过这个字段把节点名改写为可访问地址。
- 启动预探测和正式 Cluster 客户端都会使用该映射，避免预探测先于正式连接失败。

### `redis.password`

- Redis 认证密码。
- 无密码可留空。

### `redis.db`

- Redis DB 编号。
- Cluster 模式通常固定为 `0`。

### `redis.pool_size`

- Redis 连接池大小。

### `redis.tls`

- 是否启用 TLS 连接 Redis。
- 对云 Redis、托管 Redis、代理层要求 TLS 的场景需要开启。

### `redis.tls_insecure_skip_verify`

- 是否跳过 TLS 证书校验。
- 仅测试环境、代理转发或证书不完整的特殊场景建议开启。
- 生产环境通常建议关闭。

## Kafka 配置 `kafka`

这组配置承载 Kafka 公共连接参数和按业务划分的 Topic。`collector.kafka` 未单独配置 broker、批次和写超时时会继承这里。

### `kafka.enabled`

- 是否启用公共 Kafka 生产者。

### `kafka.brokers`

- Kafka broker 地址列表。
- Collector Kafka 未配置 `brokers` 时会继承该值。

### `kafka.batch_size`

- 单批次最大发送消息数。
- Collector Kafka 未配置 `batch_size` 时会继承该值。

### `kafka.write_timeout`

- Kafka 写超时时间，单位秒。
- Collector Kafka 未配置 `write_timeout` 时会继承该值。

### `kafka.topics.user_tag`

- 可供自定义用户标签事件 hook 使用的 Topic。
- 默认用户标签骨架不直接写入 Kafka。

## 通用收集器配置 `collector`

这组配置用于把高频结构化事件按 Kafka -> Redis Stream -> DB outbox 的顺序可靠收集，再由 Worker 批量投送给 Processor。调用方只投递数据，不投递 SQL。

### `collector.enabled`

- 是否启用通用收集器。
- API-only 模式只保留 `Enqueue` 能力；后台 Kafka/Redis/DB worker 只在 Worker 模式启动。

### `collector.transport`

- 首选载体，支持 `auto`、`kafka`、`redis`、`db`。
- `auto` 表示优先 Kafka，再 Redis Stream，最后 DB outbox 兜底。
- 即使首选 Kafka 或 Redis，写入失败时仍会落 DB outbox，保证不丢事件。

### `collector.kafka.enabled`

- 是否启用 Kafka 载体。

### `collector.kafka.brokers`

- Kafka broker 地址列表。
- 为空时继承 `kafka.brokers`。

### `collector.kafka.topic`

- Collector 事件 Topic。

### `collector.kafka.group_id`

- Kafka 消费组 ID，用于把 Kafka 消息可靠落地到 DB outbox。

### `collector.kafka.batch_size`

- Kafka 写入和读取批次大小。
- 小于等于 0 时继承 `kafka.batch_size`。

### `collector.kafka.write_timeout`

- Kafka 写超时时间，单位秒。
- 小于等于 0 时继承 `kafka.write_timeout`。

### `collector.kafka.read_timeout`

- Kafka 读超时时间，单位秒。

## CDC 配置 `cdc`

消费 Debezium Kafka 事件，仅 worker 模式启动；变更后需重启。

### `cdc.enabled`

- 是否启用 CDC 消费器。

### `cdc.brokers`

- Kafka broker 地址列表。
- 为空时继承 `kafka.brokers`。

### `cdc.group_id`

- Kafka 消费组 ID，用于控制 offset 和多实例分摊。

### `cdc.consumer_name`

- 当前进程消费者名称前缀，用于日志和后续排障。

### `cdc.read_timeout_seconds`

- 单次 Kafka 读取超时时间，单位秒。

### `cdc.retry_backoff_seconds`

- 处理失败后的重试间隔，单位秒。

### `cdc.max_retry_times`

- 单条消息最大失败次数。

### `cdc.dead_letter_topic`

- 死信 Topic；为空时失败消息不提交。

### `cdc.topics`

- Debezium Topic 消费规则列表。
- `topic` 是 Kafka Topic 名称。
- `table` 是表路由键，格式为 `db.table`。

### `collector.redis.enabled`

- 是否启用 Redis Stream 载体。

### `collector.redis.stream`

- Redis Stream 业务名称，运行时会自动追加 `app:{app_id}:` 前缀，避免多站点共用 Redis 时串流。

### `collector.redis.group`

- Redis Stream 消费组。

### `collector.redis.consumer`

- 消费者名前缀；程序会自动追加实例 ID，避免多实例消费者名冲突。

### `collector.redis.max_len`

- Redis Stream 最大长度近似值；`<=0` 表示不裁剪。

### `collector.redis.count`

- 单次读取或认领 Redis Stream 消息数量。

### `collector.db.runner_batch_size`

- DB worker 单轮领取 outbox 事件上限，也是投送给 Processor 的批量上限。

### `collector.db.runner_interval_seconds`

- DB worker 轮询间隔，单位秒。

### `collector.db.running_lease_seconds`

- running 任务租约时间，单位秒。
- 任务超过租约仍未完成时，会被回收为 retry，避免 Worker 崩溃后永久卡住。

### `collector.db.max_retry_times`

- 最大失败重试次数。
- 达到该次数后任务进入死信，等待人工重试或排障。

## 通用归档配置 `archive`

这组配置控制热表到历史表的归档节奏和保留边界。

### `archive.enabled`

- 是否启用通用归档模块。

### `archive.safe_delay_minutes`

- 安全归档延迟，单位分钟。
- 归档上界会同时受 `archive_jobs[].archive_delay_days` 和该安全延迟约束，避免归档仍可能被业务修改的近实时数据。

### `archive.lock_ttl_seconds`

- 归档规划、watermark 推进和历史表清理的短锁 TTL，单位秒。

### `archive.lease_ttl_seconds`

- 单个归档区间被 worker 领取后的租约 TTL，单位秒。
- worker 异常退出后，租约过期的区间会被其它 worker 重新领取。

### `archive.max_concurrent_jobs`

- 单次归档工作流内不同归档目标的并发数。
- 配置会被代码硬上限保护，避免多个大表归档同时打满同一套 MySQL I/O。

### `archive.default_batch_size`

- 默认单批搬迁条数。
- 代码会限制到安全上限，避免误配置生成过大的 `IN` 条件和事务。
- 每个归档热表需要由 DBA 确认存在 `(time_column, primary_key)` 复合索引；`admin_log` 可参考
  `internal/database/assets/archive_admin_log_index.sql`，避免大表按时间窗口扫描时额外排序。

### `archive.default_delete_batch_size`

- 默认单批删除条数。
- 删除只按已经复制并校验成功的主键集合执行，避免对热表做无界条件删除。

### `archive.batch_delay_milliseconds`

- 每个归档批次提交后的保护性等待时间，单位毫秒。
- 大量历史积压时用于主动让出 MySQL I/O，降低连续 `INSERT ... SELECT` 和 `DELETE` 对主库的冲击；配置过大时会被代码硬上限保护。
- 生产建议初始值为 `200`，表示每个已提交批次后等待 200ms；在当前 `admin_log` 单批归档 `3000`、单批删除 `1000`、2 个 worker
  的配置下，能让 MySQL 有短暂 I/O 和锁释放窗口，同时不会明显拉长小时级归档任务。
- 调优建议：MySQL 压力平稳但历史积压明显时可用 `100~200`；出现写入 RT 升高、锁等待、IOWait 或从库延迟时临时调到 `300~500`
  ；只有低峰大回补且业务压力很高时才临时使用 `800~1000`。不建议长期超过 `1000`，代码硬上限为 `5000ms`。
- 估算执行时间时按“批次数 × 等待时间”计算额外耗时；例如 200 个批次、`200ms` 等待约增加 40 秒，`500ms` 等待约增加 100 秒。

### `archive.default_max_history_tables`

- 每个归档任务默认保留的历史表数量。
- 超出后按最老历史表优先淘汰，单次只处理少量历史表，剩余表留给下一轮低峰任务继续清理。

### `archive_jobs`

- 单表归档策略列表，发布后由运行配置数据表 active release 承载。
- `runtime_config.source=file` 时继续读取 `config_files.runtime`；`database` 时 YAML 中的 `archive_jobs` 仅作为手动导入或草稿为空时的兼容种子，不参与后续合并。
- 归档工作流 `archive.run` 的 `targets` 支持三种写法：
    - `admin_log`：同一轮先归档再删除，适合单周期串行处理。
    - `admin_log#archive`：只规划和搬迁归档窗口，适合每 15 分钟或每小时追水位。
    - `admin_log#delete`：只删除已归档且达到删除边界的热表数据，适合与归档不同频率调度。

### `archive_jobs[].database`

- 热表和历史表所属数据库，默认填写 `main`。
- 如需使用扩展库，填写 `site_mysql.<name>` 中的 `<name>`。
- 归档控制表 `archive_segment`、`archive_watermark` 默认使用主库，除非归档组件显式覆盖。

### `archive_jobs[].time_column_type`

- 归档时间列类型，支持 `time`、`string`、`unix`；为空默认 `time`。
- `time` 由数据库驱动解析为 `time.Time`；`string` 按字符串时间处理；`unix` 按 Unix `int64` 处理。

### `archive_jobs[].time_column_format`

- 字符串时间列格式，仅 `time_column_type=string` 生效，使用 Go layout。
- 为空默认 `2006-01-02 15:04:05`；如列值为 `2026/05/28 13:27:00`，配置 `2006/01/02 15:04:05`。
- 字符串时间列必须使用可按字典序比较时间先后的格式；只有日期粒度的格式不会启用秒级归档/删除窗口。

### `archive_jobs[].time_column_unix_unit`

- Unix 整数时间单位，仅 `time_column_type=unix` 生效。
- 支持 `seconds`、`milliseconds`；为空默认 `seconds`。

### `archive_jobs[].archive_condition`

- 自定义归档过滤条件，留空表示当前时间窗口内全部数据都参与归档。
- 条件会同时作用于首次最小时间探测、归档游标读取和 `INSERT ... SELECT`
  复制事务；复制事务会再次叠加主键集合与时间半开窗口，避免跨窗口写入历史表。只能配置无占位符、无注释、无子查询、无写入动作的
  `WHERE` 谓词片段，例如 `status = 1 AND is_deleted = 0`。
- 调整该条件只影响尚未处理的归档窗口；已完成窗口如果需要重新纳入或排除数据，必须配合人工回退水位/区间或补偿任务。

### `archive_jobs[].delete_condition`

- 自定义热表清理附加条件，留空表示清理条件只继承 `archive_condition` 和时间窗口。
- 实际删除时会按 `archive_condition AND delete_condition`
  生效，并在读取待删主键后、事务删除前再次校验主键集合、当前删除窗口和归档区间，避免归档条件只覆盖部分行时误删未归档数据或跨窗口删除。
- 条件安全规则与 `archive_condition` 相同；如果变更条件导致旧窗口仍有热表残留，需要通过后续删除任务继续清理或执行人工补偿。

### `archive_jobs[].split_unit`

- 历史表物理拆分粒度，支持 `none/year/quarter/month/week/day/custom_days`。
- `none` 表示所有历史数据写入固定历史表；其它值按时间生成不同历史表。
- `split_unit` 控制历史表命名和建表数量，不再强制等同于归档或删除窗口。

### `archive_jobs[].hot_keep_days`

- 热表目标保留天数。
- `delete_delay_days <= 0` 时会默认使用该值，表示数据至少保留这么多天后才允许物理删除。

### `archive_jobs[].archive_delay_days`

- 归档从几天前开始推进。
- 例如 `2` 表示每个周期归档两天前对应时间窗；`<=0` 时默认等于 `hot_keep_days`。
- `archive_watermark.watermark_time` 只表示该排他上界之前、满足归档条件的数据已完整复制到历史表，不表示热表已经删除；热表删除边界由 `delete_delay_days` 和 `delete_window_seconds` 独立控制。

### `archive_jobs[].archive_window_seconds`

- 单个归档窗口秒数。
- `900` 表示按 15 分钟窗口归档，`3600` 表示按 1 小时窗口归档；`<=0` 时按 `split_unit` 规划归档区间。
- 配置窗口后仍会受 `split_unit` 自然边界保护，避免按天、按月历史表出现跨天或跨月数据。
- 首次无水位线时会从最早可归档数据所在窗口开始追水位，避免因月中首条数据而从月初空窗口慢慢推进。

### `archive_jobs[].archive_window_mode`

- 归档窗口推进模式，支持 `auto`、`fixed`，为空默认 `auto`。
- `fixed` 严格按 `archive_max_windows_per_run` 和 `delete_max_windows_per_run` 推进，适合要求每轮固定窗口数的低扰动场景。
- `auto` 先按 `archive_max_windows_per_run` 推进基础窗口；若上一窗口行数和耗时都很轻，会继续追赶到 `archive_auto_max_windows`，用于空窗口或稀疏数据快速追平水位。
- 删除阶段也复用 auto 轻量阈值；旧空窗口或稀疏窗口会继续追赶，遇到重删除窗口或未推进窗口后停止本轮追赶。

### `archive_jobs[].archive_max_windows_per_run`

- 单次运行最多归档窗口数。
- 窗口模式默认 `1`；在 `fixed` 模式中这是严格上限，在 `auto` 模式中这是基础窗口数。

### `archive_jobs[].archive_auto_max_windows`

- `auto` 模式单次最多追赶窗口数。
- 默认 `200`，代码硬上限为 `1000`；该值不会小于归档或删除基础窗口数，删除阶段也受该值限制。

### `archive_jobs[].archive_auto_light_rows`

- `auto` 模式判断轻量窗口的行数阈值。
- 默认 `20000`，代码会按单批安全上限裁剪；窗口归档/删除行数大于该值时，本轮不再继续额外追赶。
- 删除阶段还会受 `delete_batch_size` 限制，避免连续清理重窗口。

### `archive_jobs[].archive_auto_light_ms`

- `auto` 模式判断轻量窗口的耗时阈值，单位毫秒。
- 默认 `3000`，代码硬上限为 `30000`；归档或删除窗口耗时大于该值时，本轮不再继续额外追赶。

### `archive_jobs[].delete_disabled`

- 是否禁用热表删除。
- `true` 表示只归档不删除，适合上线初期观测历史表完整性。

### `archive_jobs[].delete_delay_days`

- 删除从几天前开始推进。
- 默认等于 `hot_keep_days`，可独立设置为大于 `archive_delay_days`，实现“先归档两天前数据，热表保留 32 天后再删”。

### `archive_jobs[].delete_window_seconds`

- 单次删除窗口秒数。
- 删除节点会在已归档区间内按该窗口逐片删除，即使历史表按月分表，也不会一次删除整月热表数据。

### `archive_jobs[].delete_max_windows_per_run`

- 单次运行最多删除窗口数。
- 删除窗口大于归档窗口且未显式配置时，程序会按二者比例补齐默认值，例如 1 小时删除窗口对应 4 个 15 分钟归档窗口。
- `archive_window_mode=auto` 时，该值是删除基础窗口数；旧空窗口或稀疏窗口可继续追赶到 `archive_auto_max_windows`，但超过轻量阈值、`delete_batch_size` 或耗时阈值后会停止本轮追赶。

### `archive_jobs[].history_table_name_rule`

- 历史表命名模板，留空时按 `history_table_prefix + split_unit` 使用内置命名。
- 支持 `{prefix}`、`{table}`、`{yyyy}`、`{yyyymm}`、`{yyyymmdd}`、`{isoweek}`、`{quarter}`、`{range_start}`、`{range_end}` 占位符。
- 渲染结果必须符合 MySQL 标识符规则，只允许字母、数字和下划线，避免动态表名注入。

## 用户标签配置 `user_tag`

这组配置控制用户标签工作流的默认运行参数。

### `user_tag.enabled`

- 是否启用用户标签模块。
- 启动期注册开关；实例以 `false` 启动时不会注册 `user_tag` handler 和 workflow，热加载改成 `true` 后需要重启实例才能生效。

### `user_tag.event_hook_enabled`

- 是否启用标签得失事件 hook 派发。
- 未注册 hook 时保持关闭；开启后 outbox 中已有事件但无 hook 会进入重试状态。

### `user_tag.default_shard_total`

- 默认工作流任务分片总数。
- 用于全量、增量、定向任务的分片并发计算；与物理分表数量独立，小量数据默认从 1 开始，按吞吐压力再扩。

### `user_tag.default_batch_size`

- 默认单批处理数量。
- 主要用于运行期 UID 和通用阶段游标处理。

### `user_tag.default_worker_count`

- 节点内部 worker 默认值。
- 当前骨架只用于参数校验和后续扩展预留。

### `user_tag.runtime_shard_total`

- 运行期 UID 索引分片数量。
- 默认 1000，`shard_no` 固定为 `uid % 1000`，便于按分片索引查询。

### `user_tag.result_shard_total`

- 用户标签结果物理分表数量。
- 默认 1，支持按 10、100、1000 等口径逐步扩容；运行时按需创建 `user_tag_%d`、`user_tag_%d_tmp`。

### `user_tag.diff_batch_size`

- 标签差异解析批次大小。
- 后续接入 `resolve_changes` 时用于控制单批差异计算规模。

### `user_tag.event_batch_size`

- 事件 outbox 派发批次大小。
- `dispatch_hooks` 和 `user_tag.event_outbox.retry_scan` 使用该值控制单轮派发规模；异常扫描会覆盖所有 outbox 分片，不只处理 0 号分片。

## 周期任务配置 `task_periodic`

周期任务大列表不再写入主配置 `task.periodic`。默认数据库模式下，通过后台“任务运维 / 运行配置”维护草稿并发布 active release。

`runtime_config.source=file` 时，仍可从 `config_files.runtime` 指向文件读取顶层 `task_periodic`；`database` 时 YAML 中的 `task_periodic` 仅作为手动导入或草稿为空时的兼容种子。

### `task_periodic[].name`

- 周期任务名称。

### `task_periodic[].enabled`

- 是否启用该周期任务。
- 未配置时默认启用；只有显式配置 `enabled: false` 才会关闭。

### `task_periodic[].cron`

- cron 表达式，支持 5 段分钟级表达式、6 段秒级表达式和 `@every` 描述符。
- 5 段表达式示例：`*/5 * * * *`，表示每 5 分钟触发一次。
- 6 段秒级表达式示例：`*/10 * * * * *`，第一段是秒，表示每 10 秒触发一次。
- `cron` 和 `every_seconds` 只能配置一个；如果只是固定秒级间隔，优先使用 `every_seconds`，可读性更高。

### `task_periodic[].every_seconds`

- 固定秒级间隔，配置后会转换成 `@every Ns` 交给调度器执行。
- 最小值为 `5`，低于该值会跳过当前周期任务，避免误配置造成队列压力。
- 示例：`every_seconds: 10` 表示每 10 秒触发一次；不要在该字段填写 `*/10 * * * * *`，这种 6 段表达式应填写到 `cron`。

### `task_periodic[].workflow`

- 触发的工作流名称。

### `task_periodic[].queue`

- 投递队列名。

### `task_periodic[].targets`

- 工作流目标列表。
- 含义由具体业务定义。

### `task_periodic[].shard_total`

- 任务分片总数。

### `task_periodic[].gray_percent`

- 工作流节点声明支持灰度时，按 `workflow_id + node + shard_index` 稳定哈希筛选需要执行的分片，范围 `0~100`。
- 未声明支持灰度的工作流节点会执行全部分片，该字段不会生效；默认运行配置中不需要为 order、归档、用户标签任务填写。

### `task_periodic[].retry`

- 覆盖默认重试次数。

### `task_periodic[].timeout_seconds`

- 覆盖工作流节点超时时间，单位秒。
- 默认运行配置不填写该字段，优先使用各工作流节点在代码中定义的业务超时；只有确实要统一覆盖所有节点超时时才配置。

### `task_periodic[].unique_key`

- 周期任务唯一键。

### `task_periodic[].unique_ttl_seconds`

- 周期任务唯一键 TTL，单位秒。

## 运行配置来源 `runtime_config`

`runtime_config` 只控制会随业务增长的大列表运行配置来源。基础设施配置仍从启动 YAML 读取。

### `runtime_config.source`

- 支持 `file`、`database`；省略时默认 `file`，当前样例显式配置为 `database`。
- `file`：继续从 `config_files.runtime` 指向的 YAML 读取 `task_periodic` 和 `archive_jobs`。
- `database`：启动时读取 `runtime_config_state` 指向的 active release 快照；没有已发布版本时优先发布迁移种下的草稿表，草稿为空再导入当前文件大列表并发布初始版本。
- 后续变更必须通过后台“运行配置”页面保存草稿并发布，不再人工修改运行期文件的大列表段。

### `runtime_config.poll_interval_seconds`

- DB 模式下实例轻量轮询 active 版本的间隔秒数，默认 `30`，最小 `5`。
- 轮询只读取 table-cache 的 `runtime_config:state`；版本未变化时不读取发布快照和明细表。

## 外部告警配置 `alert`

这组配置用于把后台任务终态失败、周期任务配置异常和任务系统运行异常等需要人工介入的问题推送到外部群告警。当前支持 Lark 群机器人。

### `alert.lark.enabled`

- 是否启用 Lark 群机器人告警。
- 启用后，异步任务、周期触发任务和工作流节点进入终态失败时会发送 Lark 卡片告警；中间重试失败不会发送。
- 周期任务配置异常会在启动期校验和调度同步时发送 Lark 卡片告警；同一条异常配置默认按 5 分钟窗口限频推送，窗口内重复触发会计入下一次告警的触发次数，避免同步循环重复刷屏。
- 周期调度同步、cron 注册、leader 校验、队列巡检、周期入队、归档任务配置异常和 archived 过期清理等任务系统运行异常也会发送 Lark 卡片告警；同一异常指纹默认按 5 分钟窗口合并，队列积压超限这类主动保护性跳过不会按 P1 推送。
- Collector 后台消费、批处理收集器、缓存异步刷新投递、Excel 异步导出、配置热加载和服务启动/平滑停止这类不一定会直接返回给操作者的后台失败，也会复用同一条运行异常告警链路；事件 ID、jobId、cache key 等高基数字段只进入错误摘要，不进入告警指纹，避免故障时刷屏。
- `task_periodic[].workflow=task_report.daily_summary` 会在每天 10 点发送周期任务与工作流运行日报卡片，默认统计昨日 10 点到今日 10 点来源为 `periodic` 的 completed/archived 任务、工作流实例、失败明细、慢任务和当前队列状态；该能力不需要新增 DDL，依赖 active release 中的周期任务配置和当前进程已注册的 `task_report` 插件。
- 日报统计依赖 Asynq completed/archived 任务保留数据；生产建议 `task.completed_retention_seconds` 大于 `86400`，推荐 `90000` 秒以上，否则成功任务可能在日报统计前被清理，日报会给出数据完整性提示。

### `alert.lark.webhook_url`

- Lark 群机器人 webhook URL，仅支持 `http`/`https`。
- 为空时读取 `alert.lark.webhook_url_ref` 指向的文件。

### `alert.lark.webhook_url_ref`

- Lark 群机器人 webhook URL 文件路径。

### `alert.lark.secret`

- Lark 群机器人签名密钥。
- 为空时读取 `alert.lark.secret_ref` 指向的文件。

### `alert.lark.secret_ref`

- Lark 群机器人签名密钥文件路径。

### `alert.lark.timeout_seconds`

- Lark HTTP 请求超时时间，单位秒。
- 默认 `5`，最大 `30`。

### `alert.lark.at_all`

- 是否在告警中 @所有人。
- 生产建议谨慎开启，避免频繁打扰。

### `alert.lark.max_error_bytes`

- 错误摘要最大字节数。
- 默认 `800`，超出后按 UTF-8 安全截断。

### 本地真实发送验证

本地可通过受环境变量保护的 live smoke test 验证真实 Lark 群机器人链路：

```bash
ADMIN_LARK_LIVE=1 \
ADMIN_LARK_WEBHOOK_URL='https://open.feishu.cn/open-apis/bot/v2/hook/xxx' \
ADMIN_LARK_SECRET='可选签名密钥' \
go test ./internal/bootstrap -run TestTaskFailureLarkAlertLive -v
```

- 未设置 `ADMIN_LARK_LIVE=1` 时测试会跳过，避免日常测试误发群消息。
- `ADMIN_LARK_AT_ALL=true` 时会在测试消息中 @所有人，默认不启用。

## 推荐阅读顺序

如果你第一次接手 `admin`，建议按下面顺序阅读：

1. 先看 `etc/config.sample.yaml`，了解所有字段名和示例结构。
2. 再看 `etc/config.yaml`，确认当前环境的实际取值。
3. 如果是用户标签开发，再看《用户标签实现与调度说明》文档，理解配置是如何参与工作流执行的：`功能模块/用户标签/用户标签实现与调度说明.md`。
