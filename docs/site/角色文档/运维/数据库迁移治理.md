# 数据库迁移治理

## 先看重点

- 生产已有库上线只走：`migrate-status` → `migrate-dry-run` → 备份 → `migrate-up` → `migrate-status`。
- 空库初始化或已初始化库补齐基线登记才使用 `migrate-bootstrap`；非空库执行前必须核对目标库、表结构和关键种子。
- `migrate-bootstrap` 会放开 bootstrap-only 拦截并登记全部未登记版本；它不会修复字段、索引或历史种子漂移。
- 出现未知 `blocked`、发布后 checksum drift、疑似错库或无法确认备份时，停止发布。
- DROP、TRUNCATE、DELETE、字段删除或批量修复必须单独评审，不走普通快速上线。
- 文档权限只允许集中维护 `document_permission_seed.sql`，不要为新增单篇文档拆出 `document_permission_*.sql` 增量迁移。
- SQL seed 显式保留自增主键 `id` 时，新增行按 `id` 递增放置，不按业务相似性插入到中间。

## 常用执行路径

| 场景 | 执行步骤 | 关键判断 |
| --- | --- | --- |
| 查看当前库状态 | `make migrate-status MIGRATE_CONFIG=./etc/config.yaml` | 只读，可快速发现 pending、blocked 和 checksum drift |
| 已有库上线普通增量 | `migrate-status` → `migrate-dry-run` → 备份 → `migrate-up` → `migrate-status` | 生产标准路径 |
| 新环境空库初始化 | 确认空库 → 备份/快照 → `migrate-bootstrap` → `migrate-status` | 新库首次建表和写入基线种子 |
| 已初始化库缺少基线登记 | `migrate-status` → `migrate-dry-run` → 核对表结构和关键种子 → 备份 → `migrate-bootstrap` → `migrate-status` | 只补登记和缺失种子，不修复结构漂移 |
| dry-run 出现 `blocked bootstrap-only` | 先确认库状态 | 确认是基线补登记才 bootstrap；库状态不清时停止 |
| status 出现 checksum drift | 暂停发布 | 未发布阶段可收口；正式发布后只能新增正向迁移 |

## 命令入口

所有命令在 `admin` 目录执行，`MIGRATE_CONFIG` 必须指向目标环境的实际配置文件：

```bash
cd admin
```

| 命令 | 实际动作 | 是否写库 | 适用场景 | 安全边界 |
| --- | --- | --- | --- | --- |
| `make build-tools` | 构建 `bin/admin-migrate` | 否 | 发布打包前确认迁移二进制可构建 | 不连接数据库 |
| `make migrate-status MIGRATE_CONFIG=./etc/config.yaml` | 读取 `schema_migrations` 并输出状态 | 否 | 日常巡检、发布前确认状态 | 只读；版本表不存在时按未执行迁移展示 |
| `make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml` | 输出执行计划和拦截原因 | 否 | 每次上线前必须执行 | 只读；不能替代 SQL 评审和备份 |
| `make migrate-up MIGRATE_CONFIG=./etc/config.yaml` | 执行未登记且未被拦截的迁移并登记版本 | 是 | 已有库普通增量迁移 | 执行前必须完成 dry-run、备份和计划确认 |
| `make migrate-bootstrap MIGRATE_CONFIG=./etc/config.yaml` | 以 `-allow-bootstrap -allow-destructive` 执行全部未登记迁移 | 是，高风险 | 空库初始化；已初始化库补齐基线登记或缺失种子 | 非空库执行前必须核对表结构和关键种子 |
| `make integration-test` | 启动本地集成 MySQL 并运行迁移集成测试 | 否，不碰业务库 | 修改迁移执行器、拦截规则或资产校验后验证 | 依赖 Docker；结束后清理本地容器和卷 |

等价二进制命令：

```bash
bin/admin-migrate -f ./etc/config.yaml -action=status
bin/admin-migrate -f ./etc/config.yaml -action=dry-run
bin/admin-migrate -f ./etc/config.yaml -action=up
bin/admin-migrate -f ./etc/config.yaml -action=up -allow-bootstrap -allow-destructive
bin/admin-migrate -version
```

## 已有库上线步骤

1. 确认配置文件指向目标库，尤其是 MySQL 地址、库名、用户名和生产环境标识。
2. 构建迁移工具：

```bash
make build-tools
```

3. 查看当前状态：

```bash
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
```

4. 预览上线计划：

```bash
make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml
```

5. 只有 dry-run 输出符合预期、没有未知 blocked、没有发布后 drift 时，才做数据库备份并执行：

```bash
make migrate-up MIGRATE_CONFIG=./etc/config.yaml
```

6. 执行后复核：

```bash
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
```

7. 核对 `schema_migrations` 中的 `version/name/asset/checksum/applied_at`，并按迁移内容抽查表结构、关键种子数据和后台页面。

文档权限迁移执行后，工具会自动刷新权限树、角色权限、管理员权限和路由候选权限缓存。若迁移 SQL 已执行但 Redis 刷新失败，修复 Redis 连接后重新执行 `migrate-up`，工具会基于已登记版本再次刷新权限缓存。

新增单篇 Markdown 文档权限时，只更新 `internal/database/assets/document_permission_seed.sql` 与 `internal/routealias/docs.go`，不要新增独立 `document_permission_*.sql` 或 `sync_document_permission_*` 迁移。已有库如果对应迁移版本已登记，发布单中手动执行新增的 `INSERT IGNORE` 权限行，并在执行后刷新权限相关缓存；纯手工 SQL 不会触发迁移工具的自动刷新。

维护带显式 `id` 的 seed SQL 时，新增行必须按自增主键递增顺序落位；即使业务语义相近，也不要把高 id 行插入到相似模块附近。`go test ./internal/database` 会检查同一资产同一表内的 seed id 顺序。

## 空库初始化与基线补登记

执行 `migrate-bootstrap` 前至少确认：

```sql
SELECT DATABASE();
SHOW TABLES;
SHOW TABLES LIKE 'schema_migrations';
```

空库初始化时确认目标库为空、配置文件无误、备份或快照已准备后执行：

```bash
make migrate-bootstrap MIGRATE_CONFIG=./etc/config.yaml
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
```

非空库执行前还要抽查 `SHOW CREATE TABLE` 和关键种子数据。当前基线资产以 `CREATE TABLE IF NOT EXISTS` 和 `INSERT IGNORE` 为主，重复执行不会覆盖已有运营数据；但执行成功会登记对应版本，后续不会再执行同一迁移。核对不一致时停止执行，改用新的正向修复迁移。

## 适用范围

本文只说明 `admin` 仓库内置迁移工具的执行边界、命令选择和发布步骤。迁移入口是 `cmd/migrate`，迁移清单来自 `internal/database.defaultMigrationSpecs`，SQL 资产位于 `internal/database/assets/`，执行结果登记到 `schema_migrations`。

迁移只管理 `admin` 自己的库表。涉及 API 仓库、业务库分表、数据回填或历史数据修复时，必须单独制定迁移和回填方案，不能直接套用本文命令。

## 版本表与状态

`schema_migrations` 记录已成功执行的迁移版本，字段包含 `version`、`name`、`asset`、`checksum` 和 `applied_at`。迁移工具会按 `version` 升序处理迁移，已登记版本不会重复执行。

| 状态 | 含义 | 处理方式 |
| --- | --- | --- |
| `applied` | 版本已登记，本次不会再执行 SQL | 核对 checksum；发布后不允许历史迁移 drift |
| `pending` | 版本未登记，且当前命令只预览计划 | 先评审 SQL、备份，再执行 `migrate-up` |
| `executed` | 本轮已执行 SQL 并登记版本 | 执行后复核表结构、数据和 `schema_migrations` |
| `blocked` | 被 bootstrap-only 或 destructive 安全策略拦截 | 不要强行执行，先确认库状态和命令选择 |

`checksum drift ignored for unreleased migration` 表示已登记版本的 SQL checksum 与当前资产不一致。当前版本尚未正式发布时允许继续收口，但发布前必须清零 drift；正式发布后只能新增更高版本正向迁移，不修改历史 SQL。

## 失败处理

- `status` / `dry-run` 失败：通常是配置、网络、账号权限或版本表读取异常。先修连接和权限，不执行 `up`。
- `up` 被 `blocked` 拦截：不要加 `allow` 参数绕过。先确认迁移类型和目标库是否正确。
- `up` 执行 SQL 失败：停止发布，保留输出中的 `version/name/asset/statement`，核对已执行到哪条 SQL。
- `bootstrap` 在非空库失败：不要手工补写 `schema_migrations`。先按失败资产和语句处理冲突，再重新执行迁移命令。
- MySQL DDL 可能隐式提交，不能完全依赖事务自动回滚。恢复方式以数据库备份、快照回滚或新增正向补偿迁移为准。
- 本工具没有 `down` 命令；已发布迁移需要回退时，必须新增更高版本的正向修复迁移或按备份恢复。

## 发布要求

上线前必须完成：

1. `make migrate-dry-run MIGRATE_CONFIG=...` 确认计划。
2. 核对 `checksum`，正式发布后历史版本不得被修改。
3. 对破坏性 SQL 单独评审，并准备回滚或补偿方案。
4. 执行后核对 `schema_migrations` 登记版本。
5. 迁移执行器或 SQL 资产发生变化时，执行 `go test ./internal/database`；涉及真实 MySQL 行为时补跑 `make integration-test`。

迁移执行器会按分号拆分 SQL，并跳过文件内 `BEGIN`、`COMMIT`、`ROLLBACK`、`START TRANSACTION`，不要求 MySQL DSN 开启 `multiStatements=true`。
