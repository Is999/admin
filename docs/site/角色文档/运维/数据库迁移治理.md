# 数据库迁移治理

## 命令入口

迁移工具随主程序一起构建：

```bash
make build-tools
```

常用命令：

```bash
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml
make migrate-up MIGRATE_CONFIG=./etc/config.yaml
make migrate-bootstrap MIGRATE_CONFIG=./etc/config.yaml
make integration-test
```

`migrate-status` 和 `migrate-dry-run` 只读取 `schema_migrations` 并输出计划；`migrate-up` 会创建版本表、执行未登记迁移并写入 `version/name/asset/checksum/applied_at`。执行到文档权限迁移时，迁移工具会自动刷新权限树、角色权限、管理员权限和路由候选权限缓存。

当前版本尚未发布，admin 迁移工具默认允许已登记迁移资产继续调整，并在状态输出中用 `checksum drift ignored for unreleased migration` 标注。正式发布前必须冻结历史迁移或启用严格 checksum 校验，发布后只能追加更高版本正向迁移。

## 基线 SQL

当前后台历史 SQL 已登记为 bootstrap-only/destructive：

- 默认 `migrate-up` 会拦截这些迁移，不会在线执行。
- 只允许空库初始化时，在确认目标库和备份后执行 `make migrate-bootstrap MIGRATE_CONFIG=...`。
- 已有生产库禁止直接执行历史基线 SQL；需要变更时新增更高版本正向迁移。

## 发布要求

上线前必须完成：

1. `make migrate-dry-run MIGRATE_CONFIG=...` 确认计划。
2. 核对 `checksum`；正式发布后历史版本不得被修改。
3. 对破坏性 SQL 单独评审并准备回滚或补偿方案。
4. 执行后核对 `schema_migrations` 登记版本。

迁移执行器会按分号拆分 SQL 并跳过文件内 `BEGIN/COMMIT`，不要求 MySQL DSN 开启 `multiStatements=true`。

`make integration-test` 会使用 `deploy/integration/docker-compose.yml` 启动 MySQL，并在真实 MySQL 上校验基线默认拦截逻辑。
