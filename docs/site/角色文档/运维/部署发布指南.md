# 部署发布指南

## 目标

本文只保留 `admin` 上线、发布和首次管理员自举的必要流程。更细的接口字段见对应接口文档。

## 部署前检查

上线前确认：

1. 已执行 `make ci`，或至少完成 `gofmt`、`go test ./...`、`go build`、`go build ./cmd/admin ./cmd/migrate`。
2. 已通过 `make migrate-dry-run MIGRATE_CONFIG=...` 确认数据库迁移计划。
3. 当前仓库的 `internal/database.DefaultMigrations()` 与 `internal/database/assets/*.sql` 清单一致；bootstrap-only 基线 SQL 只通过 `make migrate-bootstrap` 显式执行。
4. `internal/handler/route_contract.go` 与实际路由注册保持一致，`go test ./internal/handler` 通过。
5. `etc/config.yaml` 已按环境从 `etc/config.sample.yaml` 生成。
6. `Mode=pro/prod/production` 环境已配置强随机 `app_id`、`app_key`、`jwt_secret`，Redis 未开启 `tls_insecure_skip_verify`。
7. `snowflake.redis.enabled=true` 已启用，且同一套 `admin/api` 部署使用同一 `snowflake.redis.scope`；同一业务唯一域必须使用相同 `NextID(namespace)` 和 `snowflake.redis.namespaces.<namespace>.node_count`，当前 `user` 使用 10 个 node_id。仅手工强管控场景允许改用全局唯一的 `snowflake.worker_id`；如启用 `snowflake.segment`，确认对应 namespace 的 `scope/step/prefetch_threshold` 已压测。
8. `user.route_shard_count` 仅使用 `1/2/4/8/16/32/64/128/256/512/1024`；提升前已完成联系方式密文和 HMAC、四张 `user_identity_*` 表回填、目标物理表创建，并计划统一重启所有会写 `user` 的 `admin/api` 实例。
9. API 仓库业务用户迁移已执行到 `202606220006 create_user_identity_oauth`，否则后台用户管理新增用户会因缺少登录身份索引表失败。
10. 如启用 `security.secret_key`，已配置当前 `app_id`、密钥版本和必要 AES/RSA 材料；生产灰度密钥必须配置 `gray_salt`。
11. `runtime_config.source=file` 时，`config_files.runtime` 指向的运行期配置文件存在，周期任务配置在顶层 `task_periodic`。
12. `runtime_config.source=database` 时，确认当前库已有 active release；首次无 active release 时，优先使用迁移种下的 `runtime_task_periodic/runtime_archive_job` 草稿表生成初始发布，草稿为空才回退当前运行期文件里的 `task_periodic/archive_jobs`。
13. API、Worker、Scheduler 共用同一套 `task.redis`。
14. 同一站点所有实例使用相同 `app_id`，不同站点使用不同 `app_id`。
15. `/api/live`、`/api/ready`、`/api/metrics` 在目标环境按内网或网关白名单可访问。

## 构建

```bash
make ci
make package VERSION=20260605
```

`make ci` 会执行格式检查、全量测试和二进制构建。没有 `make` 的环境可等价执行：

```bash
gofmt -w $(find . -name '*.go' -not -path './vendor/*')
go test ./...
go build -o bin/admin ./cmd/admin
go build -o bin/admin-migrate ./cmd/migrate
```

发布产物至少包含：

- `bin/admin`
- `bin/admin-migrate`
- `dist/admin-<version>.tar.gz`
- `etc/config.yaml`
- `etc/config.d/runtime.yaml`
- 必要的证书、密钥和运行期配置文件

仓库提供的部署模板：

- `deploy/docker/Dockerfile`
- `deploy/systemd/admin-control.service`
- `deploy/systemd/admin-worker.service`
- `deploy/integration/docker-compose.yml`

## 数据库版本治理

上线工具应先执行 dry-run，再执行 up：

```bash
make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml
make migrate-up MIGRATE_CONFIG=./etc/config.yaml
```

当前仓库把历史初始化 SQL 纳入 `DefaultMigrations()` 清单，记录 `version`、`name`、`asset`、`checksum`、`bootstrap_only` 和 `destructive`。运维约束：

- 空库初始化或已初始化库补齐基线登记时，在确认目标库、dry-run、核对表结构和备份后执行 `make migrate-bootstrap MIGRATE_CONFIG=...`。
- 已有生产库不要盲目执行 bootstrap；它只补登记和缺失种子，不修复字段、索引或历史种子漂移。
- 未发布阶段允许迁移资产继续收口，`migrate-status` 会标注 checksum drift；正式发布后只能新增正向迁移文件，不修改历史 SQL 内容。
- 文档权限只维护 `document_permission_seed.sql` 主清单，不为单篇文档新增碎片化迁移；已上线环境手动执行新增 `INSERT IGNORE` 后必须刷新权限相关缓存。
- 带显式自增主键 `id` 的 seed 行必须按 `id` 递增排列，不能按业务相似性插入到中间。
- 发布前用 `go test ./internal/database` 校验迁移清单、资产覆盖和 checksum，并确认不再存在 drift 标注。

详细规则见 [数据库迁移治理](角色文档/运维/数据库迁移治理.md)。

## 启动期硬校验

服务启动和热加载会在进入运行态前校验以下风险：

- `jwt_secret` 长度过短。
- 主 Redis 地址为空、连接池大小无效或 `redis.type` 不在 `single/cluster`。
- 雪花 Redis 租约参数不安全，Segment 号段参数非法，或手工 `snowflake.worker_id` 缺失、越界。
- `user.route_shard_count` 不在 `1/2/4/8/16/32/64/128/256/512/1024` 档位内。
- `task.redis` 只配置了 DB、密码、TLS 等非地址字段但未配置独立地址。
- Collector 启用但缺少 Kafka topic、broker 或 consumer group。
- Lark 告警开启但缺少 webhook。
- `security.secret_key` 缺少 `app_id`、版本号、AES/RSA 材料或灰度参数不自洽。
- 生产环境使用占位密钥、弱 `app_key`、空 `app_id` 或 Redis 跳过 TLS 证书校验。

## 运行模式

`-mode` 是位掩码：

| mode | 角色 | 说明 |
| --- | --- | --- |
| `1` | API | 提供 HTTP 接口，只投递任务，不消费任务 |
| `2` | Worker | 消费队列并执行任务 |
| `3` | API + Worker | 执行任务并暴露 HTTP 探针，适合需要服务健康检查的 Worker 部署 |
| `4` | Scheduler | 按周期配置投递任务，不执行任务 |
| `5` | API + Scheduler | 推荐控制面 |
| `6` | Worker + Scheduler | 调度并执行任务，不提供 HTTP 接口，通常只用于小环境或专用后台实例 |
| `7` | API + Worker + Scheduler | 本地或小环境一体化 |

启动命令：

```bash
./bin/admin -f ./etc/config.yaml -mode 5
./bin/admin -f ./etc/config.yaml -mode 2
./bin/admin -f ./etc/config.yaml -mode 3  # Worker 需要 HTTP 健康探针时使用
```

也可以在配置中写：

```yaml
run_mode: 5
```

命令行 `-mode` 优先级高于配置文件 `run_mode`。

## 推荐拓扑

### 本地或测试

```bash
./bin/admin -f ./etc/config.yaml -mode 7
```

### 生产推荐

- `cron-control`：`-mode 5`，提供 API 与周期调度。
- `cron-worker`：`-mode 2`，只执行队列任务，不暴露 HTTP。
- `cron-worker-probed`：`-mode 3`，执行队列任务并暴露 `/api/live`、`/api/ready`，适合 Kubernetes、负载均衡或进程托管平台需要 HTTP 健康探针的场景。

链路：

```text
后台 / 内部调用方
  -> cron-control(API + Scheduler)
  -> task.redis
  -> cron-worker(Worker 或 API + Worker)
```

说明：

- Scheduler 多副本会通过 Redis leader 锁保证同一时刻只有一个实例投递周期任务。
- Worker 可以按任务压力水平扩容。
- 如果运行平台不需要 HTTP 健康探针，`cron-worker` 使用 `-mode 2` 即可。
- 如果运行平台需要 HTTP 健康探针，`cron-worker` 使用 `-mode 3`，只把探针端口暴露给内网或平台健康检查，不对外开放后台入口。

## 运行配置数据库化迁移

本仓库只把 `task_periodic` 和 `archive_jobs` 这类大列表运行配置数据库化。`workflows.user_tag` 等单一小配置仍留在 YAML。

首次启用按下面顺序执行：

1. 空库初始化或基线补登记后确认 baseline DDL 已包含四张运行配置表、草稿明细种子、权限码和 MFA 场景；当前未上线阶段不保留单独运行配置表结构增量 SQL。
2. baseline 不写默认发布快照。生产启用前确认迁移已种下 `runtime_task_periodic/runtime_archive_job` 草稿明细；如草稿为空，再保留 `config_files.runtime` 中的 `task_periodic/archive_jobs` 作为兼容导入种子或先通过后台导入并发布。
3. 将启动 YAML 设置为 `runtime_config.source=database`。
4. 启动或重启 `cron-control`，首次无 active release 时会自动发布草稿表快照；草稿为空时才导入当前运行期文件并发布初始版本。
5. 打开后台“任务运维 / 运行配置”，确认 active release、发布历史和当前快照正确；后续只通过草稿和发布管理。
6. 再滚动重启 `cron-worker`，确认 `/api/ready` 和任务队列正常。

运行期变更只通过后台页面保存草稿并发布。不要人工直改明细表后等待自动生效；实例只轮询 table-cache 的 active 版本，版本未变化不会扫描明细表。

## 首次上线管理员自举

> 重点：首次上线如果无法登录后台，优先使用内网自举接口重置既有超级管理员账号。该接口只能内网访问，不创建新账号，不提升角色。

接口：

```http
POST /internal/auth/init-admin-bootstrap
```

最小调用：

```bash
BASE_URL='http://127.0.0.1:8888'

curl -sS -X POST "${BASE_URL}/internal/auth/init-admin-bootstrap" \
  -H 'Content-Type: application/json' \
  --data-raw '{"username":"admin","password":"Temp@1234"}'
```

成功后确认：

- `code = 1`
- `data.operation = "reset"`
- `data.needResetPassword = 1`
- `data.mfaStatus = 0`

随后立刻完成：

1. 使用临时密码登录后台。
2. 按页面要求修改密码。
3. 如启用 MFA，完成 MFA 绑定。
4. 确认菜单、权限码和个人中心信息正常。

详细字段见 [内网初始化管理员接口](接口文档/后台系统/内网初始化管理员接口.md)。

## 发布流程

1. 合并代码并完成测试。
2. 构建二进制和配置包。
3. 先发布 `cron-control`，确认 `/api/ready` 正常。
4. 发布 `cron-worker`。使用 `-mode 2` 时在任务队列页确认 Worker 在线；使用 `-mode 3` 时同时确认 `/api/live` 和 `/api/ready` 正常。
5. 打开任务队列页面，确认队列、Worker 和 Scheduler 状态正常。
6. 如变更周期任务，文件模式检查配置热加载状态；数据库模式检查“运行配置”页面 active 版本和任务队列 `data.scheduler.lastSyncStatus`。
7. 如首次上线，执行管理员自举并完成登录验证。

## 验证清单

发布后至少验证：

- `/api/live` 返回成功。
- `/api/ready` 中 MySQL、Redis、task_queue 状态正常。
- `/api/metrics` 可被 Prometheus 从内网抓取，公网不可直接访问。
- 后台可以登录，权限菜单正常。
- ID 生成指标 `admin_idgen_generated_total{biz_type,strategy,result}` 有基线数据；启用 Segment 时同时观察 `admin_idgen_segment_remaining{biz_type}`、`admin_idgen_segment_allocations_total{biz_type,result}` 和 `admin_idgen_snowflake_lease_events_total{biz_type,event}`。
- `GET /api/tasks/queues` 能看到队列、Worker 和 Scheduler。
- 手工触发一个低风险工作流后，任务列表可查到任务并能进入详情。
- 运行配置草稿发布后，配置热加载页面无持续失败。
- `runtime_config.source=database` 时，运行配置页面 active 版本、发布历史、当前快照与预期一致。
- Collector 启用时，检查 `docs/grafana/collector-dashboard.json` 对应吞吐、ready、failed 和 dead-letter 指标。
- Lark 告警启用时，确认后台任务终态失败、Collector/缓存刷新/Excel 导出/配置热加载/服务生命周期失败都复用 5 分钟运行异常限频，告警内含影响、错误摘要和处理建议。
- Prometheus 可加载 `docs/prometheus/admin-alerts.yml`，并按生产基线调整阈值。
- 数据库迁移改动可执行 `make integration-test`，在真实 MySQL 上校验默认拦截策略。
- 新增或调整 HTTP 接口时，确认 `go test ./internal/handler` 的路由契约测试通过。

## 回滚

回滚时按相反顺序处理：

1. 暂停发布流量或摘除异常实例。
2. 恢复上一版本二进制和配置。
3. 先恢复 `cron-control`，再恢复 `cron-worker`。
4. 检查队列积压和失败任务。
5. 如果本次包含在线数据库迁移，先按迁移单确认是否需要反向脚本或补偿任务。
6. 对失败任务只做确认后的人工重跑，避免重复执行未知 payload。

## 常见问题

### API 成功但任务没有执行

检查 Worker 是否启动、队列是否暂停、`task.redis` 是否一致。

### 周期任务没有投递

检查实例是否包含 Scheduler 能力、是否抢到 leader、`task_periodic` 是否启用且 cron 合法。

### 多站点任务串扰

检查 `app_id`。同站点必须一致，不同站点必须不同。

### 管理员无法登录

确认目标超级管理员账号已存在，然后通过内网自举接口恢复首次登录状态。
