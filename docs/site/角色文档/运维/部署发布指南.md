# 部署发布指南

## 目标

本文只保留 `admin-cron` 上线、发布和首次管理员自举的必要流程。更细的接口字段见对应接口文档。

## 部署前检查

上线前确认：

1. 已创建并导入 `internal/database` 下所需表结构。
2. `etc/config.yaml` 已按环境从 `etc/config.sample.yaml` 生成。
3. `config_files.runtime` 指向的运行期配置文件存在，周期任务配置在顶层 `task_periodic`。
4. API、Worker、Scheduler 共用同一套 `task.redis`。
5. 同一站点所有实例使用相同 `app_id`，不同站点使用不同 `app_id`。
6. `/api/live` 和 `/api/ready` 在目标环境可访问。

## 构建

```bash
go test ./...
go build -o bin/admin-cron .
```

发布产物至少包含：

- `bin/admin-cron`
- `etc/config.yaml`
- `etc/config.d/runtime.yaml`
- 必要的证书、密钥和运行期配置文件

## 运行模式

`-mode` 是位掩码：

| mode | 角色 | 说明 |
| --- | --- | --- |
| `1` | API | 提供 HTTP 接口，只投递任务，不消费任务 |
| `2` | Worker | 消费队列并执行任务 |
| `3` | API + Worker | 执行任务并暴露 HTTP 探针，适合需要服务健康检查的 Worker 部署 |
| `4` | Scheduler | 按周期配置投递任务，不执行任务 |
| `5` | API + Scheduler | 推荐控制面 |
| `6` | Worker + Scheduler | 调度并执行任务，不提供 HTTP 接口，通常只用于小环境或专用后台实例 |
| `7` | API + Worker + Scheduler | 本地或小环境一体化 |

启动命令：

```bash
./bin/admin-cron -f ./etc/config.yaml -mode 5
./bin/admin-cron -f ./etc/config.yaml -mode 2
./bin/admin-cron -f ./etc/config.yaml -mode 3  # Worker 需要 HTTP 健康探针时使用
```

也可以在配置中写：

```yaml
run_mode: 5
```

命令行 `-mode` 优先级高于配置文件 `run_mode`。

## 推荐拓扑

### 本地或测试

```bash
./bin/admin-cron -f ./etc/config.yaml -mode 7
```

### 生产推荐

- `cron-control`：`-mode 5`，提供 API 与周期调度。
- `cron-worker`：`-mode 2`，只执行队列任务，不暴露 HTTP。
- `cron-worker-probed`：`-mode 3`，执行队列任务并暴露 `/api/live`、`/api/ready`，适合 Kubernetes、负载均衡或进程托管平台需要 HTTP 健康探针的场景。

链路：

```text
后台 / 内部调用方
  -> cron-control(API + Scheduler)
  -> task.redis
  -> cron-worker(Worker 或 API + Worker)
```

说明：

- Scheduler 多副本会通过 Redis leader 锁保证同一时刻只有一个实例投递周期任务。
- Worker 可以按任务压力水平扩容。
- 如果运行平台不需要 HTTP 健康探针，`cron-worker` 使用 `-mode 2` 即可。
- 如果运行平台需要 HTTP 健康探针，`cron-worker` 使用 `-mode 3`，只把探针端口暴露给内网或平台健康检查，不对外开放后台入口。

## 首次上线管理员自举

> 重点：首次上线如果无法登录后台，优先使用内网自举接口重置既有超级管理员账号。该接口只能内网访问，不创建新账号，不提升角色。

接口：

```http
POST /internal/auth/init-admin-bootstrap
```

最小调用：

```bash
BASE_URL='http://127.0.0.1:8888'

curl -sS -X POST "${BASE_URL}/internal/auth/init-admin-bootstrap" \
  -H 'Content-Type: application/json' \
  --data-raw '{"username":"admin","password":"Temp@1234"}'
```

成功后确认：

- `code = 1`
- `data.operation = "reset"`
- `data.needResetPassword = 1`
- `data.mfaStatus = 0`

随后立刻完成：

1. 使用临时密码登录后台。
2. 按页面要求修改密码。
3. 如启用 MFA，完成 MFA 绑定。
4. 确认菜单、权限码和个人中心信息正常。

详细字段见 [内网初始化管理员接口](../../接口文档/后台系统/内网初始化管理员接口.md)。

## 发布流程

1. 合并代码并完成测试。
2. 构建二进制和配置包。
3. 先发布 `cron-control`，确认 `/api/ready` 正常。
4. 发布 `cron-worker`。使用 `-mode 2` 时在任务队列页确认 Worker 在线；使用 `-mode 3` 时同时确认 `/api/live` 和 `/api/ready` 正常。
5. 打开任务队列页面，确认队列、Worker 和 Scheduler 状态正常。
6. 如变更周期任务，检查 `data.scheduler.lastSyncStatus` 和 `lastEnqueueAt`。
7. 如首次上线，执行管理员自举并完成登录验证。

## 验证清单

发布后至少验证：

- `/api/live` 返回成功。
- `/api/ready` 中 MySQL、Redis、task_queue 状态正常。
- 后台可以登录，权限菜单正常。
- `GET /api/tasks/queues` 能看到队列、Worker 和 Scheduler。
- 手工触发一个低风险工作流后，任务列表可查到任务并能进入详情。
- `task_periodic` 配置变更后，配置热加载页面无持续失败。

## 回滚

回滚时按相反顺序处理：

1. 暂停发布流量或摘除异常实例。
2. 恢复上一版本二进制和配置。
3. 先恢复 `cron-control`，再恢复 `cron-worker`。
4. 检查队列积压和失败任务。
5. 对失败任务只做确认后的人工重跑，避免重复执行未知 payload。

## 常见问题

### API 成功但任务没有执行

检查 Worker 是否启动、队列是否暂停、`task.redis` 是否一致。

### 周期任务没有投递

检查实例是否包含 Scheduler 能力、是否抢到 leader、`task_periodic` 是否启用且 cron 合法。

### 多站点任务串扰

检查 `app_id`。同站点必须一致，不同站点必须不同。

### 管理员无法登录

确认目标超级管理员账号已存在，然后通过内网自举接口恢复首次登录状态。
