# 部署发布指南

本文只说明 `admin` 的构建、配置、迁移、启动、验证和回滚。字段语义、任务实现和接口契约见文末专项文档。

## 部署前检查

| 检查项 | 要求 |
| --- | --- |
| 构建 | `make ci` 通过，发布版本已执行 `make package VERSION=<version>` |
| 配置 | 从 `etc/config.sample.yaml` 生成环境配置；生产密钥不使用示例值，Redis 不跳过 TLS 证书校验 |
| 数据库 | 已备份并确认 `migrate-status` 和 `migrate-dry-run`；Admin/API 依赖的库表版本均到位 |
| 共享边界 | 同站点实例使用相同 `app_id`、`task.redis` 和 Snowflake scope；不同站点的 `app_id` 不同 |
| 外部依赖 | MySQL、Redis、Task Redis、Kafka 及已启用的 S3、ClamAV、Lark 均可访问 |
| 运行配置 | 数据库模式已有 active release；首次启动时至少有迁移草稿或可导入的运行期文件 |
| 功能开关 | 未交付的 CDC、用户标签和测试场景保持关闭；Collector 在生产环境保持启用 |
| 可选资源 | 启用 IP 归属地时准备只读 XDB；启用 ClamAV 时确认客户端、daemon 和病毒库可用 |
| 网络 | `/api/live`、`/api/ready`、`/api/metrics` 只对内网、网关或监控系统开放 |

配置取值和重启边界见[配置字段说明](角色文档/后端开发/配置字段说明.md)。

## 构建与产物

```bash
make ci
make package VERSION=20260605
```

发布时准备：

- `dist/admin-<version>.tar.gz`
- 环境配置、运行配置和密钥文件
- 启用 `ip_region` 时的 XDB 只读数据卷
- 需要的证书和外部服务配置

部署模板位于 `deploy/docker`、`deploy/systemd` 和 `deploy/integration/docker-compose.yml`。

### 可选本地 IP 归属地数据

当前固定使用 ip2region `v3.17.0`：

| 文件 | 字节数 | SHA-256 |
| --- | ---: | --- |
| `ip2region_v4.xdb` | `11114380` | `6307a9696f5711f84bcb8b25f07894de68a64a0ed4a1cc7e990562dd3084f210` |
| `ip2region_v6.xdb` | `37258897` | `5b93da35ac28bc316dccc54a758381f7a874ae0461dd51ff5df5e34815586f11` |

下载到临时文件，校验后再替换正式文件：

```bash
IPREGION_DIR=/data/admin/ip2region
IPREGION_V4_FILE="$IPREGION_DIR/ip2region_v4.xdb"
IPREGION_V6_FILE="$IPREGION_DIR/ip2region_v6.xdb"
IPREGION_V4_TMP="$IPREGION_DIR/.ip2region_v4.xdb.tmp.$$"
IPREGION_V6_TMP="$IPREGION_DIR/.ip2region_v6.xdb.tmp.$$"
install -d -m 0755 "$IPREGION_DIR"
trap 'rm -f "$IPREGION_V4_TMP" "$IPREGION_V6_TMP"' EXIT

curl --fail --location --retry 3 --connect-timeout 10 --max-time 300 --output "$IPREGION_V4_TMP" \
  https://raw.githubusercontent.com/lionsoul2014/ip2region/v3.17.0/data/ip2region_v4.xdb
printf '%s  %s\n' '6307a9696f5711f84bcb8b25f07894de68a64a0ed4a1cc7e990562dd3084f210' "$IPREGION_V4_TMP" | sha256sum --check --strict

curl --fail --location --retry 3 --connect-timeout 10 --max-time 300 --output "$IPREGION_V6_TMP" \
  https://raw.githubusercontent.com/lionsoul2014/ip2region/v3.17.0/data/ip2region_v6.xdb
printf '%s  %s\n' '5b93da35ac28bc316dccc54a758381f7a874ae0461dd51ff5df5e34815586f11' "$IPREGION_V6_TMP" | sha256sum --check --strict

chmod 0444 "$IPREGION_V4_TMP" "$IPREGION_V6_TMP"
[ ! -f "$IPREGION_V4_FILE" ] || cp -p "$IPREGION_V4_FILE" "$IPREGION_V4_FILE.bak"
[ ! -f "$IPREGION_V6_FILE" ] || cp -p "$IPREGION_V6_FILE" "$IPREGION_V6_FILE.bak"
mv -f "$IPREGION_V4_TMP" "$IPREGION_V4_FILE"
mv -f "$IPREGION_V6_TMP" "$IPREGION_V6_FILE"
```

macOS 使用 `shasum -a 256 <文件>` 对照上表。只需 IPv4 时删除 IPv6 命令并将 `ipv6_xdb_path` 留空。

```yaml
ip_region:
  enabled: true
  ipv4_xdb_path: "/data/admin/ip2region/ip2region_v4.xdb"
  ipv6_xdb_path: "/data/admin/ip2region/ip2region_v6.xdb"
```

容器将 `/data/admin/ip2region` 挂载为只读卷；systemd 使用宿主机绝对路径并确保运行用户可读。发布前验证真实数据库：

```bash
IPREGION_TEST_IPV4_XDB=/data/admin/ip2region/ip2region_v4.xdb \
IPREGION_TEST_IPV6_XDB=/data/admin/ip2region/ip2region_v6.xdb \
go test -count=1 -race ./internal/infra/ipregion
```

XDB 或路径变更后滚动重启。回滚时恢复 `.bak` 或上一版本 XDB 并再次滚动重启；不依赖热加载替换内存库。

### 可选 ClamAV

不扫描时配置 `file_storage.virus_scanner.name=noop`。启用 ClamAV 时：

```yaml
file_storage:
  virus_scanner:
    name: "clamav"
    command: "clamdscan"
    config_file: "/etc/clamav/clamd.conf"
    timeout_seconds: 120
```

Docker 参考 `deploy/docker/clamd.conf`；systemd 需在宿主机安装 `clamdscan` 和 daemon。`StreamMaxLength` 不得小于最大上传文件，TCP 3310 只对应用所在受信网络开放。启动和 readiness 检查失败时不应继续发布。

## 数据库迁移

已有库按以下顺序执行：

```bash
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml
# 备份并核对执行计划后
make migrate-up MIGRATE_CONFIG=./etc/config.yaml
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
```

空库初始化或已确认的基线补登记才使用 `make migrate-bootstrap`。出现 checksum drift、未知 `blocked`、错库或无备份时停止发布。完整边界见[数据库迁移治理](角色文档/运维/数据库迁移治理.md)。

## 运行模式与拓扑

`mode` 是能力位掩码：`1=API`、`2=Worker`、`4=Scheduler`。推荐部署：

| 场景 | 启动方式 |
| --- | --- |
| 本地一体化 | `./bin/admin -f ./etc/config.yaml -mode 7` |
| 生产控制面 | `./bin/admin -f ./etc/config.yaml -mode 5` |
| 生产 Worker | `./bin/admin -f ./etc/config.yaml -mode 2` |
| 需 HTTP 探针的 Worker | `./bin/admin -f ./etc/config.yaml -mode 3` |

命令行 `-mode` 优先于 `run_mode`。控制面和 Worker 共用 `task.redis`；Scheduler 可多副本部署，实际投递由 Redis leader 租约保证单主。平台终止宽限期不得低于 30 秒。

## 首次部署

1. 完成数据库迁移。
2. 确认 `runtime_config.source`：文件模式准备运行期文件；数据库模式无 active release 时，确保迁移草稿非空，草稿为空时提供运行期文件作为初始导入源。
3. 启动控制面，确认 `/api/ready` 和 active release 后再启动 Worker。
4. 在任务页确认 Worker、Scheduler 和运行配置版本。
5. 无法使用已有超级管理员登录时，从内网调用自举接口：

```bash
curl -sS -X POST 'http://127.0.0.1:8888/internal/auth/init-admin-bootstrap' \
  -H 'Content-Type: application/json' \
  --data-raw '{"username":"super999","password":"<一次性强随机密码>"}'
```

该接口只重置已存在的超级管理员，不创建账号或提升角色。成功后立即登录、修改密码并按配置绑定 MFA。完整约束见[内网初始化管理员接口](接口文档/后台系统/内网初始化管理员接口.md)。

## 发布流程

1. 确认构建产物、配置、迁移计划和回滚制品。
2. 执行数据库迁移。
3. 滚动发布控制面，确认 readiness。
4. 滚动发布 Worker，确认 Worker 在线和队列无异常积压。
5. 核对运行配置 active release 或文件热加载状态。
6. 执行发布后验证。

## 发布后验证

- `/api/live` 和 `/api/ready` 成功，MySQL、Redis、任务系统状态正常。
- `/api/metrics` 可被 Prometheus 从内网抓取，公网无法直接访问。
- 后台可登录，菜单、权限和个人中心正常。
- 任务页可看到队列、Worker 和 Scheduler；手工触发一个低风险任务并确认终态。
- Collector 启用时，确认审计日志可持久化且无持续失败。
- 数据库运行配置模式下，确认 active release、发布历史和当前快照一致。
- 启用 IP 归属地或 ClamAV 时，分别验证登录 IP 归属地和 EICAR 阻断链路。

## 回滚

1. 摘除异常实例，恢复上一版本二进制、配置和可选 XDB。
2. 先恢复控制面，再恢复 Worker。
3. 运行配置发布异常时，通过发布历史回滚 active release。
4. 检查队列积压和失败任务；只重跑已确认幂等的任务。
5. 数据库变更按迁移单恢复备份或执行正向补偿，不假设二进制回滚会自动回退 DDL。
6. 重新执行发布后验证。

## 常见问题

- API 成功但任务未执行：检查 Worker、队列暂停状态和 `task.redis`。
- 周期任务未投递：检查 Scheduler 能力、leader 租约、active release 和 cron。
- 多站点任务串扰：检查 `app_id`，同站点必须一致，不同站点必须不同。
- 管理员无法登录：确认已存在超级管理员，再从内网使用自举接口恢复。
