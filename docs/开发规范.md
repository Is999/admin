# admin 开发规范

> **AI 开发必读**：使用 AI 做代码生成、重构、补测试或补文档前，必须先阅读 [AI开发规范](site/角色文档/后端开发/AI开发规范.md)。该文档是可复制提示词的唯一维护入口。

## 命名规范

- Go 包名短小、全小写、不使用下划线。
- 请求结构体以 `Req` 结尾，响应结构体以 `Resp` 结尾，列表项以 `Item` 结尾。
- 常量按领域分组，业务码必须落在对应号段。
- 路由别名使用 `domain.resource.action`，例如 `collector.overview`。
- Redis Key 统一放在 `common/rediskeys`。

## 注释规范

以下内容必须补中文注释：

- 变量、常量、方法。
- 结构体及内部字段。
- map 字段含义。
- 数组或切片字段含义。
- HTTP 路由。
- GORM Model。
- 方法内部关键逻辑。
- 接口及其内部方法。

API 请求响应、配置、任务 payload、持久化 JSON 等顶层稳定契约里的 map、数组或切片字段，必须说明 key/value 或元素结构、单位、排序、去重和空值语义，不能只注释字段名。

## Handler 规范

- handler 只解析参数、调用 logic、写响应。
- 参数解析失败统一返回 `codes.ParamError`。
- 业务失败不在 handler 内拼接英文或中文固定文案，应使用 `i18n` key。
- 失败响应应通过 `helper.NewJSONResp(ctx, w)` 写出，确保 `traceId/spanId`、HTTP 状态码和业务码统一回填。

## Logic 规范

- logic 内不直接创建 DB、Redis、Kafka 等依赖。
- logic 内错误向上返回，避免重复打印日志。
- 业务内层禁止随意打印 `error` 级日志；无法向上返回的后台兜底场景才允许直接调用 `loggerx.Errorw`。

## GORM 规范

- 查询优先使用 `Model/Where/Scopes/Find/First/Updates` 链式调用。
- 写操作使用 `WriteDB`，读操作使用 `ReadDB`。
- 涉及事务时，事务内继续使用同一 `tx`。
- 原生 SQL 只允许用于动态 DDL、批量切表和 ClickHouse 聚合。

## 代码文件格式规范

- `*.sql.tmpl` 属于代码文件，用于承载 `go:embed` 加载的原生 SQL 模板，不是临时 SQL、导出 SQL 或可清理产物，禁止加入
  `.gitignore` 或清理脚本。
- `*.lua` 属于代码文件，用于承载 `go:embed` 加载的 Redis Lua 脚本，不是临时脚本或运维手工脚本，禁止误删、改名或移动后不同步
  Go 侧 `//go:embed`。
- 新增 `*.sql.tmpl` 必须在文件头使用 SQL 注释说明业务用途、SQL 方言、调用方、参数来源和安全边界；SQL 方言必须明确标注 MySQL、ClickHouse 或 ANSI SQL，必要时补充版本或方言特性；动态表名、字段名、条件片段必须来自白名单或已转义值，用户输入仍通过参数绑定。
- 新增 `.lua` 必须在文件头使用 Lua 注释说明业务用途、调用方、`KEYS`/`ARGV` 含义和原子性边界；涉及删除、释放锁、续租等敏感操作必须说明
  owner 比对或防误删策略。
- `*.sql.tmpl` 文件头统一按“代码资产、SQL 方言、调用方、参数来源、安全边界”说明；`.lua` 文件头统一按“代码资产、调用方、KEYS/ARGV、原子性边界”说明。
- SQL/Lua 实际执行前必须通过 `embedasset.StripLeadingLineComments(..., "--")` 或当前包渲染 helper 剥离文件头说明，禁止把带说明的原始脚本直接交给 MySQL、ClickHouse 或 Redis 执行。
- IDE 可以把 `*.sql.tmpl` 关联为 SQL/ClickHouse/MySQL 方言，把 `.lua` 关联为 Lua；高亮配置不能改变文件后缀，避免破坏
  `go:embed` 路径。

## 错误包装

```go
if err != nil {
    return errors.Tag(err)
}
```

需要补业务上下文时：

```go
if err != nil {
    return errors.Wrapf(err, "用户标签计算失败 workflow_id=%s node=%s shard=%d/%d", workflowID, node, shardIndex, shardTotal)
}
```

## 日志规范

业务内层禁止随意打印 `error` 级日志。`error` 日志只允许在 handler、worker、scheduler 或不可返回错误的兜底协程等最外层统一打印；中间层通过 `errors.Tag`、`errors.Wrap` 或 `errors.Wrapf` 保留上下文。

```go
loggerx.Infow(ctx, "任务节点 开始处理",
    logx.Field("workflow_id", workflowID),
    logx.Field("mode", mode),
    logx.Field("node", node),
    logx.Field("shard", shard),
)
```

错误日志只在最外层统一打印：

```go
loggerx.Errorw(ctx, "任务节点 处理失败", err)
```

## 验收

```bash
go test ./...
```

必须保证：

- 新增业务码有中英文文案。
- 新增路由有 `RouteMeta`、审计动作和中文说明；需要权限控制时补权限 SQL，token 白名单路由必须写明例外原因。
- 新增错误有 `errors.Tag` 或 `errors.Wrap/Wrapf`。
- 新增日志包含 trace/workflow/task/collector 所需字段。
- 新增失败出口会把 `traceId/spanId` 回传给前端，便于用户反馈和日志检索对齐。

## AI 开发规范

AI 参与开发、重构和补文档时必须遵循以下约束：

- 先阅读现有代码和文档，按当前项目风格实现，不能生成独立于现有架构的新风格代码。
- 优先延续当前项目模式和现有分层；遇到成熟通用能力、高风险基础设施或手写实现容易出错的场景，必须先检查现有实现和成熟依赖/框架。确需新增抽象、目录、框架或依赖时，先说明必要性、收益、替代方案、影响范围和验证方式，并让开发确认后再落地。
- 代码必须简洁、逻辑清晰、易读易维护；合理封装只能用于降低真实复杂度、复用稳定边界或隔离外部依赖，禁止为简单逻辑过度设计，也禁止因怕新增依赖而手搓不可靠实现。
- 所有新增变量、常量、结构体字段、方法、路由、GORM Model、关键逻辑必须补中文注释；顶层稳定契约里的 map、数组或切片字段必须说明 key/value 或元素结构、单位、排序、去重和空值语义。
- 数据库操作优先 GORM 链式调用，只有动态 DDL、批量切表、ClickHouse 聚合或 GORM 难以表达的高性能聚合允许原生 SQL。
- 原生 SQL 必须放入 `*.sql.tmpl` 并通过 `go:embed` 加载，禁止在 Go 业务代码中直接内联多行 SQL、`fmt.Sprintf("SELECT ...")` 或 `Exec("RENAME TABLE ...")` 这类模板外 SQL。
- Redis Lua 必须放入 `.lua` 并通过 `go:embed` 加载，禁止在 Go 业务代码中直接使用 `redis.NewScript(`...`)` 内联脚本。
- `*.sql.tmpl` 和 `.lua` 都是 `go:embed` 代码资产，必须提交、补文件头中文注释，禁止当作临时模板或脚本清理。
- 新增 SQL/Lua 模板时必须同步模板渲染方法和最小单测，确保占位符已完全渲染、`KEYS/ARGV` 注释完整，并在实际执行前剥离文件头说明。
- Redis 高基数 Key 必须使用索引集合、精确 Key 或登记模板，禁止业务接口依赖通用 `SCAN`。
- 相同的分页、去重、压缩、模板渲染、Redis 精确 key 和错误包装逻辑应先查找现有公共 helper；确认没有现成方法时再新增，并放在对应包的公共 helper 文件中。
- 错误必须使用 `github.com/Is999/go-utils/errors` 的 `errors.Tag`、`errors.Wrap` 或 `errors.Wrapf` 包装。
- 同一个函数里，同一个 `error` 值只包装一次；跨函数边界可以继续包装，因为那是新的调用语义和上下文，避免裸错误漏出去。
- 业务中间层只返回错误，禁止随意打印 `error` 级日志；`error` 日志只在 handler、worker、scheduler 等最上层统一打印。
- 用户标签、任务系统、收集器相关日志必须带 `workflow_id`、`mode`、`node`、`shard`、`task_id`、`event_id` 等链路字段。
- 新增接口必须补接口文档，格式遵循 `docs/site/接口文档/接口文档统一规范.md`。
- 新增业务码必须补中文和英文文案，确保不同业务错误可通过状态码追踪；错误消息不能暴露 SQL、密钥、内部表名或真实下游错误。
- 每次改动后必须运行对应测试；能跑全量时执行 `go test ./...`。

更完整的 AI 协作约束见：`docs/site/角色文档/后端开发/AI开发规范.md`。
