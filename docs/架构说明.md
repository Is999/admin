# admin 架构说明

> **AI 开发必读**：所有 AI 生成、重构和补文档工作必须先阅读 [AI开发规范](site/角色文档/后端开发/AI开发规范.md)，重点遵守最外层统一打印 `error` 日志、SQL/Lua 嵌入资产文件头格式和执行前注释剥离规则。

## 架构目标

`admin` 以 go-zero 推荐的 `handler / logic / svc / config / model` 分层为主轴，同时保留 Cron 项目已有的任务引擎、运行时插件、Collector、用户标签和归档能力。重构目标不是简单搬迁目录，而是把权限、审计、错误链、日志、trace、健康检查和文档统一收口。

## 目录职责

- `common/codes`: 统一业务码、HTTP 状态码映射和业务码号段。
- `common/i18n`: 统一中英文响应文案，入口按 `Accept-Language` 选择语言。
- `common/rediskeys`: Redis Key 模板，目录名避免下划线，包名保持 `keys`。
- `helper`: 响应写出、上下文管理员等 HTTP 辅助能力。
- `internal/bootstrap`: 多模式启动、组件生命周期和注册规格派生。
- `internal/handler`: HTTP 路由和参数解析，不写业务逻辑。
- `internal/logic`: 业务用例层；按 `admin`、`profile`、`rbac`、`message`、`task`、`collector`、`usertag`、`file`、`security`、`health` 分包。`cache` 子包收口登录态缓存、table-cache 管理、精准失效和 Lua 资产；根包只保留 `BaseLogic`、缓存防穿透、MySQL duplicate 判断和稳定 helper。
- `internal/model`: GORM Model 和仓储辅助。
- `internal/svc`: `ServiceContext`，唯一依赖装配入口。
- `internal/task/queue`: Asynq 队列、DAG 工作流、周期调度、Leader 租约。
- `internal/task/runtime`: 任务插件注册运行时和核心插件。
- `internal/infra/collectorx`: Kafka-only 收集器和失败账本重试。
- `internal/jobs`: 定时任务和工作流业务实现收口目录，后续新增任务优先在该目录按业务建子包。
- `internal/jobs/usertag`: 用户标签分片计算、规则、仓储和工作流。
- `internal/jobs/export`: 管理员异步导出任务。
- `internal/jobs/archive`: 热表异步归档、管理员日志查询和 SQL 模板资产。
- `pkg`: Excel、存储、文件传输、批处理等可复用能力。

## 启动模式

- `-mode=1`: API，仅启动 HTTP Server。
- `-mode=2`: Worker，仅启动任务消费。
- `-mode=4`: Scheduler，仅启动周期调度。
- `-mode=7`: 全量启动，适合本地联调。

生产建议 API、Worker、Scheduler 独立部署，分别横向扩容。

## 链路追踪

- HTTP 入口优先继承标准 `traceparent`。
- 当前端只传 `X-Trace-Id` 时，后端会把 UUID/32位十六进制格式转换为 OTel trace。
- 响应头统一回写 `X-Trace-Id / X-Span-Id`。
- 响应体统一回写 `traceId / spanId`，前端错误提示可直接展示可检索链路号。
- 日志顶层字段只保留 `trace_id / span_id / route / user_id / error / error_chain / workflow_id / mode / node / shard_index / shard_total` 等公共检索维度，其余细节进入 `content`。

## 错误与日志

- 业务内层只返回 `errors.Tag(err)` 或 `errors.Wrap/Wrapf` 后的错误。
- HTTP handler、Worker 顶层、Collector worker 顶层统一打印 `error` 级日志；业务内层禁止随意打印 `error` 级日志。
- 错误日志使用 `loggerx.Errorw`，顶层保留 `error` 和 `error_chain`，其它错误详情随日志正文写入 `content`。
- 普通信息日志使用 `loggerx.Infow`，调试日志使用 `loggerx.Debugw`，慢操作使用 `loggerx.Sloww`。

## 响应码

- 10xxx: 通用业务。
- 20xxx: 认证权限。
- 30xxx: 后台系统。
- 40xxx: 任务系统。
- 41xxx: 工作流。
- 42xxx: Collector。
- 43xxx: 用户标签。
- 44xxx: 文件传输。
- 50xxx: 外部依赖。

`codes.HTTPStatus(code)` 是业务码到 HTTP 状态码的唯一映射入口。

## 数据库访问

- 默认使用 GORM 链式调用。
- 动态 DDL、批量切表、ClickHouse 聚合允许原生 SQL。
- 原生 SQL 必须参数化，并在代码中写清中文逻辑说明。
- 业务代码默认使用 `DatabaseMain`；确需拆库时通过 `site_mysql.<name>` 注册命名扩展库。

## 健康检查

- `GET /api/live`: 存活检查，不访问外部依赖。
- `GET /api/ready`: 就绪检查，探测 MySQL、Redis、Kafka、TaskQueue、Collector。
